# ARCHITECTURE_DECISIONS.md — DalyHub Architecture Decision Records

> The record of **major architectural decisions and their reasoning**. When you need to know *why* the system is built a certain way — before you change it — the answer is here.
>
> These ADRs are load-bearing. The [architecture philosophy in `AGENTS.md §9`](../../AGENTS.md#9-architecture-philosophy) summarises them; this document is the authority on the *why*. The technical *how* is in [`ARCHITECTURE_OVERVIEW.md`](../architecture/ARCHITECTURE_OVERVIEW.md).

---

## What an ADR is, and how to add one

An **Architecture Decision Record** captures a single significant decision: the context, the choice, and the consequences. ADRs are **append-only** — we don't rewrite history. To change a past decision, add a **new** ADR that supersedes the old one and update the old one's status to `Superseded by ADR-NNN`.

**Add a new ADR whenever** a change: introduces or removes a core concept; changes a kernel contract; picks a foundational technology; or deviates from an existing ADR. Adding the ADR is part of the [Definition of Done](../../AGENTS.md#18-definition-of-done).

**ADR template:**

```markdown
## ADR-NNN: <Title>
- **Status:** Proposed | Accepted | Superseded by ADR-MMM
- **Context.** What forces and constraints are at play?
- **Decision.** What we decided, stated plainly.
- **Consequences.** What this makes easy, what it makes hard, what we accept.
- **Alternatives considered.** What we rejected and why.
```

Statuses below are **Accepted** unless noted. Numbering is stable and permanent.

## ADR-001: Area Hierarchy
- **Status:** Accepted. Implemented by [ADR-014](#adr-014-spine-hierarchy-completion-and-rollup-semantics) (spine hierarchy, completion & rollup semantics).
- **Context.** DalyHub must make a whole life legible. Users need to see how daily action connects to long-term intention. A flat list of tasks cannot express this; an over-deep tree creates bureaucracy.
- **Decision.** Adopt a first-class, kernel-level **Area → Goal → Project → Task** hierarchy with rollup semantics. Areas are permanent domains; Goals are optional outcomes; Projects are finite work; Tasks are atomic actions. This is a kernel concept, not a per-module convention. (See [`AGENTS.md §4`](../../AGENTS.md#4-the-area--goal--project--task-model).)
- **Consequences.** *Easy:* consistent organisation and progress rollup everywhere; the intention↔action view (`AREA-03`) becomes possible. *Hard:* every entity must resolve to an Area; rollup must stay correct and performant. *Accepted:* Goals are optional to avoid forcing structure where it doesn't help.
- **Alternatives considered.** Flat tags-only (rejected: no rollup, no legibility). Deeper hierarchy (rejected: bureaucracy, violates "structure serves clarity"). Per-module hierarchies (rejected: fragmentation).

## ADR-002: EntityLinks
- **Status:** Accepted. Implemented by [ADR-011](#adr-011-entitylink-persistence-and-lifecycle) (persistence & lifecycle).
- **Context.** DalyHub's core value is connection — a meeting produces tasks, a note documents a project, a person recurs everywhere. Per-module foreign keys would make each relationship bespoke and invisible to other modules.
- **Decision.** Provide **typed, bidirectional EntityLinks** as a kernel primitive: any entity may link to any other; links are queryable from both ends and rendered through shared UI (the [link picker](../design/DESIGN_SYSTEM.md#forms) and backlinks). (See [`AGENTS.md §9.5`](../../AGENTS.md#95-entitylinks).)
- **Consequences.** *Easy:* cross-module relationships and backlinks with no per-module work; the connectedness the product promises. *Hard:* referential integrity and cascade behaviour on delete must be handled centrally; link types need governance to avoid sprawl. *Accepted:* a general link table over rigid schema-per-relationship.
- **Alternatives considered.** Foreign keys per relationship (rejected: rigid, not cross-module). Freeform tags only (rejected: untyped, not bidirectional).

## ADR-003: Workspace Isolation
- **Status:** Accepted.
- **Context.** A user may keep distinct contexts (personal life, a side venture) that shouldn't bleed together, and data isolation is also a security boundary.
- **Decision.** Make **Workspace** the top-level isolation boundary. Every entity belongs to a workspace; every query is workspace-scoped **server-side**; search, activity, and permissions are all scoped to it. (See [`AGENTS.md §9.4`](../../AGENTS.md#94-workspace-isolation).)
- **Consequences.** *Easy:* clean separation of contexts; isolation doubles as a security control ([security requirements](../../AGENTS.md#17-security-requirements)). *Hard:* every data path must carry workspace scope; cross-workspace features (if ever needed) require explicit design. *Accepted:* scoping overhead on every query in exchange for a hard boundary.
- **Alternatives considered.** Single global space with tags (rejected: no real isolation, weaker security). Per-module isolation (rejected: inconsistent, leaky).

## ADR-004: AI Proposal Architecture
- **Status:** Accepted.
- **Context.** AI must help without eroding user control or trust, and DalyHub holds highly sensitive data. Autonomous AI mutation would violate the product's core promise that the user is always in control.
- **Decision.** AI is a **proposer, never an autonomous actor**. It emits structured, reviewable **proposals** (suggested creates/links/edits) that the user accepts, edits, or rejects — in whole or in part. Nothing is written without approval. The AI operates over the same model as human actions, with **per-action opt-in** for sensitive entities. (See [`AGENTS.md §8`](../../AGENTS.md#8-ai-philosophy) and roadmap `AI-01`/`AI-04`.)
- **Consequences.** *Easy:* trust, auditability, privacy, and reversibility; AI features can be added safely. *Hard:* every AI capability needs a proposal schema and review UI; no "just do it" shortcuts. *Accepted:* extra friction on AI actions as the price of control.
- **Alternatives considered.** Autonomous agents with undo (rejected: violates control/trust, unsafe with sensitive data). AI writing directly to a staging area only (rejected: still bypasses explicit consent).

## ADR-005: Shared Activity Model
- **Status:** Accepted. Implemented by [ADR-012](#adr-012-activity-persistence-and-atomic-mutation-recording) (persistence & atomic recording).
- **Context.** Every module needs history (Timeline) and the product needs a cross-cutting Activity Feed and an audit trail. Per-module event logs would fragment this and duplicate work.
- **Decision.** A **single append-only Activity model**: every meaningful mutation appends a uniform event. Record [Timeline](../design/DESIGN_SYSTEM.md#timeline) and cross-scope [Activity Feed](../design/DESIGN_SYSTEM.md#activity-feed) render this one model at different scopes. It also serves as a security-relevant audit trail. (See [`AGENTS.md §9.6`](../../AGENTS.md#96-shared-activity-model).)
- **Consequences.** *Easy:* uniform history everywhere for free; audit trail; one component to build/maintain. *Hard:* mutations must reliably emit events; the stream needs virtualisation and retention strategy at scale. *Accepted:* write-time cost of appending events.
- **Alternatives considered.** Per-module logs (rejected: fragmentation, duplicated UI). Derive history from diffs on read (rejected: costly, lossy).

## ADR-006: Markdown Strategy
- **Status:** Accepted. Implemented by [ADR-015](#adr-015-markdown-source-and-safe-rendering-pipeline) (Markdown source & safe rendering pipeline).
- **Context.** Notes, descriptions, and Diary need rich text that stays portable, exportable, and safe. Proprietary rich-text blobs risk lock-in and XSS.
- **Decision.** Author and store long-form text as **Markdown**, rendered through **one shared, sanitising renderer**. This keeps content portable and diff-able and supports [data portability](../roadmap/ROADMAP_V2.md#-x-04--export--data-portability). Raw HTML is not trusted; rendering sanitises. (See [`AGENTS.md §9.7`](../../AGENTS.md#97-markdown-strategy).)
- **Consequences.** *Easy:* portability, export, safety, one renderer to harden. *Hard:* very rich formatting (complex tables, embeds) is constrained to what the pipeline supports. *Accepted:* format constraints in exchange for ownership and safety.
- **Alternatives considered.** Proprietary block/JSON document model (rejected: lock-in, heavier). Raw HTML (rejected: XSS, not portable).

## ADR-007: Module Registry
- **Status:** Accepted. Implemented by [ADR-013](#adr-013-module-registry-contract-and-discovery) (contract & discovery).
- **Context.** DalyHub is many modules on a small kernel. Wiring each module through central switch statements would make the shell a bottleneck and every module addition a merge-conflict magnet.
- **Decision.** Modules **self-register** via a **Module Registry**: routes, entity types, commands (for the [Command Palette](../design/DESIGN_SYSTEM.md#command-palette)), search providers, and settings. The shell discovers modules through the registry; adding a module doesn't require editing central code. (See [`AGENTS.md §9.2`](../../AGENTS.md#92-module-registry).)
- **Consequences.** *Easy:* modules are independent and independently implementable (matches [ROADMAP](../roadmap/ROADMAP_V2.md) structure); search/commands/routes compose automatically. *Hard:* the registry contract must be stable and well-specified; a bad module registration can affect the shell. *Accepted:* an indirection layer for decoupling.
- **Alternatives considered.** Central hard-wired module list (rejected: bottleneck, conflicts). Fully independent micro-frontends (rejected: over-engineered for one product, breaks shared kernel benefits).

## ADR-008: Initial Application Platform and Toolchain
- **Status:** Accepted.
- **Context.** DalyHub V2 is a clean redevelopment ([`ROADMAP_V2` Phase 0](../roadmap/ROADMAP_V2.md#phase-0--foundation-fnd)). Before building the kernel we need a reliable application skeleton: a language, framework, build, test, lint/format, type-check, CI, and a deployment target. Prior ADRs settled the *shape* (small kernel, modular userland) but not the concrete platform; [`ARCHITECTURE_OVERVIEW.md`](../architecture/ARCHITECTURE_OVERVIEW.md) described the platform only as a *proposed direction* oriented around the Cloudflare Developer Platform. DalyHub V1 ran on a Next.js/Docker/VPS stack whose operational weight (self-managed servers, container builds, a fragile and slow end-to-end setup) worked against the product's calm, low-friction ethos. FND-01 must convert the proposed direction into a settled, verifiable foundation without prematurely committing to storage services or product architecture.
- **Decision.** Adopt the following as DalyHub V2's application platform and toolchain, using the official Cloudflare React Router starter (`create-cloudflare`, `--framework=react-router`) as the reference scaffold:
  - **React Router v8 in framework (full-stack) mode** — routing, SSR, loaders/actions, and resource routes in one typed framework.
  - **React 19 + TypeScript (strict)** — the typed component model the Design System will build on.
  - **Vite** with the **official Cloudflare Vite plugin** — one build tool that runs server code in the Workers runtime during dev and preview, so local behaviour matches production.
  - **Cloudflare Workers** as the runtime/deployment target, managed with **Wrangler** (local dev, type generation, deploy).
  - **pnpm via Corepack** — deterministic, lockfile-pinned dependency management.
  - **Vitest + React Testing Library** for unit/component tests; **Playwright** (single Chromium project) for one minimal browser smoke test.
  - **ESLint (flat config) + Prettier** for linting and formatting; **GitHub Actions** for CI.
  - TypeScript is pinned to the 5.9 line (not the newer native TS 7 preview), because the framework's type generation, `typescript-eslint`, and editor tooling target the 5.x language service; adopting TS 7 now would be a material incompatibility for no FND-01 benefit.
- **Why this fits DalyHub.** A single full-stack framework on one runtime keeps the surface small and coherent — the same discipline the kernel demands. SSR-by-default serves the [performance budgets](../../AGENTS.md#16-performance-expectations) (fast first paint, lean payloads). Running the *actual* Workers runtime locally via the Vite plugin means "drive the real thing before claiming done" ([testing philosophy](../../AGENTS.md#14-testing-philosophy)) is cheap and honest. Managed edge compute removes servers-to-babysit, which supports *calm* as an operational property, not just a UI one.
- **Why preferable to rebuilding V1's Next.js/Docker/VPS stack.** The V1 stack coupled the app to self-managed infrastructure: Docker image builds, a VPS to patch and secure, and an E2E suite that was slow and flaky. That is recurring operational tax and a reliability risk with no product payoff for a single-owner product. Workers + Wrangler give zero-server deploys, a local runtime that mirrors production, and a fast, deterministic test path — directly addressing V1's fragility.
- **Consequences.** *Easy:* one command to develop, one to build, one to deploy; production-faithful local runtime; fast CI; a clean base for the kernel. *Hard:* code must respect Workers runtime constraints (no arbitrary Node APIs; `nodejs_compat` where needed); some libraries assuming a full Node/browser environment need care; SSR adds a server render path to reason about. *Accepted:* coupling to the Cloudflare platform for compute in exchange for operational simplicity — mitigated by keeping storage/service choices as separate, later decisions (below) and the kernel abstracted from the platform ([`ARCHITECTURE_OVERVIEW.md`](../architecture/ARCHITECTURE_OVERVIEW.md)).
- **Storage is explicitly out of scope here.** This ADR selects compute, framework, and toolchain only. **D1, KV, R2, Durable Objects, and any other storage/service remain later roadmap decisions** (starting at [FND-02](../roadmap/ROADMAP_V2.md#-fnd-02--data-kernel-entities--storage)) unless already accepted in another ADR. No database or bindings are introduced by FND-01.
- **Alternatives considered.**
  - *React SPA + a separate Worker API.* Two deployables, two type boundaries, and manual data-fetching wiring. Rejected: more moving parts and worse first-paint than an integrated SSR framework, for no benefit at this stage.
  - *Next.js on Workers.* Familiar, but a heavier framework whose Workers support is less first-class than React Router's, and closer to the V1 stack we are deliberately leaving. Rejected: weight and fit.
  - *Return to Docker + Postgres on a VPS.* Maximum control and a conventional relational DB. Rejected: reintroduces exactly the operational fragility and slowness V1 suffered; premature to pick a database before the [data kernel](../roadmap/ROADMAP_V2.md#-fnd-02--data-kernel-entities--storage) is designed.

## ADR-009: Data Kernel Storage
- **Status:** Accepted.
- **Context.** [FND-02](../roadmap/ROADMAP_V2.md#-fnd-02--data-kernel-entities--storage) implements the entity substrate every DalyHub record builds on: a uniform record (id, workspace, type, title, timestamps, soft-delete) that must be persisted, migration-managed, strongly typed, and testable in the real runtime. [ADR-008](#adr-008-initial-application-platform-and-toolchain) settled compute (Cloudflare Workers) but deliberately left storage open. [`ARCHITECTURE_OVERVIEW.md`](../architecture/ARCHITECTURE_OVERVIEW.md#platform) named D1 as the *proposed* relational store, to be confirmed by its own ADR before anything relies on it. DalyHub is currently single-user and Cloudflare-native, so the store should add persistence without adding a server to operate.
- **Decision.**
  - **Cloudflare D1** is the initial relational store for the kernel. SQLite-compatible SQL is the storage language.
  - **Schema changes use committed, sequential Cloudflare D1 SQL migrations** in `migrations/` (`0001_create_entities.sql`, …), applied with Wrangler's D1 migration tooling.
  - The kernel uses **a small, DalyHub-owned typed repository over prepared D1 statements** (`EntityRepository` contract in `app/kernel/entities`; D1 adapter in `app/platform/storage/d1`). All data values are bound parameters — never interpolated. **No ORM is introduced in FND-02.**
  - **Domain-specific fields belong to domain tables (or deliberately designed extensions), not to the base entity table.** The `entities` table carries only the shared record header; it is never an unstructured JSON dumping ground. Domain tables arrive with later roadmap items.
  - An ORM may be reconsidered later **only through a new ADR** if real query complexity justifies it.
- **Why this fits DalyHub.** DalyHub is single-user and Cloudflare-native, so D1 provides relational persistence without introducing a separate database server to run, patch and back up — keeping *calm* an operational property, consistent with ADR-008. Plain SQL migrations remain portable, inspectable and easy to recover (aligned with the "own your data" principle). The data kernel is small enough that an ORM would currently add more abstraction than value. A DalyHub-owned repository contract keeps D1-specific APIs from leaking into modules, preserving the small-kernel/portable-adapter boundary; if D1 is ever swapped, only the adapter changes.
- **Consequences.** *Easy:* real relational persistence with zero servers; migrations are legible git-tracked SQL; the typed contract gives modules a stable, storage-agnostic API; integration tests run against real D1 in the Workers runtime. *Hard:* SQLite/D1 constraints apply (no stored procedures, limited ALTER, per-database size/write limits); hand-written SQL must be disciplined as query complexity grows; workspace scoping is carried but not yet *enforced* as a security boundary (that is [FND-03](../roadmap/ROADMAP_V2.md#-fnd-03--workspace-isolation)). *Accepted:* writing SQL by hand instead of using an ORM, in exchange for a smaller, more transparent kernel.
- **Alternatives considered.**
  - *Drizzle ORM (or another TypeScript ORM/query builder).* Type-safe schema and queries, good D1 support. Rejected for FND-02: adds a dependency and an abstraction layer the current, deliberately-tiny schema does not need; revisit via a new ADR if/when query complexity justifies it.
  - *Another query builder (e.g. Kysely).* Lighter than an ORM but still an abstraction over a single small table with simple access paths. Rejected: the prepared-statement repository is clearer for this surface.
  - *PostgreSQL (Hyperdrive/external).* Mature and powerful, but reintroduces an external server to operate and pay for — exactly the operational weight ADR-008 moved away from — with no payoff for a single-user app at this stage.
  - *Generic JSON-document storage (KV/R2/JSON column).* Would make the entity a schemaless blob. Rejected: loses relational integrity, typed constraints, and efficient indexed queries/pagination; contradicts the decision to keep domain data in real tables.
  - *One independent table per entity type, with no common substrate.* Rejected: fragments the uniform record header, duplicates lifecycle/soft-delete logic per type, and undermines the shared kernel the whole architecture depends on.
  - *Reusing the DalyHub V1 Prisma schema.* Rejected: V1's schema and ORM carry assumptions and debt this greenfield kernel is deliberately leaving behind; the V1 schema is not copied.

## ADR-010: Server-side Workspace Context
- **Status:** Accepted.
- **Context.** [ADR-003](#adr-003-workspace-isolation) established Workspace as DalyHub's top-level isolation *and security* boundary, and [FND-02](../roadmap/ROADMAP_V2.md#-fnd-02--data-kernel-entities--storage) carried a `workspace_id` on every entity — but did not yet *enforce* it: the column had no foreign key, and the repository accepted a `workspaceId` argument on every call, so any caller could read or write any workspace. [FND-03](../roadmap/ROADMAP_V2.md#-fnd-03--workspace-isolation) must turn that carried field into a real boundary. This ADR records the concrete implementation of ADR-003; ADR-003 remains the governing principle. Authentication and sessions do not exist yet (single-user), and building isolation must not wait for them ([FND-09](../roadmap/ROADMAP_V2.md#-fnd-09--app-shell-routing--auth)).
- **Decision.**
  - **Workspace is a separate kernel/security record, not an ordinary `entities` row.** It lives in its own minimal `workspaces` table (`id`, `created_at`, `updated_at` only — no name, membership, role, billing, theme or settings). Entities *reference* a workspace through a database **foreign key** (`entities.workspace_id → workspaces.id`, `ON DELETE RESTRICT`); the database — not just application code — rejects an orphaned entity.
  - **Workspace context is established at the server composition boundary.** An `environment → resolver → WorkspaceContext → workspace-scoped EntityRepository` flow (`app/platform/workspaces`) wires scope once, server-side.
  - **Module-facing repositories are created already bound to one `WorkspaceContext`.** `createEntityRepository(db, context)` returns a repository whose every method operates only within that workspace. **No module-facing method accepts a `workspaceId`** — module code cannot pass, select or override the scope per operation. Every SQL statement still constrains `workspace_id = ?` with the bound value, always parameterised.
  - **Workspace context is resolved only from trusted server-side state.** The single-user resolver reads a configured `DEFAULT_WORKSPACE_ID` binding, validates it, and confirms the workspace exists in D1. The resolver interface (`WorkspaceContextResolver.resolve()`) **takes no arguments** — there is structurally nowhere to pass a request, header, cookie, query string, route parameter, form field or JSON body, so untrusted input cannot influence scope. It **fails closed**: missing/blank/malformed configuration or a non-existent workspace raises a typed error and never falls back to another workspace or auto-creates one.
  - **Cursors are bound to their scope.** Pagination cursors are versioned and carry the workspace, type filter and include-deleted state that produced them; a cursor replayed under a different scope is rejected as `InvalidCursorError`. Cursors are not signed/encrypted — workspace ids are scope identifiers, not secrets — versioning binds *shape and scope*, which is what correctness needs.
  - **`WorkspaceId` is a validated, branded type** so raw strings cannot drift through the kernel unchecked. It is a **scope identifier, not an authentication secret.**
  - **No global mutable workspace state and no `AsyncLocalStorage`.** Context is passed **explicitly** as a value.
  - **FND-09 may replace the static resolver with an authenticated one** implementing the same `WorkspaceContextResolver` interface, **without changing module repository contracts.**
  - **Administrative cross-workspace operations, if ever needed, use separate, deliberately-named contracts** (e.g. the low-level `WorkspaceRepository` used only at bootstrap), never a bypass on the scoped module repositories.
- **Why explicit context passing, over the alternatives.**
  - *Accepting `workspaceId` in every module call* (the FND-02 shape) makes the safe path and the unsafe path identical: forgetting a filter, or trusting a caller-supplied id, silently crosses the boundary. Binding scope at construction makes cross-workspace access *unrepresentable* in module code rather than merely discouraged.
  - *Trusting an `X-Workspace-ID` header* (or any request field) lets the client choose its own scope — the classic broken-access-control vulnerability. Scope must be derived server-side from trusted state, never from the request. A request-free `resolve()` enforces this structurally.
  - *Accepting workspace from query strings or form data* has the same defect as a header and additionally leaks scope into URLs/caches. Rejected for the same reason.
  - *Global mutable state* (a module-level "current workspace") is a data-race and test-isolation hazard on a request-concurrent runtime, and invites spooky action at a distance. Explicit values are safe under concurrency and trivially testable.
  - *`AsyncLocalStorage`* would thread context implicitly, but it is an ambient, hidden dependency: it hides the boundary the whole feature exists to make explicit, complicates testing, and adds a Workers-runtime footgun. An explicit parameter is clearer and cheaper.
  - *Making Workspace an ordinary entity* would give it soft-delete, arbitrary types and cross-workspace links it must never have, and would make "the boundary" just another row inside the thing it is meant to bound — circular. A dedicated security record with a real foreign key is a genuine boundary the database enforces.
  - *Postponing isolation until authentication is built* would leave every later foundation item (links, activity, the spine) built on an unscoped kernel, to be retrofitted dangerously later. Isolation is separable from *who* the user is: resolve *which* scope from trusted config now; swap in *authenticated* resolution at FND-09 behind the same interface.
- **Consequences.** *Easy:* modules get a repository that is correct-by-construction — there is no unscoped path to misuse; the database guarantees referential integrity even against raw SQL; swapping in authenticated resolution later touches only the resolver. *Hard:* every entity now requires its workspace to exist first (creates fail closed otherwise), and cursors from before this change are invalidated (acceptable — no released consumers). *Accepted:* a small `workspaces` table and a resolution step on the server path, in exchange for a real, enforced boundary.
- **Alternatives considered.** Covered inline above (per-call `workspaceId`, trusted header, query/form scope, global state, `AsyncLocalStorage`, Workspace-as-entity, defer-until-auth) — all rejected. Signed/encrypted cursors were also considered and rejected: workspace ids are not secrets, and versioned scope-binding already gives the correctness guarantee without key management.

## ADR-011: EntityLink Persistence and Lifecycle
- **Status:** Accepted.
- **Context.** [ADR-002](#adr-002-entitylinks) established **typed, bidirectional EntityLinks** as a kernel primitive but left the concrete persistence and lifecycle open. [FND-04](../roadmap/ROADMAP_V2.md#-fnd-04--entitylinks) must make links real on the existing kernel: the [entity substrate](#adr-009-data-kernel-storage) (FND-02) and the [enforced, server-derived workspace boundary](#adr-010-server-side-workspace-context) (FND-03). The hard questions ADR-002 flagged — how a relationship is stored, how referential integrity and deletion behave, and how link types are governed — are answered here. This ADR **implements** ADR-002; it does not replace it. Activity events ([FND-05](../roadmap/ROADMAP_V2.md#-fnd-05--shared-activity-model)) and link-type registration via the Module Registry ([FND-06](../roadmap/ROADMAP_V2.md#-fnd-06--module-registry)) are deliberately out of scope.
- **Decision.**
  - **One row per relationship — direction preserved.** A relationship is stored **once** as a single `entity_links` row carrying `source_entity_id`, `target_entity_id` and `type`. "Bidirectional" means **discoverable and navigable from both ends**, not two rows: querying the source sees an *outgoing* link, querying the target sees the *same row* as an *incoming* link. Endpoint ids are **never reordered** — `meeting --produced_task--> task` and its reverse are different relationships and must not be conflated. There is deliberately no undirected "canonical pair".
  - **EntityLinks are NOT entities.** A link is a kernel relationship record, not a row in `entities`. It has no title, entity type, search entry, Record Header, module or user-facing page. It lives in its own `entity_links` table with only justified fields: `id`, `workspace_id`, `source_entity_id`, `target_entity_id`, `type`, `created_at`, `updated_at`, `deleted_at`.
  - **Link types are validated, branded identifiers — governed, not free text.** `EntityLinkType` is a branded string validated to a documented structural format (lowercase dotted segments, e.g. `meeting.produced_task`), stored verbatim after validation. It is an **open, reusable contract** — no database enum, no hard-coded list of every future type — so future modules register link types without a schema migration (FND-06 will connect this to the Module Registry). There is no user-facing label in the kernel.
  - **Workspace-bound repository (no per-call `workspaceId`).** The module-facing `EntityLinkRepository` is constructed already bound to one `WorkspaceContext` (`createEntityLinkRepository(db, context)`), exactly like the entity repository. No method accepts a `workspaceId`; both endpoints of every link are constrained to the bound workspace in SQL, values always bound. Links **cannot cross workspace boundaries**.
  - **Database-enforced same-workspace endpoints.** A parent `UNIQUE (workspace_id, id)` key on `entities` lets `entity_links` use **composite foreign keys** `(workspace_id, source/target_entity_id) → entities (workspace_id, id)` — so a cross-workspace endpoint is impossible at the database level, not merely discouraged in code. All foreign keys use `ON DELETE RESTRICT`; a `CHECK` forbids self-links; a `UNIQUE (workspace_id, source, target, type)` index gives each directed relationship one stable identity.
  - **Idempotent create with in-place restore.** Creating a relationship that does not exist inserts it (`created`); one that exists and is active is an idempotent `already_exists`; one that exists but was unlinked is **restored in place** (`restored`) — **never a new id**. The uniqueness index is the final backstop against concurrent duplicate inserts (a losing racer re-reads and reconciles).
  - **Atomic active-endpoint enforcement.** The active-endpoint requirement is folded into the mutating statement itself, not left to a separate pre-check that a concurrent soft-delete could invalidate: `create` inserts via `INSERT ... SELECT ... WHERE EXISTS (source active) AND EXISTS (target active)`, and `restore` (and the create-after-unlink restore) re-activate only when the same correlated `EXISTS` clauses hold. If the statement affects no row, the repository re-reads and returns a defined outcome (already-active/already-exists) or `EntityLinkEndpointNotFoundError` — never a generic invariant failure. Lifecycle mutations are also concurrency-idempotent (conditional-update-then-reconcile).
  - **Unlink is reversible soft deletion; endpoint deletion does NOT cascade.** Unlinking sets `deleted_at` (idempotent; endpoints untouched). Crucially, **soft-deleting an endpoint entity does not delete or soft-delete its link rows**. Instead, normal link queries exclude a link when *either* endpoint is soft-deleted; restoring the endpoint makes still-active links visible again with their **original id**; and a relationship explicitly unlinked before/during entity deletion **stays unlinked** after restoration. There is no public hard-delete.
  - **Dedicated, scope-bound cursor.** Link pagination uses its **own versioned cursor** (not the entity cursor), bound to `(version, workspaceId, anchorEntityId, direction filter, type filter, createdAt, linkId)`. A cursor from another workspace, anchor, direction or type filter is rejected as a typed invalid-cursor error; ordering is deterministic on `(created_at, id)`. Cursors are treated as untrusted input and are neither signed nor encrypted — ids are scope identifiers, not secrets (consistent with ADR-010). The payload is base64url over **UTF-8 bytes** (so Unicode ids paginate) and decoded with a **fatal** UTF-8 decoder, so malformed bytes are rejected as an invalid cursor rather than silently substituted with U+FFFD.
  - **No N+1 counterpart lookup.** `listForEntity` returns each link with its `direction` and the **active counterpart entity**, fetched via a single joined query, so a caller can render/navigate without a second unscoped read.
- **Why soft-hide over destructive cascade on entity soft-delete.** Cascading a soft-delete from an entity onto every one of its links would **destroy relationship information the product exists to preserve**: after restoring the entity we could not tell which links were genuinely unlinked by the user from which were merely swept up by the cascade, so restoration could not faithfully rebuild the graph. Hiding links whose endpoint is inactive (a read-time filter) keeps the relationship data intact and makes entity delete/restore **losslessly reversible**, while still ensuring a "dangling" link to a hidden entity never surfaces. It also keeps entity soft-delete O(1) instead of fanning out writes across the link graph, and it leaves `EntityRepository.softDelete()` unchanged — deletion behaviour stays centralised in link queries rather than smeared across every module's mutations.
- **Consequences.** *Easy:* any active entity links to any other in its workspace; the same row is queryable from either end; duplicates are impossible; cross-workspace links are structurally impossible; entity delete/restore is reversible and lossless; modules get a correct-by-construction, workspace-scoped repository with no unscoped path. *Hard:* the both-direction listing is a `UNION ALL` of an outgoing and an incoming join that must stay index-backed; the parent `UNIQUE (workspace_id, id)` key is an extra index on the hot `entities` table; link-type governance currently lives only in kernel validation until FND-06 adds registration. *Accepted:* a read-time "either endpoint inactive → hidden" filter on every link query, in exchange for lossless, centrally-governed deletion behaviour and no cascade writes.
- **Atomicity & the FND-05 boundary.** FND-04 creates and mutates only EntityLink records; it introduces **no** Activity events and **no** generic mutation/transaction coordinator. [FND-05](../roadmap/ROADMAP_V2.md#-fnd-05--shared-activity-model) will introduce the shared Activity model and define how entity/link mutations and activity appends are coordinated atomically; entity mutations are deliberately **not** pre-modified for Activity here.
- **Alternatives considered.**
  - *Bespoke foreign-key columns per module.* Rejected (ADR-002): rigid, invisible to other modules, no shared lifecycle — the exact fragmentation the kernel primitive removes.
  - *Two rows per relationship (A→B and B→A).* Rejected: duplicates every write and invites the two rows to drift out of sync; "bidirectional" is a *query* property, achievable from one row.
  - *Undirected canonical pair with sorted endpoint ids.* Rejected: erases direction, which is meaningful (`produced_task` is not its own inverse); a reversed relationship may mean something different and must stay distinct.
  - *Free-form tags instead of typed links.* Rejected: untyped and not governable; link types must be stable machine identifiers, not uncontrolled labels.
  - *Embedding links as JSON arrays on entities.* Rejected: not queryable from the other end without scanning, no referential integrity, no shared lifecycle — contradicts the relational, integrity-enforced kernel (ADR-009).
  - *Destructive cascade when an entity is soft-deleted.* Rejected (see "Why soft-hide" above): lossy and irreversible.
  - *Treating links as ordinary entities.* Rejected: a link would gain a title, type, soft-delete-as-entity and even cross-links to itself, and "the relationship" would become a row inside the thing it relates — circular; a dedicated relationship record with database-enforced composite keys is a genuine boundary.
  - *Adding link behaviour directly to each module.* Rejected: duplicates integrity/lifecycle logic per module and re-creates bespoke relationships; links are a kernel primitive precisely so a link made in one module is visible in every other with no extra work.

## ADR-012: Activity Persistence and Atomic Mutation Recording
- **Status:** Accepted.
- **Context.** [ADR-005](#adr-005-shared-activity-model) established a **single append-only Activity model** — every meaningful mutation appends a uniform event; Timeline and the Activity Feed render it; it doubles as an audit trail — but left the concrete persistence and, critically, the *coupling between a domain mutation and its event* open. [FND-05](../roadmap/ROADMAP_V2.md#-fnd-05--shared-activity-model) makes Activity real on the existing kernel: the [entity substrate](#adr-009-data-kernel-storage) (FND-02), the [enforced workspace boundary](#adr-010-server-side-workspace-context) (FND-03) and [EntityLinks](#adr-011-entitylink-persistence-and-lifecycle) (FND-04). [ADR-011](#adr-011-entitylink-persistence-and-lifecycle) deliberately deferred to FND-05 the question of how a mutation and its event are recorded atomically. This ADR **implements** ADR-005; it does not replace it. The Module Registry ([FND-06](../roadmap/ROADMAP_V2.md#-fnd-06--module-registry)) and the Timeline/Activity-Feed UI are out of scope.
- **Decision.**
  - **One shared stream for all modules.** A single STRICT `activities` table holds every event for every entity type and module — no entity-specific history tables, no EntityLink-specific logs, no per-module audit or timeline tables. Per-module event logs are prohibited.
  - **Append-only through the application contract.** The module-facing `ActivityRepository` (`app/kernel/activity`) is **read-only**: `getById`, `listForWorkspace`, `listForEntity` — no update, delete, soft-delete or restore. An event is a historical fact; once appended it is immutable. Events are appended ONLY as the atomic side effect of a meaningful domain mutation, through an internal recording seam used by the D1 repositories, never through a module-facing method. There is no `updated_at`/`deleted_at` on `activities`. Database triggers to block administrative deletion were considered and **not** added (brittle migration/testing cost for little gain over the read-only contract + `ON DELETE RESTRICT`); the non-negotiable guarantee — that no normal application or module repository can mutate or remove an event — is met by the contract. Retention, archival and administrative purge policy are explicitly OUT OF SCOPE.
  - **Normalised subject association, one *or many* entities per event.** Which entities an event relates to is recorded in a separate STRICT `activity_subjects` table (`workspace_id, activity_id, entity_id, role`), never as a single entity id embedded on the event row. This is required because one EntityLink mutation relates to BOTH endpoints: a single `entity_link.*` event carries a `source` subject and a `target` subject, so the same event appears in both entity timelines while remaining one event. A composite primary key `(workspace_id, activity_id, entity_id)` forbids duplicate associations; composite foreign keys `(workspace_id, activity_id) → activities (workspace_id, id)` and `(workspace_id, entity_id) → entities (workspace_id, id)` (both `ON DELETE RESTRICT`) make cross-workspace subjects structurally impossible and preserve a deleted entity's Timeline (soft-delete never removes the entity row).
  - **Atomic recording via one D1 batch, coupled to the changed-row outcome.** A domain mutation and its Activity append execute as ONE `D1Database.batch()` — a single SQL transaction. Per the official Cloudflare D1 documentation and proven in the real Workers/D1 integration suite, batched statements run sequentially and **a failure in any statement rolls back the entire sequence**. The domain statement is FIRST and uses `RETURNING`. The event insert is guarded `WHERE changes() > 0` and each subject insert `WHERE EXISTS` the just-inserted event — so the append happens **iff the domain statement actually changed a row**. This makes the guarantees hold without a "log after the write" anti-pattern: an Activity-insert failure (e.g. a duplicate event id) rolls back the domain change; a no-op or losing-race mutation (its conditional statement changed nothing) appends nothing; and under concurrency only the caller whose statement changed the row appends an event. The chosen mechanism (`changes()` carried across a batched statement, and per-statement `meta.changes`) is proven against real D1 in `test/kernel/activity-atomic.test.ts` before being relied upon. `never do await repository.update(); await activity.append();` — the two are one statement batch.
  - **Trusted, server-derived actor context.** Every event carries `actor: { type, id }`. The actor is established at the composition boundary and threaded into the mutation repositories (`resolveWorkspaceScope`); it is NEVER accepted through a module method parameter, so module code cannot spoof it. `type` is a validated, open machine identifier (e.g. `system`, `user`, `ai`, `import`, `integration`) — not a database enum or a closed hard-coded list — and `id` is nullable. Until [FND-09](../roadmap/ROADMAP_V2.md#-fnd-09--app-shell-routing--auth) provides authenticated users, the boundary supplies `{ type: "system", id: null }`; FND-09 swaps in an authenticated `user` actor behind the same seam WITHOUT a schema change.
  - **Validated, branded event type; small validated payload.** `ActivityType` is a branded, non-empty, bounded, structurally-validated lowercase dotted identifier (e.g. `entity.created`, `entity_link.unlinked`), stored verbatim — no database enum, no display label in the kernel, so future modules add types without a migration. Each event carries a small JSON-object payload validated by a shared helper: it must be a JSON object, is recursively checked (rejecting functions, symbols, `undefined`, `bigint`, cyclic structures and non-finite numbers), is bounded in nesting depth and in **encoded byte size**, and is serialised exactly once on write and re-parsed/validated on read (corrupt stored JSON becomes a typed `ActivityPayloadError`). Payloads explain the event only — never a full entity snapshot, never a replacement for a domain table.
  - **One trusted timestamp; deterministic newest-first ordering.** The domain record's mutation timestamp and the event's `occurred_at` derive from the SAME injected clock call. Ordering is deterministic on `(occurred_at, id)` and queries are newest-first. Both the workspace feed and entity Timeline are bounded and **cursor-paginated** with a dedicated, versioned cursor bound to `(version, workspaceId, scope-kind, entityId?, type-filter, occurredAt, activityId)` — a workspace cursor cannot be replayed on an entity Timeline (or vice versa), an entity-A cursor cannot be used for entity B, and a filtered cursor cannot be reused under another filter. Cursor contents are untrusted input (base64url over UTF-8, fatal decoding), validated on decode, and every value reaching SQL is bound. Queries avoid N+1 by loading a page's subjects in one `IN (...)` query.
  - **No backfill.** No Activity history is fabricated for entities or links created before migration `0004`. The Activity stream begins when FND-05 is deployed.
- **Consequences.** *Easy:* uniform history/audit everywhere for free; one event can relate to many entities; a link event surfaces in both endpoints' timelines; the domain change and its audit event cannot diverge; deleted entities keep their Timeline; modules get a correct-by-construction, workspace-scoped, read-only Activity API with no unscoped or write path to misuse. *Hard:* every meaningful mutation now issues a `batch()` instead of a single statement; the `changes()`-guarded append and the atomic-batch contract must be preserved by every future repository that records Activity; entity `update` uses a bounded optimistic retry so a concurrent title change cannot record a stale before/after. *Accepted:* the write-time cost of an extra event+subject insert per meaningful mutation, in exchange for a trustworthy, atomic, append-only history.
- **Alternatives considered.**
  - *Per-module history/event tables.* Rejected (ADR-005): fragmentation and duplicated UI; a link made in one module would be invisible to another's history.
  - *Embedding a single entity id on the event row.* Rejected: cannot represent a link event that relates to both endpoints; a normalised subject table is required.
  - *"Write the domain change, then append the event" (two calls).* Rejected: a crash or error between them persists the domain change while losing its audit event — exactly the integrity failure the audit trail must not have. One atomic batch removes the window.
  - *Deciding "did it change?" only by a pre-read, then appending unconditionally.* Rejected: under concurrency the pre-read can be stale, so a losing racer would append a spurious event. Guarding the append on the domain statement's `changes()` couples the event to the *actual* changed-row outcome.
  - *`changes()` used without proof.* Rejected as-is: its behaviour across D1 batched statements is verified in real integration tests before being relied on (and per-statement `meta.changes` is inspected rather than assumed).
  - *Deduplicating events after the fact.* Rejected: correctness must come from coupling the event to the state change, not from a post-hoc cleanup that races and can leave the audit trail wrong.
  - *A database enum (or hard-coded closed list) for event/actor/role identifiers.* Rejected: every new module event would need a schema migration; validated branded identifiers are an open, governable contract (FND-06 may later register them).
  - *Storing full entity snapshots in the payload, or treating Activity as the source of truth.* Rejected: DalyHub is not event-sourced — domain tables remain the source of truth; Activity is an immutable history/audit stream derived atomically from meaningful mutations. Payloads carry only what explains the event, bounded in size.
  - *Database triggers to hard-block deletion.* Considered; not adopted for FND-05 — brittle migration/testing/maintenance cost over the read-only contract plus `ON DELETE RESTRICT`, which already prevents any module-facing removal. May be revisited if an administrative purge surface is ever built.

## ADR-013: Module Registry Contract and Discovery
- **Status:** Accepted.
- **Context.** [ADR-007](#adr-007-module-registry) established that modules **self-register** their capabilities (routes, entity types, commands, search providers, settings) through a Module Registry, so adding a module never means editing a central switch statement. [FND-06](../roadmap/ROADMAP_V2.md#-fnd-06--module-registry) makes that mechanism real on the existing kernel: the [entity substrate](#adr-009-data-kernel-storage) (FND-02), the [workspace boundary](#adr-010-server-side-workspace-context) (FND-03), [EntityLinks](#adr-011-entitylink-persistence-and-lifecycle) (FND-04) and [Activity](#adr-012-activity-persistence-and-atomic-mutation-recording) (FND-05). FND-04 and FND-05 deliberately deferred link-type and Activity-type *registration* to here. This ADR **implements** ADR-007; it does not replace it. FND-06 builds the registry and its discovery mechanism ONLY — it does not build navigation, route composition, the command palette, global search or settings surfaces; those are [FND-09](../roadmap/ROADMAP_V2.md#-fnd-09--app-shell-routing--auth) and later, which consume the typed seams established here.
- **Decision.**
  - **Trusted, compiled-in registration — not a plugin marketplace.** Modules are trusted application code compiled into DalyHub. There is **no** remote module loading, `eval`, user-provided JavaScript, arbitrary package installation, request-controlled registration, or database-controlled executable module definitions. The registry is assembled only from source-resident manifests at build time.
  - **One declarative, side-effect-free manifest per module.** Each module exposes a single `ModuleDefinition` (a small identity header — `id`, `name`, optional `description`/`order` — plus readonly capability collections). A manifest is plain data; evaluating it must NOT access D1, resolve a workspace, read request data, perform network requests, mutate global state, eagerly import module UI, or run a command/search. The `defineModule(...)` helper is a typed identity function that performs no registration and no hidden side effects — there is deliberately no `registerModule(...)` writing to a mutable singleton at import time.
  - **Automatic, build-time discovery — no central list.** Discovery uses a constrained Vite `import.meta.glob("./*/module.ts", { eager: true })` over one documented convention: `app/modules/<module-id>/module.ts` with a single `default` export. Vite transforms the glob into static imports at build time, so discovery is deterministic, trusted, build-time-only code that works under Vite, React Router and Cloudflare Workers with **no Node filesystem access in the deployed Worker**. Manifests are eagerly imported (they are small and side-effect free); the heavy module UI they reference stays lazy. There is **no** manually-maintained `const modules = [projectsModule, …]` array. The Vite-specific glob lives in `app/modules/discover-modules.ts`; the pure collection/validation (`collectModuleDefinitions`) and assembly (`createModuleRegistry`) live in the kernel with no Vite dependency.
  - **Immutable registry, validated once.** `createModuleRegistry` validates every manifest and descriptor, fails fast on any collision, prepares lookup maps once, and returns a **frozen** registry. There is no mutable global service locator and no registration after construction. Callers cannot change registry state by mutating a source manifest (validators build fresh defensive copies), a returned array (frozen), a nested contribution (deep-frozen) or a lookup result. Ordering is deterministic and independent of filesystem enumeration order: modules sort by optional `order` (specified first, ascending) then id; contributions keep authored order; collision detection runs over that sorted order, so a duplicate failure is deterministic. Unknown lookups uniformly return `null`.
  - **Static declaration separate from runtime execution.** A command or search descriptor carries a handler, but building the registry never invokes it. Handlers run only when explicitly called through a typed runtime seam (`ModuleRuntimeContext`) and receive their dependencies **explicitly** — they must not capture a global D1 binding, a global workspace, mutable request state, a caller-supplied workspace id or a caller-supplied Activity actor. FND-09 and later work supply the authenticated, workspace-scoped context behind that seam.
  - **Governance, not database enums.** The registry governs application discoverability and ownership of declared machine identifiers. It does NOT turn entity, link or Activity types into database enums, add a migration, or couple the D1 repositories to a registry singleton. The existing open, validated identifier contracts from FND-02/04/05 remain the source of truth — the registry **reuses** their validators (`validateEntityType`, `parseEntityLinkType`, `parseActivityType`) rather than duplicating them — and registry membership is **not** yet a precondition for low-level Entity/EntityLink/Activity persistence (the real product modules and FND-07 types do not exist yet).
  - **Module identity is a stable slug.** `ModuleId` is a branded, validated lowercase hyphenated slug (`projects`, `notes`, `day-diary`), never derived from a display label; empty, whitespace, uppercase, dotted, underscored, path-traversal and over-length ids are rejected, and duplicate ids fail fast. Command/route/search/setting ids are **namespaced under the declaring module** (`projects.create`), which both makes them globally unique and lets the registry verify ownership.
  - **Kernel-reserved Activity types.** The kernel-owned lifecycle event types (`entity.created`, `entity.updated`, `entity.deleted`, `entity.restored`, `entity_link.created`, `entity_link.unlinked`, `entity_link.restored`) are reserved; a module registering one fails. The kernel is not pretended to be a userland module.
  - **Route contributions are declarative and adaptable.** A route contribution is the minimum FND-09 needs — stable id, path or index status, optional parent, optional shell metadata, and a route-module reference that is stored and never invoked to build the registry. The registry validates duplicate route ids, conflicting paths under the same effective parent, empty/query/hash/traversal paths, and unresolved/self/cross-module/**cyclic** parents (cross-module parenting is unsupported in FND-06; a parent cycle would leave routes un-composable, so it is rejected). A platform adapter (`app/platform/modules/route-contribution-adapter.ts`) resolves the flat route list into a nesting tree FND-09 maps onto React Router — kept out of the kernel so the kernel stays free of router types.
    - **FND-09 refinement (see [ADR-016 §5.10](#adr-016-cloudflare-access-identity-app-shell-and-registry-driven-routing)).** FND-06 modelled the route-module reference as a **lazy `() => import(...)` thunk**. FND-09's compatibility spike proved React Router v8 **framework mode** composes its route tree from **build-time file references** in `app/routes.ts` (used for per-route type generation, SSR and production code splitting) and offers no runtime import-thunk seam — so a thunk cannot drive framework routing. The contribution therefore carries a declarative, module-relative **`file`** string (e.g. `routes/index.tsx`), validated (`validateRouteFile`) to be traversal-free and inside the owning module, which the platform React Router adapter resolves to `app/modules/<module-id>/<file>`. This is still fully lazy: the string is plain data, so the registry never loads a page component and React Router code-splits each route module. The route composition and navigation remain automatic and registry-driven; no central per-module list exists.
- **Consequences.** *Easy:* a new module is a new directory plus one manifest — routes/commands/search/settings/types compose automatically with no central edits; the registry is correct-by-construction, immutable and cheap to build; FND-09 consumes stable typed seams. *Hard:* the registry contract must stay stable and well-specified, and every capability's validation/collision rule must be kept correct as real modules arrive; the discovery glob must remain the one convention. *Accepted:* a build-time discovery indirection and per-descriptor validation cost, in exchange for decoupled, independently-implementable modules.
- **What FND-06 deliberately does NOT build.** Navigation, app shell, command-palette UI, global-search UI, settings UI/persistence, route guards, authentication, real product modules or the FND-07 spine types, search indexing, D1 migrations, and any dynamic/third-party/remote module loading. The registry exposes the typed seams these will consume; it does not build them.
- **Alternatives considered.**
  - *A manually-maintained central module array.* Rejected: it is exactly the central bottleneck/merge-magnet ADR-007 removes; automatic glob discovery is the whole point.
  - *A mutable global registry with `registerModule()` at import time.* Rejected: import-order-dependent, non-deterministic, a test-isolation and concurrency hazard, and it hides the boundary. A single validated, immutable snapshot is safer and clearer.
  - *Branding capability identifiers as database enums.* Rejected (consistent with FND-02/04/05): every new module type would need a migration; validated branded identifiers are an open, governable contract.
  - *Eagerly importing route/command/search modules during discovery.* Rejected: it would load module UI and run handlers just to build the registry, breaking the performance budget and the static/runtime separation. Manifests are eager; everything heavy is lazy.
  - *Runtime/remote/plugin module loading.* Rejected: DalyHub is a single-owner trusted application, not a plugin marketplace; remote code loading is a security non-starter.
  - *A dependency-analysis framework to enforce module boundaries.* Rejected as over-engineering: a small repository test that resolves import specifiers against the `app/modules` tree enforces "no cross-module internal imports" without a heavy tool.

## ADR-014: Spine Hierarchy, Completion and Rollup Semantics
- **Status:** Accepted. Concretely implements [ADR-001](#adr-001-area-hierarchy) (it does not replace it), and builds on [ADR-009](#adr-009-data-kernel-storage) (Entities), [ADR-011](#adr-011-entitylink-persistence-and-lifecycle) (EntityLinks), [ADR-012](#adr-012-activity-persistence-and-atomic-mutation-recording) (Activity) and [ADR-013](#adr-013-module-registry-contract-and-discovery) (Module Registry).
- **Context.** [FND-07](../roadmap/ROADMAP_V2.md#-fnd-07--area--goal--project--task-hierarchy) builds the first real DalyHub domain model on top of the finished kernels. ADR-001 fixed the *shape* (Area → Goal → Project → Task, Goals optional, Areas permanent) but not its persistence, its invariants, its completion model or how progress is derived. This ADR fixes those concretely and minimally, without inventing the future Projects/Tasks feature set. The permitted structure is: an Area holds Goals, Projects and Tasks; a Goal holds Projects; a Project holds Tasks; a Project may sit directly under an Area or advance a Goal; a Task may float directly in an Area or belong to a Project.
- **Decision.**
  - **The spine is a kernel-level domain contract.** Its invariants live in a shared spine kernel (`app/kernel/spine`) and a workspace-bound domain repository (`SpineRepository`), not in the four product modules. The four modules (`areas`, `goals`, `projects`, `tasks`) register their metadata through FND-06, but hierarchy correctness is written once, in the kernel — never duplicated across four modules.
  - **Generic entities remain the shared identity substrate.** Every Area, Goal, Project and Task is an ordinary `entities` row and keeps the shared header (id, workspace, type, title, timestamps, `deletedAt`). The only additive domain state is a single `spine_records` table carrying `(workspace_id, entity_id, kind, completed_at)`. No four replacement identity tables.
  - **Hierarchy relationships are EntityLinks.** Structural parentage uses the existing EntityLink primitive with five stable, directed types — `goal.belongs_to_area`, `project.belongs_to_area`, `project.advances_goal`, `task.belongs_to_area`, `task.belongs_to_project` — always directed **child → parent** (child is the source). No bespoke parent foreign-key columns, no JSON hierarchy, no duplicate second parent reference.
  - **Exactly one active parent.** Every active Goal has one Area parent; every active Project has one Area **or** Goal parent; every active Task has one Area **or** Project parent; an Area has none. The database enforces *at most one active structural parent per child* with a **partial unique index** over `entity_links (workspace_id, source_entity_id)` restricted to the five structural types where `deleted_at IS NULL`. The `SpineRepository` enforces the valid parent kind and the existence of one parent; an invalid hierarchy write fails atomically and leaves no orphan behind.
  - **Minimal completion model.** Completion is a single `completedAt: Date | null`; reopening clears it. Areas never complete (enforced by a DB CHECK). Completion is independent of soft-deletion, and never cascades to parents or children — the user stays in control. FND-07 deliberately does **not** add status, priority, dates, descriptions, ordering, weights, milestones or percent overrides.
  - **Rollups are derived, not stored.** Progress is computed from current active descendants with a small, fixed number of bounded SQL queries — never cached counters, never "rollup changed" events. A Project reports its active direct Tasks; a Goal reports its active direct Projects and all active Tasks under them; an Area reports its active Goals, all active Projects directly under it or under its Goals, and all active Tasks directly under it or under those Projects. `ratio = completed / total`, or `null` when `total = 0` (never `NaN`, never 100% for an empty container).
  - **The domain repository is authoritative.** The four spine entity types and the five structural link types are **reserved**: the generic Entity and EntityLink repositories refuse to create, mutate lifecycle state or structurally mutate a reserved record/link, raising a typed `ReservedEntityTypeError` / `EntityLinkReservedTypeError`. Generic *reads* still return them. This is enforced with static reserved-identifier sets shared from the spine kernel — **not** a mutable ModuleRegistry singleton, and general persistence does not depend on registry membership.
  - **No event sourcing.** The domain tables are the source of truth. Activity remains an immutable audit/history stream derived atomically from successful mutations; current hierarchy and completion state are never reconstructed by replaying Activity.
  - **Atomic mutations.** Creating a child atomically writes the `entities` row, the `spine_records` row, the structural EntityLink, and the `entity.created` + `entity_link.created` events (all subjects) in ONE `D1Database.batch()`; any failure rolls everything back. Parent validity is folded into the mutating SQL (`INSERT … WHERE EXISTS (active parent of the required kind)`) so a parent soft-deleted between read and write cannot commit an orphan. Move records `entity_link.unlinked` then `entity_link.created`/`entity_link.restored`; complete/reopen record the kind's `*.completed`/`*.reopened` event on the same clock as the domain timestamp. The existing `D1ActivityRecorder` is reused for every append.
- **Consequences.** *Easy:* one correct hierarchy implementation the future Areas/Goals/Projects/Tasks UIs and the alignment view (`AREA-03`) consume through a single typed seam; rollups are always consistent because they are derived; a spine record is a normal entity everywhere else in the system. *Hard:* the spine invariants are load-bearing and must stay correct under concurrency (guaranteed by conditional SQL and the partial unique index, proven with real Workers/D1 tests), and the reserved-type guard must be kept in step with the shared identifier set. *Accepted:* a deliberately minimal completion model (just `completedAt`) and derived-only rollups, in exchange for a small, unambiguous foundation later modules extend rather than fight.
- **Alternatives considered.**
  - *Four replacement identity tables (one per spine type).* Rejected: it abandons the shared entity substrate (ADR-009), breaks uniform EntityLinks/Activity/search, and duplicates identity four ways.
  - *Bespoke `parent_id` foreign-key columns on each spine table.* Rejected: it duplicates a relationship EntityLinks already model, and (ADR-011) hierarchy is exactly the cross-spine relationship the link primitive exists for.
  - *A stored/cached rollup counter per container.* Rejected: cache-invalidation hazard and a source of drift; the hierarchy is fixed-depth and personal-scale, so bounded derived queries are correct and cheap.
  - *Enforcing reserved types via the mutable ModuleRegistry.* Rejected: it would couple low-level persistence to a runtime singleton (ADR-013 explicitly keeps them decoupled); a static shared identifier set is simpler and safe.
  - *Event-sourcing the hierarchy from Activity.* Rejected: Activity is an append-only audit stream (ADR-012), not the system of record; replay is neither needed nor desirable at this scale.

## ADR-015: Markdown Source and Safe Rendering Pipeline
- **Status:** Accepted. Concretely implements [ADR-006](#adr-006-markdown-strategy) (Markdown strategy) — it does not replace it — and builds on the [platform/toolchain](#adr-008-initial-application-platform-and-toolchain) (Cloudflare Workers, ADR-008).
- **Context.** [FND-08](../roadmap/ROADMAP_V2.md#-fnd-08--markdown-pipeline) builds the shared foundation for **all** long-form text in DalyHub — future Notes, Diary entries and entity descriptions. ADR-006 fixed the *strategy* (author/store Markdown; render through one shared sanitising renderer; raw HTML untrusted) but not the concrete contract, dependency choice, sanitisation policy, URL/image handling or the rendering boundary. DalyHub holds the most private data a person has and renders **untrusted** user Markdown, so the renderer is security-critical ([AGENTS.md §17](../../AGENTS.md#17-security-requirements)) and must run in the Cloudflare Workers runtime (no Node/DOM). This ADR fixes those concretely. FND-08 builds the pipeline **only** — no persistence tables, no product routes, no module, no editor, and none of the later Notes/Diary/description features.
- **Decision.**
  - **Markdown is the durable source (§4.1).** Long-form content is stored as plain Markdown source. The pipeline never silently trims, reflows, rewrites headings, changes list markers, strips unsupported syntax from storage, converts Markdown to a proprietary JSON model, persists generated HTML, or makes a rich-text editor document the source of truth. A validated `MarkdownSource` (a branded string produced by `parseMarkdownSource`) preserves the accepted text **byte-for-byte** and remains exportable. Raw HTML may remain present in the stored source, but it must never become executable DOM when rendered.
  - **Rendering is derived (§4.2).** Rendered HTML is generated from Markdown when needed and is **disposable**. FND-08 adds no rendered-HTML column, no Markdown cache, no database trigger, no render-version column, no background rendering and no cache-invalidation infrastructure. Caching may be reconsidered later only if measurement proves it necessary.
  - **One shared renderer (§4.3).** All modules use one supported rendering API (`MarkdownRenderer` in `app/kernel/markdown`, implemented by `app/platform/markdown`). Notes, Diary and descriptions may **not** independently choose another parser, another sanitiser, another URL/raw-HTML policy, a direct `dangerouslySetInnerHTML`, or ad-hoc regex sanitisation. The pipeline is a **kernel capability**, not a Notes-only utility.
  - **Sanitise after parsing (§4.4).** The pipeline is AST/document-tree based (`unified`: `remark-parse` → `remark-gfm` → `remark-rehype` → `rehype-sanitize` → `rehype-stringify`). Raw HTML is **disabled at conversion** — `remark-rehype` runs with `allowDangerousHtml: false`, so raw HTML nodes are dropped before they can become DOM — and the resulting tree then passes through a **strict allowlist** sanitiser as defence in depth. No regex sanitisation; no trusting parser output merely because raw HTML is off; no browser-only DOM sanitiser needing JSDOM/a full DOM inside Workers; no user code execution; no embedded-HTML evaluation; no arbitrary plugins from user content.
  - **Controlled HTML sink (§4.5).** The renderer produces a branded `SanitizedMarkdownHtml` string that **only** the shared renderer can mint. One small React component (`MarkdownContent`, `app/shared/markdown`) consumes it and is the **only** application source permitted to use `dangerouslySetInnerHTML`. A repository test fails if a second usage is introduced anywhere in `app/`.
  - **No remote content execution or loading (§4.6).** Rendering causes **no** automatic third-party request. FND-08 renders no `<img>` from user Markdown, and allows no iframe, video/audio, object/embed, content-supplied SVG/MathML, Mermaid/executable diagrams, link-metadata/oEmbed fetching, or any URL fetch during rendering. Image Markdown becomes safe **non-embedded** content — a labelled safe link (or plain alt text when the URL is unsafe) — never an auto-loaded image. A future Assets/R2 feature will add an explicit trusted-attachment policy.
  - **Stateless and deterministic (§4.7).** Rendering has no network, database, environment, workspace or request dependency, mutates no global state, and produces identical output for the same source. The processor is an immutable constant reused across calls — not a mutable singleton or service locator.
  - **Restrained profile & policies.** Supported: CommonMark plus the useful GFM subset (paragraphs, headings, emphasis/strong, strikethrough, ordered/unordered/nested lists, blockquotes, inline/fenced code, thematic breaks, hard/soft breaks, links, autolinks, tables, task lists). Excluded (this item): raw executable HTML, custom attributes/inline styles, script/style/iframe/object/embed/form/button/inputs, SVG/MathML, Mermaid, directives, custom JS plugins, wikilinks, mentions, record-linking, math, syntax highlighting, embedded media, and **footnotes** (GFM's footnotes are stripped, since supporting them would require pipeline-generated element `id`s — the DOM-clobbering surface this item avoids by also generating **no** heading ids). Raw HTML is **ignored** in rendered output (never selectively "safe-looking" HTML). The **URL policy** is an allowlist — relative paths, fragments, `http`, `https`, `mailto`, `tel` — that mirrors browser href normalisation (strips tab/newline/CR, trims whitespace, rejects remaining control chars, rejects protocol-relative forms) so obfuscation cannot smuggle a scheme; `target` is never accepted and external links are not auto-opened in a new tab. The **sanitisation allowlist** is a single frozen schema permitting only the semantic elements of the profile (plus the GFM-owned task-list classes and a disabled task-list checkbox); no user-controlled `id`/`style`/`class`/`name`/`target`/`src`/`srcdoc`/`on*`/`data-*`/`aria-*`. Code is rendered semantically and always escaped, with **no** syntax highlighting and no language class. Task-list checkboxes are **display-only** and are explicitly **not** DalyHub Task records. The **source size limit** is 1 MiB measured in **UTF-8 bytes**. Errors are typed and safe (never leaking source, generated HTML, parser internals, paths or environment).
- **Consequences.** *Easy:* every future long-form surface (Notes, Diary, descriptions) gets portable, exportable, XSS-safe rendering through one hardened seam with a single reviewed HTML sink; the branded types make "unsanitised HTML reaching the DOM" unrepresentable; the renderer is deterministic and trivially testable, and adds nothing to the initial client bundle until a module imports it. *Hard:* very rich formatting (embeds, raw HTML, media, math, diagrams, syntax highlighting) is deliberately unavailable until later, explicitly-designed items; the parser/sanitiser stack is a new (permissively-licensed) dependency tree to keep patched. *Accepted:* a restrained profile and a first runtime dependency family, in exchange for portability, safety and one shared implementation.
- **Alternatives considered.**
  - *A proprietary block/JSON document model (Notion-style).* Rejected (ADR-006): lock-in and heavier; Markdown stays portable, diff-able and export-safe.
  - *Persisting rendered HTML (or a render cache) alongside source.* Rejected: rendered output is derived and disposable; a cache adds invalidation/drift hazards with no proven need at personal scale — revisit only on measurement.
  - *`react-markdown` (render straight to a React element tree).* Reasonable and avoids an HTML string, but pairs with the same remark/rehype/sanitiser stack while pulling React into the render path and encouraging per-call plugin arrays. Rejected for the foundation: a string-producing `unified` pipeline with one branded sink is smaller, framework-agnostic (usable off the React path, e.g. export), and keeps the sanitiser non-optional. Avoided adding **both** stacks.
  - *`DOMPurify` (+ JSDOM) for sanitisation.* Rejected: DOMPurify is DOM-based and needs a real DOM/JSDOM to run server-side, which is a poor fit (and weight) inside Cloudflare Workers; `rehype-sanitize` (`hast-util-sanitize`) sanitises the hast tree with no DOM and is Workers-native.
  - *Regex-based sanitisation, or trusting the parser because raw HTML is off.* Rejected: regex HTML sanitisation is notoriously bypassable, and defence in depth requires an allowlist over the parsed tree even though raw HTML is already disabled.
  - *Rendering Markdown images as `<img>` (optionally proxied).* Rejected for FND-08: auto-loading remote images leaks IP/metadata, enables tracking pixels and mixed content; images become non-embedded safe content until a deliberate Assets/R2 attachment policy exists.
  - *A rich-text editor dependency (Tiptap/ProseMirror/CodeMirror/Lexical/Milkdown/Monaco) now.* Rejected: authoring is later work; FND-08 is the storage/render foundation. A future editor must still save Markdown **source**.
  - *Generating heading/footnote ids and anchor affordances.* Rejected for FND-08: pipeline-generated ids are a DOM-clobbering surface; anchor behaviour will be designed deliberately later.

## ADR-016: Cloudflare Access Identity, App Shell and Registry-Driven Routing
- **Status:** Accepted. Concretely implements the auth/shell/theme scope [ADR-007](#adr-007-module-registry) and [ADR-008](#adr-008-application-platform-and-toolchain) left open, and builds on [ADR-010](#adr-010-server-side-workspace-context) (server-side workspace context), [ADR-012](#adr-012-activity-persistence-and-atomic-mutation-recording) (Activity actor) and [ADR-013](#adr-013-module-registry-contract-and-discovery) (Module Registry). It refines ADR-013's route-contribution shape (see §5.10).
- **Context.** [FND-09](../roadmap/ROADMAP_V2.md#-fnd-09--app-shell-routing--auth) turns the completed kernel into a secured, navigable application: an app frame, registry-driven routing and navigation, authentication, authenticated workspace composition and basic theme infrastructure. DalyHub holds a person's most private data and is single-user first, so authentication is security-critical ([AGENTS.md §17](../../AGENTS.md#17-security-requirements)) and must run in the Cloudflare Workers runtime. ADR-008 pinned the toolchain but left identity open. This ADR fixes identity, the request-authentication boundary, the route-contract refinement framework mode requires, and the theme mechanism — without starting the Design System (DS-01) or any product module.
- **Decision.**
  - **§5.1 Cloudflare Access owns login and session lifecycle.** Cloudflare Access is DalyHub's initial identity-aware proxy and session provider. DalyHub implements **no** passwords, password resets, local login forms, OAuth callbacks, local session-cookie format, duplicate session store, users-table-to-authenticate, or third-party auth SDK. The Access application token is the session presented on each request — and DalyHub still validates it itself.
  - **§5.2 The application validates every protected request.** The Worker reads `Cf-Access-Jwt-Assertion` and cryptographically verifies the JWT with `jose` against the team's Access JWKS: signature, RS256 only, issuer (the team domain), audience (the AUD tag), `exp`/`nbf`, a valid `sub`, a valid verified `email`, and that the token is an identity token (a service token bearing `common_name`/empty `sub` is rejected). It never trusts the mere presence of a header, an unverified payload, `CF_Authorization`, a client-supplied identity header, a query/form field, or an unsigned dev token.
  - **§5.3 Single-owner enforcement.** A configured `OWNER_EMAIL` is required; comparison is trimmed-lowercase. DalyHub independently rejects a valid Access JWT for any other email, so an accidentally-broadened Access policy cannot grant access. The stable `sub` is the Activity actor id (never the email). The raw JWT never enters loader data, React context, HTML, logs, Activity payloads, client bundles or error messages.
  - **§5.4 No persisted user table yet.** Identity is an in-memory `AuthenticatedUser { subject, email }` / `AuthenticatedSession { user, issuedAt, expiresAt }`. No migration is added for a single owner; a persisted user/profile model waits until the product needs editable profile state or multiple users.
  - **§5.5 Authentication occurs before route execution.** Authentication runs at the **Worker request boundary**, guaranteed to run before every protected loader/action (not relying on parent-loader ordering). `/health` and static build assets stay public; everything that can return private data — including React Router data/manifest requests — requires authentication. Public paths are matched EXACTLY (`/health-anything` is not exempt).
  - **§5.6 Workspace remains server-derived.** Authentication answers *who*; workspace resolution answers *which trusted scope*. The session never selects a workspace via header, claim, param, query, cookie, form or JSON. The configured `DEFAULT_WORKSPACE_ID` resolves through the existing request-free resolver ([ADR-010](#adr-010-server-side-workspace-context)); FND-09 composes the validated session's `sub` into `Activity actor { type: "user", id: sub }`, threaded into the same entity/EntityLink/spine repositories. Module method calls cannot supply or override the actor; the `system` actor remains for genuinely system-initiated work.
  - **§5.7 Access logout** is a plain link to the Cloudflare-managed `/cdn-cgi/access/logout`; no local-cookie-deletion pretends to be logout, and the JWT is never in the logout URL.
  - **§5.8 Explicit development authentication.** A separate development authenticator behind the same `Authenticator` contract, enabled only through explicit trusted server-side config: production/default `AUTH_MODE=cloudflare-access`; `AUTH_MODE=development` is rejected unless `ENVIRONMENT` is explicitly `development`/`test`; a missing mode defaults to Cloudflare Access; an unknown mode fails closed. No "accept any JWT" path, no `import.meta.env.DEV` bypass, no request-driven activation. The dev identity is fixed server-side and validated. `.dev.vars` stays git-ignored; `.dev.vars.example` carries non-personal placeholders.
  - **§5.9 Registry-driven routing & navigation.** Modules own their routes; the shell composes route configuration and primary navigation from automatically-discovered module manifests. Adding a module route requires only a manifest entry plus the module-owned route file — never editing `app/routes.ts`, a central navigation array, or any switch statement.
  - **§5.10 Framework-compatible route descriptors (refines [ADR-013](#adr-013-module-registry-contract-and-discovery)).** React Router v8 framework mode composes routes from **build-time file references**; the FND-06 `lazy: () => import(...)` thunk cannot produce framework route config with type generation, SSR and code splitting. The route contribution is therefore a declarative module-relative **`file`** string (validated traversal-free and inside the owning module), which the platform adapter resolves to `app/modules/<module-id>/<file>`. `app/routes.ts` is evaluated by React Router's bare config loader (which cannot resolve the `~` alias), so route composition is driven from per-module `routes.manifest.ts` descriptors (pure data, type-only imports) globbed there; the same descriptors flow through the validated registry at runtime. The kernel stays free of React Router/Vite types; routes stay module-owned, code-split, automatic and free of central lists; regression tests cover the revised contract.
  - **§5.11 Theme infrastructure, not DS-01.** FND-09 provides only the `system`/`light`/`dark` mechanism: a validated `ThemePreference`, a server-read cookie applied to `<html data-theme>` during SSR (no flash, no client cookie read, no `localStorage`, no state library, no table, no Activity). DS-01 later owns the full token/colour/spacing/typography/elevation/motion system; FND-09 introduces only the minimal CSS variables a usable, legible shell needs.
- **Consequences.** *Easy:* the owner signs in through Cloudflare with no auth code to maintain; every protected request is cryptographically verified once at the boundary; module routes and navigation compose automatically; theme is flash-free and stateless; no new schema. *Hard:* the Worker must be the sole entry (an unprotected `workers.dev` origin would bypass Access — see [DEPLOYMENT.md](../development/DEPLOYMENT.md)); the JWKS/verification must stay correct and bounded; the route contract had to be refined. *Accepted:* dependence on Cloudflare Access as the identity provider, one new runtime dependency (`jose`), and configuration (team domain, AUD, owner email) that must be supplied at deploy time and never committed.
- **Alternatives considered.**
  - *A local auth stack (Auth.js/Lucia/Passport/Clerk/Supabase/Firebase, password login, a session store, a users table).* Rejected: Cloudflare Access already provides the identity-aware proxy and session; DalyHub would only duplicate it and add attack surface, secrets and schema for a single owner.
  - *Trusting the `Cf-Access-Jwt-Assertion` header (or `CF_Authorization`) without verifying the signature.* Rejected: a header/cookie can be spoofed on an unprotected origin; the token must be cryptographically validated in the Worker (issuer/audience/alg/time/owner).
  - *Authenticating in a root loader only.* Rejected: child loaders/actions can run without a strict guarantee they follow the root loader; the Worker boundary is the only place guaranteed to run before all protected data functions.
  - *Keeping the FND-06 lazy route thunk.* Rejected: incompatible with React Router framework-mode route configuration (§5.10); forcing it would mean a runtime second router or losing type-gen/SSR/code-splitting.
  - *Building DS-01 tokens now.* Rejected: out of scope; FND-09 ships only the theme mechanism and a restrained shell, leaving the visual language to DS-01.

## ADR-017: Design Tokens and the Shared Record Layout
- **Status:** Accepted. Delivers the [Design System](../design/DESIGN_SYSTEM.md) foundations [ADR-016 §5.11](#adr-016-cloudflare-access-identity-app-shell-and-registry-driven-routing) deliberately deferred, and builds on the [platform/toolchain](#adr-008-initial-application-platform-and-toolchain) (React on Cloudflare Workers) and the [spine hierarchy](#adr-014-spine-hierarchy-completion-and-rollup-semantics) whose records the layout will host.
- **Context.** [DS-01](../roadmap/ROADMAP_V2.md#-ds-01--design-tokens--theming) and [DS-02](../roadmap/ROADMAP_V2.md#-ds-02--shared-record-layout-header--summary--tabs) are the shared visual foundation every later module reuses. FND-09 shipped only a provisional shell and a `system`/`light`/`dark` theme *mechanism* (ADR-016 §5.11), explicitly leaving the token system and shared components to DS. DS-02 depends on and immediately consumes DS-01. They were implemented together as an **owner-approved, dependency-aligned exception** to the one-item-per-PR rule ([AGENTS.md §12](../../AGENTS.md#12-development-workflow)): DS-02 is the first real consumer that proves DS-01, and doing both at once avoided redoing the shell integration twice. Each item keeps independent acceptance evidence.
- **Decision.**
  - **§17.1 Tokens are the single source of truth for values.** Every design value (colour, typography, spacing, sizing, shape, elevation, motion, z-index, layout) is a CSS custom property in [`app/styles/tokens.css`](../../app/styles/tokens.css). Application CSS and components consume `var(--dh-*)`; hard-coding a raw value where a token exists is prohibited and tested.
  - **§17.2 Semantic naming.** Tokens are `--dh-<group>-<role>[-<variant>]`, named for purpose not appearance (`--dh-color-danger-surface`, never `--dh-red-500`). There is no exposed raw palette. Colour *fills* (`--dh-color-accent` + `--dh-color-on-accent`) are distinct from colour *foregrounds* (`--dh-color-accent-text`); feedback colours are a solid/surface/text triple.
  - **§17.3 One name, two theme maps.** Light and dark are maps over the same semantic names; only colour/elevation tokens differ. The FND-09 flash-free `data-theme` mechanism is preserved (server-rendered from the cookie), with `system` following `prefers-color-scheme` and reduced-motion collapsing durations. A TS registry ([`app/shared/tokens`](../../app/shared/tokens)) mirrors token names and the colour maps for typed consumption and tests.
  - **§17.4 No new UI framework.** DS-01/DS-02 add **zero runtime dependencies**: plain CSS custom properties, CSS container queries for responsive adaptation, and React components already in the stack. This honours the restraint AGENTS.md requires and keeps the bundle disciplined.
  - **§17.5 One entity-agnostic Record Layout.** [`app/shared/record-layout`](../../app/shared/record-layout) is the single scaffold for every record (header, summary, tabs, state-aware content). It is configured by plain typed props, assumes no entity type, owns structure and accessibility (WAI-ARIA Tabs, landmarks, focus, non-colour state), and adapts to its container (not the viewport) so it drops into the DS-03 Drawer later with no change. Its public API is intentionally small and documented.
  - **§17.6 A development-only demonstration.** A dev-only route (`/design/record-layout`, excluded from production builds by a `NODE_ENV` guard in `app/routes.ts`) proves the layout across several configurations. It is not a module, does not touch registry-driven navigation, and ships no domain functionality.
- **Consequences.** *Easy:* every later module builds records and themed UI from one token system and one layout, with contrast and completeness verified by tests; theme stays flash-free and stateless; no dependency or schema change. *Hard:* the two dark colour blocks (explicit vs system) are duplicated in CSS (media queries cannot read custom properties) and are kept in parity by a test; breakpoints live in both CSS and TS and are sync-tested. *Accepted:* DS-01 and DS-02 shipped in one PR as an owner-approved exception, each with independent acceptance evidence.
- **Alternatives considered.**
  - *A UI/component framework or CSS-in-JS (MUI, Tailwind, vanilla-extract, styled-components).* Rejected: AGENTS.md forbids a heavy framework without explicit justification; plain CSS custom properties + container queries meet every requirement with no dependency, no build-time styling engine and no bundle cost.
  - *A raw palette exposed to components (`--dh-blue-500`).* Rejected: defeats theming and semantics; components must speak in roles, so only semantic tokens are exposed.
  - *A single dark block via CSS nesting/preprocessing to avoid duplication.* Rejected: pure CSS cannot share a declaration block across an attribute selector and a media-query selector without a build step; a parity test makes the duplication safe instead.
  - *Shipping DS-01 and DS-02 as separate PRs.* Rejected for this case only, on the owner's approval: DS-02 is DS-01's first consumer and the shell integration would otherwise be done twice; the combination is recorded here and in the roadmap.
  - *Viewport media queries for the layout's responsiveness.* Rejected: a record must adapt to its container (main region now, Drawer later); container queries make it correct in both without a variant.

## ADR-018: The Shared Drawer — URL-Driven, History-Stacked, Focus-Isolated
- **Status:** Accepted. Delivers [DS-03](../roadmap/ROADMAP_V2.md#-ds-03--shared-drawer), building on [ADR-016](#adr-016-cloudflare-access-identity-app-shell-and-registry-driven-routing) (React Router framework mode, registry-driven routing, the server-rendered `data-theme` mechanism, the Worker auth boundary) and [ADR-017](#adr-017-design-tokens-and-the-shared-record-layout) (DS-01 tokens incl. z-index/elevation/motion; the container-adaptive DS-02 Record Layout that "drops into the DS-03 Drawer with no change").
- **Context.** DalyHub's UX rule is *never lose the user's place* ([AGENTS.md §6](../../AGENTS.md#6-ux-philosophy)): opening a record must not throw away where you were, and Back must always work. The Drawer is the product's default way to open any record ([DESIGN_SYSTEM.md → Drawer](../design/DESIGN_SYSTEM.md#drawer)). It must be **entity-agnostic** (no Project/Task/People logic, no D1, no repositories), must **host the DS-02 Record Layout** rather than invent a second record presentation, must be **deep-linkable and stackable**, and must meet the WAI-ARIA modal-dialog contract and WCAG 2.2 AA. Nothing in ADR-016 specified how modal/overlay state is encoded in the URL or integrated with browser history — that is the new ground here.
- **Decision.**
  - **§18.1 The open stack lives in the URL, not in component state.** The stack is a repeated `drawer` search parameter whose ordered values are the open records, backmost first (e.g. `/projects?status=active&drawer=project%3Aalpha&drawer=goal%3Anorth-star`). The rendered stack is a **pure function of the URL**, so refresh, a copied link and browser Back/Forward all restore the same state. We deliberately do **not** depend on ephemeral React Router location state, which disappears on refresh or a shared link. Keys are opaque, URL-safe tokens; the `<kind>:<id>` shape is a *consumer* convention (used by the fixture), never parsed by the Drawer. Every URL transform preserves unrelated query parameters.
  - **§18.2 Every drawer change is a real navigation, and close is provenance-aware.** Opening pushes one history entry (`preventScrollReset`) and **tags that entry** with a session-scoped token in `history.state`, recorded in a provider-owned set; re-opening the current top is a no-op (no duplicate levels, no stray metadata). Closing uses browser **Back (`navigate(-1)`) only when the current top level's token is one this provider instance issued** — i.e. it genuinely pushed that level, so the previous entry is precisely the pre-drawer state of the *same* page and Forward can restore it. Otherwise (a directly deep-linked, copied-link or refreshed level — the set is empty after a remount even though the browser preserved the `history.state` token) it removes **only the top drawer parameter in place** (a `replace`), preserving the pathname, hash and unrelated query parameters. A bare `history.state.idx > 0` check is explicitly insufficient: it only proves an earlier entry exists, not that it belongs to this page, so it could send a deep-linked user Back to a different route. Navigations are issued as an explicit `{ pathname, search, hash }` target because `setSearchParams` navigates to `"?" + params` and drops the hash. Escape, the close button and browser Back all resolve through this one path.
  - **§18.3 Content is resolved from the key, so deep links render coherently.** The single mounted `DrawerProvider` takes a caller `renderDrawer(entry) → { title, description, children, size?, preventClose?, initialFocusRef? } | null`. It is called for every level on every render, including a fresh SSR deep-link load with no background-location state; returning `null` yields a built-in, accessible not-found panel so a stale or hand-typed link fails gracefully instead of blank.
  - **§18.4 Inline render, not a portal.** The stack renders inline as a viewport-`fixed` sibling of the page (z-index from the DS-01 scale), **not** through a `createPortal` to `document.body`. This server-renders for direct deep links and degrades coherently without JavaScript, and there is no portal node to leak. Background isolation is achieved by walking from the stack node to `<body>` and marking every sibling on that ancestor path `inert` — neutralising the underlying page *and* the app shell regardless of where the provider is mounted; lower stack levels are `inert` too, so only the top drawer is interactive and only it carries `aria-modal`. Each panel's React identity is keyed by **`depth:key`** (not depth alone): depth keeps a lower level mounted when a higher one is pushed above it, while the key forces a **remount when the record at a depth is replaced**, so record-local state and mount-only initial focus never leak from a replaced record into its replacement (a duplicate record key at a different depth stays uniquely keyed).
  - **§18.5 Deterministic focus & keyboard.** On open, focus moves into the drawer (explicit `initialFocusRef`, else the close button, else the first control); Tab/Shift+Tab are trapped and wrap; Escape closes only the top level unless `preventClose` is set; on close focus returns to the opener when it still exists, else a safe fallback. Background scrolling is locked with `overflow: hidden` on the root while open, and the underlying scroll *position* is preserved by keying `ScrollRestoration` to the pathname (§18.6) plus a same-frame reassertion, so a drawer never moves the page.
  - **§18.6 Path-keyed scroll restoration.** `ScrollRestoration` is given `getKey = (location) => location.pathname`. A drawer (and a DS-02 tab) is a query-only, same-path transition, so it shares one scroll key and never triggers a scroll reset, while genuine page-to-page navigation still saves and restores scroll per route. This is the one shell-level change DS-03 makes to FND-09, and it is behaviour-preserving for cross-path navigation.
  - **§18.7 Small, entity-agnostic public API.** [`app/shared/drawer`](../../app/shared/drawer) exports `DrawerProvider`, `useDrawer` (typed controller: `openDrawer`/`replaceDrawer`/`closeDrawer`/`closeAll` + stack introspection), `DrawerTrigger`, `DrawerClose`, the URL-contract helpers and the types. Internal panel/stack/focus-trap/scroll-lock/inert machinery is **not** exported — callers never manage focus traps, portals, history entries or z-index. A generous stack-depth cap replaces further pushes with a top-replace to bound pathological loops without limiting normal use.
  - **§18.8 A development-only demonstration.** A dev-only route (`/design/drawer`, excluded from production by the same `NODE_ENV` guard in `app/routes.ts` as DS-02's fixture) proves the full journey — background state/scroll preservation, a Project drawer hosting the real Record Layout with deep-linkable tabs, a stacked Goal/Person drawer, Back/Forward/Escape, direct deep links (incl. unknown), long-content scrolling, mobile sheet and themes. It is not a module, touches no registry navigation and ships no domain functionality.
- **Consequences.** *Easy:* any future module opens any record by pushing a key — no per-entity drawer, no bespoke record scaffold, and deep links/Back/Forward/refresh work for free; the Record Layout is reused unchanged. *Hard:* modelling overlays as history entries means honouring browser Back is mandatory, and unsaved-state guarding (`preventClose`) can gate the in-app affordances but not hardware Back — full form-guard flows belong to DS-06. *Accepted:* one shell-level `getKey` change to `ScrollRestoration`; the `<kind>:<id>` deep-link shape is a development fixture convention, not a committed product record route (real routes arrive when a module adopts the Drawer). **No new dependency.**
- **Alternatives considered.**
  - *Imperative open with JSX children (`open(<Drawer/>)`), state-only.* Rejected: content held in component state cannot survive refresh or a copied link, failing the deep-link requirement.
  - *Nested/child routes per drawer, composed through the module registry.* Rejected for DS-03: it would prematurely establish real product record routes and alter the registry to serve a fixture, which the scope forbids; a query-parameter stack is registry-neutral and requires no `app/routes.ts` change beyond the dev fixture.
  - *A third-party drawer/modal library (Radix, Headless UI, vaul, react-modal).* Rejected: a small native implementation over the browser platform (`inert`, `dialog` semantics, container queries, CSS transitions) meets every requirement with **zero new dependency, licence or bundle cost**, consistent with [ADR-017 §17.4](#adr-017-design-tokens-and-the-shared-record-layout).
  - *A `createPortal` to `document.body`.* Rejected: portals do not server-render, so a direct deep link would paint blank until hydration and break the no-JS case; inline render with ancestor-sibling `inert` isolates the background just as well with nothing to leak.
  - *Default (per-entry) scroll restoration.* Rejected: it resets the underlying page's scroll when a drawer closes on a history POP; path-keyed restoration keeps drawer transitions on one scroll key so the page never moves.

## ADR-019: Shared Card Identity & Reorder, and the Filter Expression & URL Contract
- **Status:** Accepted. Delivers [DS-04](../roadmap/ROADMAP_V2.md#-ds-04--shared-cards) and [DS-07](../roadmap/ROADMAP_V2.md#-ds-07--shared-filters) together as an owner-approved, dependency-aligned exception to one-item-per-PR (DS-07 depends on DS-04 for a real shared collection to filter, and both are prerequisites of TODAY-01). Builds on [ADR-017](#adr-017-design-tokens-and-the-shared-record-layout) (DS-01 tokens; the container-adaptive DS-02 Record Layout) and [ADR-018](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) (the DS-03 Drawer and its repeated-`drawer`-parameter URL contract, which the filter URL contract must compose with).
- **Context.** Every future collection surface (Today, Projects, Goals, Areas, Search) renders entities as cards and narrows them with filters. [AGENTS.md §9.8](../../AGENTS.md#98-shared-over-bespoke) and [DESIGN_SYSTEM.md → Cards/Filters](../design/DESIGN_SYSTEM.md#cards) mandate **one** shared Card and **one** shared Filter system — a bespoke per-module card or filter bar is Product Debt on merge. Two durable contracts had to be settled now, before any module depends on them: (a) how a Card is identified and reordered accessibly without owning Drawer/business state, and (b) how a filter expression is modelled and encoded in the URL safely and forward-compatibly. Nothing prior specified either.
- **Decision.**
  - **§19.1 One entity-agnostic Card, configured by data.** [`app/shared/card`](../../app/shared/card) exposes a single `Card` (plus `CardCollection`/`ReorderableCardCollection`) driven by a small typed `CardProps`. There is no `TaskCard`/`ProjectCard`/…; entity identity is just a `typeLabel`/`icon`/`accent`/`tone` a consumer passes. The Card imports no D1, repositories, workspaces, hierarchy rules, routes or bindings.
  - **§19.2 Card identity & the primary open action.** A card carries a **stable `id`** (used for selection state, reorder keying and collection membership). The **title** is the sole primary open target — a real link (`href`) and/or button (`onOpen`); with both, an unmodified click opens via `onOpen` while a modified/middle click follows the link. The card root is a non-interactive `article`, so there is no `div onClick` and no nested-interactive violation. The Card opens the [DS-03 Drawer](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) but **does not own Drawer state or parse drawer keys** — it emits an href/callback and the consumer maps it to a drawer key. Because filter state and drawer state occupy **disjoint** URL parameters (`f`/`fmode`/`fv` vs `drawer`), opening a card from a filtered collection preserves the filters for free.
  - **§19.3 Accessible reorder emits intent, never mutation — list only for now.** `ReorderableCardCollection` implements drag (Pointer Events) **and** a keyboard equivalent (pick up / arrow-move / drop / Escape-cancel) over the browser platform with **no drag-and-drop dependency**. It announces position/movement via a live region, keeps non-reorderable cards pinned, and **emits `onReorder(nextIds, detail)`** — the consumer owns any persistence; no hidden database write occurs. The **permutation guarantee** is enforced by snapshotting the committed collection (id order + pinned set) at pickup and cancelling the drag cleanly if *anything* changes before drop — an item added, removed, reordered externally, or flipped between reorderable and pinned — so a stale, non-permutation order can never be emitted. Pointer targeting is one-dimensional (vertical), correct for a single-column **list**; the public presentation is therefore `list` only, and a genuine two-dimensional grid reorder is deferred to a later item (grid/board use the plain `CardCollection`, which offers no drag). A small native implementation meets every requirement at zero dependency/licence/bundle cost, consistent with [ADR-017 §17.4](#adr-017-design-tokens-and-the-shared-record-layout) and [ADR-018](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated).
  - **§19.4 One filter model, pure and React-free, with type-aware validation.** [`app/shared/filters`](../../app/shared/filters) defines a typed, entity-agnostic model — `FilterFieldDefinition`, `FilterExpression = { mode, clauses }`, value types (`text`/`boolean`/`enum`/`number`/`date`/`reference`/`multi-enum`) each paired with only type-appropriate operators — plus a **pure, deterministic evaluator** that handles missing/null values, compares dates by UTC calendar day, never mutates source data, and drops invalid clauses rather than throwing. Validation enforces the field's declared value **type**, not just operator arity, so a malformed clause restored from an untrusted URL (a boolean for a text field, `"banana"` for a numeric `gt`, a bad or impossible date, a non-finite number, an invalid range, a number for an enum) is rejected; enum/reference **retain** unknown option values for forward compatibility (only the type is enforced); an `operators` override may only **narrow** a type's default set (a widening override throws in development, clamps in production). The pure model imports **no React** and is re-exported from a dedicated [`model`](../../app/shared/filters/model.ts) entry (an import-guard test enforces it), so a server-backed module can translate an expression into its own query layer without resolving React; UI-only value-control rendering lives in a separate `FilterValueControls` seam. The expression is deliberately bounded and non-recursive — not a general query language.
  - **§19.5 The filter URL contract: repeated, versioned, safely encoded.** Filters live in the URL, never only in component state. The encoding is a **repeated, versioned query representation** — `fv` (format version), one `f` per clause (`field:operator` or `field:operator:<small-per-clause-json-value>`), and `fmode=or` only when needed — chosen over one opaque JSON blob so it is diff-able, forward-compatible and deterministic. It **preserves every unrelated parameter, including DS-03's repeated `drawer` parameters**, removes all residue when empty, produces deterministic URLs for equivalent states, and round-trips punctuation/spaces/Unicode/reserved characters. Decoding is total and defensive — **no `eval`, no `Function`, no unsafe deserialisation**, only bounded `JSON.parse` in `try/catch` — dropping malformed/unknown/oversized/excess clauses and ignoring unknown versions wholesale, so a hostile or hand-typed URL is safe to restore.
  - **§19.6 Storage-agnostic saved views; no persistence here.** The saved-view contract (`SavedView`, `SavedViewAdapter`) is pure data plus callbacks; DS-07 **persists nothing** (no D1, no migration). A consumer (a fixture in memory today; item X-02 in D1 later) supplies views and handles intent. The adapter exposes exactly the interactions the Filter Bar drives — select, save-as, update, delete — and advertises **no** rename callback, because the bar has no rename interaction yet; saved-view management (rename, reorder, share) arrives with X-02. A view referencing an obsolete field degrades gracefully (the clause sanitises away).
  - **§19.7 Development fixture only.** One dev-only route (`/design/cards-filters`, excluded from production by the same `NODE_ENV` guard in `app/routes.ts` as the DS-02/DS-03 fixtures) proves both systems end to end with in-memory fixture records of several entity types. It is not a module, establishes no product route or business rule, and ships no domain functionality; the shared Card/Filter **infrastructure** ships in the production bundle as reusable code, but the fixture route, data and styles do not.
- **Consequences.** *Easy:* any future module renders entities by configuring one Card and narrows them by registering typed fields — deep links, Back/Forward, refresh and Drawer-open-over-filters all work for free, and a card reorder or filter translates to that module's own persistence/query layer without importing React. *Hard:* the Card's "both href and onOpen" open semantics and the reorder's intent-only contract push persistence to consumers (correct, but they must wire it); the filter URL is bounded, so pathological inputs are truncated rather than honoured. *Accepted:* DS-04 and DS-07 shipped in one PR (owner-approved); saved views and reorder persistence are deferred to the modules/items that need them. **No new dependency.**
- **Alternatives considered.**
  - *Per-entity cards / per-module filter bars.* Rejected: the explicit anti-goal ([AGENTS.md §9.8](../../AGENTS.md#98-shared-over-bespoke)); one configured component keeps the product coherent and is a top [PRODUCT_DEBT](../product/PRODUCT_DEBT.md) target.
  - *A drag-and-drop library (dnd-kit, react-beautiful-dnd, SortableJS).* Rejected: a small Pointer-Events + keyboard implementation meets the accessibility contract (keyboard equivalent, live-region announcements, pinned items, permutation guarantee) at **zero** dependency/licence/bundle/Workers-compat cost.
  - *One opaque JSON blob in a single query parameter.* Rejected: not diff-able, not forward-compatible, harder to bound/sanitise, and worse for deterministic equality; a repeated, versioned per-clause encoding is preferable (and is what the [DS-03 drawer contract](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) established as the house style).
  - *Filter state in React/component or ephemeral location state.* Rejected: it would not survive refresh or a copied link and would break Back/Forward — the same reasoning as ADR-018 §18.1.
  - *A recursive/nested query builder.* Rejected as premature and unsafe to restore from an untrusted URL; a single-level AND/OR expression covers the collection-filtering need without an unbounded grammar.

## ADR-020: The Application Frame — Sidebar Shell, Pane, Collection Layout and Entity Identity

- **Status.** Accepted (2026-07-19). Implements [PX-02](../roadmap/ROADMAP_V2.md#-px-02--product-frame) and the shell-alignment recommendations of [`PRODUCT_EXPERIENCE.md`](../design/PRODUCT_EXPERIENCE.md) (#1–#5, #9, #11, #14, #15). Composes DS-01…DS-04/DS-07 and refines the FND-09 shell ([ADR-016](#adr-016-cloudflare-access-identity-app-shell-and-registry-driven-routing)); it does not change their contracts.
- **Context.** After FND-09, DS-01…04 and DS-07 shipped, the *component* layer was premium-grade but the *assembled* product read as "a well-made website hosting excellent components" ([`PRODUCT_EXPERIENCE.md`](../design/PRODUCT_EXPERIENCE.md) Part I): a wrapping top-bar nav that cannot scale to eleven modules, settings chrome (theme, raw email) permanently in the header, a centred document column instead of a workspace pane, no icon/entity-identity system, and no named scaffold for the product's commonest screen (a filtered collection). Every one of these is cheapest to fix **before TODAY-01** pours the first product surface into the frame. DS-01 even defined `--dh-shell-nav-width` that nothing consumed. This ADR settles the durable frame every future module inherits.
- **Decision.**
  - **§20.1 A sidebar + pane application frame.** `AppShell` ([`app/shared/shell`](../../app/shared/shell)) is a two-column grid `grid-template-columns: var(--dh-shell-nav-width) 1fr`: a **persistent left sidebar** (workspace brand, Search + Command Palette entries, registry-driven primary navigation, a spacer, the user menu) and a **full-height content pane**. The frame composes exactly once and consumes only plain data; modules render *inside* the pane and never build their own shell, provider or z-index layer (PRODUCT_EXPERIENCE Part IV §2). The FND-09 acceptance properties (registry-driven nav, skip link, `aria-current`, keyboard-completeness, the theme mechanism) carry over unchanged — this is a re-arrangement of the same semantic parts, not a rebuild.
  - **§20.2 Document-scroll with a sticky sidebar (not an internal pane scroll).** The **document** remains the scroll container and the sidebar is `position: sticky`, rather than giving the pane its own `overflow` scroll. This deliberately preserves the [DS-03 Drawer](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) contract — its body-scroll lock and `ScrollRestoration` act on the window/document — while still letting the Pane Header and FilterBar pin via `position: sticky` to the viewport. Width limits live in content types (prose/record widths on their own containers), never on the pane (PRODUCT_EXPERIENCE #2).
  - **§20.3 The Pane Header belongs to the screen.** A `PaneHeader` (title · subtitle/count · view-switcher slot · one primary-action slot) is owned by each surface, carries the page heading, and is **not** a `<header>`/`banner` element so the frame keeps exactly one `banner` landmark — the sidebar brand. It never contains theme controls, an email address or logout.
  - **§20.4 Chrome → User Menu.** Identity (email), the theme control, Settings and Sign out move into an avatar-triggered `UserMenu` pinned at the sidebar bottom (PRODUCT_EXPERIENCE #4). The theme *mechanism* is **reused unchanged** (same accessible `ThemeControl` fieldset, same cookie, same flash-free SSR — [ADR-016 §5.11](#adr-016-cloudflare-access-identity-app-shell-and-registry-driven-routing)); only the control's placement moves. The menu is an accessible disclosure (not a modal): `aria-expanded`/`aria-haspopup`, Escape closes and restores focus, outside-click closes.
  - **§20.5 One Entity Identity system.** [`app/shared/entity`](../../app/shared/entity) maps each entity-type slug (the kernel `entities.type` values) to ONE icon and ONE accent — a frozen `ENTITY_IDENTITY` map plus `EntityIcon`. Accents are CSS custom properties `--dh-entity-<type>-accent` defined in the light map and both dark blocks (contrast-tested ≥3:1 per theme) and used at **identity sites only** (icon, card edge, chip), never as text colour (PRODUCT_EXPERIENCE Part III §5). The nav icon is **derived from each module's own `entityTypes` manifest** (the navigation adapter takes an optional resolver) — the module declares its identity, the shell reads it; there is no central icon switch. The map is entity-agnostic presentation data importing no kernel branded types, D1 or module code.
  - **§20.6 An in-house outline icon set, not a dependency.** [`app/shared/icons`](../../app/shared/icons) ships one outline set (24×24, `currentColor`, 1.75px, round joins) via a small `createIcon` primitive, in the idiom PRODUCT_EXPERIENCE #3 describes (e.g. Lucide). We adopt an **in-house** set rather than adding `lucide-react` to preserve the project's zero-runtime-dependency posture and avoid a proxy/Workers-compat risk; the *set* is swappable, the entity-identity *mapping* is the durable contract. Icons are decorative by default (`aria-hidden`); a text label always accompanies them (AGENTS.md §15).
  - **§20.7 A shared Collection Layout for the commonest screen.** `CollectionLayout` ([`app/shared/collection-layout`](../../app/shared/collection-layout)) is to screens what the Record Layout is to records: an entity-agnostic scaffold with a Pane Header, a FilterBar slot, a content slot, a selection/bulk slot, and built-in Loading (Skeletons), Empty, Filtered-empty and Error states with a defined precedence (error → loading → filtered-empty → empty → children) and a sticky header. It holds **no** business logic, repositories or entity assumptions — every collection surface (Today, Projects, Areas, Goals, Notes, People) becomes configuration, so the product stays consistent at the screen level as well as the component level (PRODUCT_EXPERIENCE #5, Part IV §5).
  - **§20.8 One EmptyState and one Skeleton system.** A single `EmptyState` (icon · title · body · primary/secondary actions · illustration slot) unifies the previously-forked record/filter empty renderings (PRODUCT_EXPERIENCE #14). A `Skeleton` system (collection/card/pane) mirrors the final layout and honours reduced motion (PRODUCT_EXPERIENCE #15). Both are entity-agnostic.
  - **§20.9 Mobile navigation reuses the Drawer machinery.** On `< md` the rail collapses to an animated, scrim-backed, safe-area-aware overlay sheet that **reuses the DS-03 Drawer's focus, background-inertness and body-scroll-lock hooks** — no second focus-trap implementation exists (PRODUCT_EXPERIENCE Part VI, Focus rule). It mounts only while open (so the mount/unmount focus contract applies cleanly), closes on Escape and outside-click, restores focus to the toggle, and never jumps page content.
  - **§20.10 Compose, don't redesign.** The Tokens, Cards, Filters, Drawer and Record Layout contracts are **unchanged**; PX-02 only composes them correctly and adds shell-level and screen-level scaffolds. The one additive kernel-adjacent change is an *optional* entity-type resolver on the navigation adapter (backward compatible).
- **Consequences.** *Easy:* every future module gets a correct home for free — a sidebar row (registry-driven), a `CollectionLayout` pane, entity identity everywhere, and the empty/loading/error slots wired by construction; navigation stops being a per-launch redesign risk; the product's silhouette matches its ambition from the first real feature. *Hard:* the document-scroll + sticky-sidebar choice (rather than an internal pane scroll) is a deliberate constraint that keeps the DS-03 scroll contract intact; a genuine collapsed icon-rail and a mobile bottom-tab bar are explicitly deferred. *Accepted:* the Search/Command entries are frame affordances whose surfaces arrive with DS-08/DS-09; an in-house icon set (documented, swappable) instead of a named library. **No new runtime dependency, no migration, no product module.**
- **Alternatives considered.**
  - *Keep the top-bar nav / defer the frame until TODAY-01.* Rejected: a wrapping top bar cannot scale to eleven modules and the migration cost grows with every module shipped into the wrong frame — cheapest to fix now (PRODUCT_EXPERIENCE #1).
  - *Give the pane its own `overflow` scroll.* Rejected: it breaks the DS-03 Drawer's window/document scroll-lock and `ScrollRestoration`; a sticky sidebar over a document scroll keeps that contract while still pinning headers (§20.2).
  - *Add `lucide-react` (or another icon library).* Reasonable and explicitly named by the contract, but rejected for now to preserve the zero-dependency posture and avoid proxy/Workers-compat risk; the entity-identity mapping (the durable part) is dependency-agnostic and the set can be swapped later without touching consumers (§20.6).
  - *Let the shell map module → icon centrally.* Rejected: it reintroduces a central switch; deriving the icon from each module's own entity-type manifest keeps navigation registry-driven (§20.5, AGENTS.md §9.2).
  - *Build the collection scaffold ad hoc in TODAY-01.* Rejected: PROJ-01/AREA-01/NOTES-03 would each rebuild it slightly differently and the product would fragment at the screen level — the same argument that produced DS-02 for records (PRODUCT_EXPERIENCE #5).

## ADR-021: The Shared Timeline & Activity Feed — one renderer, one presentation view-model, in-house virtualisation

- **Status.** Accepted (2026-07-19). Delivers [DS-05](../roadmap/ROADMAP_V2.md#-ds-05--shared-timeline--activity-feed). Consumes the [FND-05](../roadmap/ROADMAP_V2.md#-fnd-05--shared-activity-model) Activity model ([ADR-012](#adr-012-activity-persistence-and-atomic-mutation-recording)) as its only event source, and reuses DS-01 tokens ([ADR-017](#adr-017-design-tokens-and-the-shared-record-layout)), the DS-03 Drawer ([ADR-018](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated)), the DS-07 filter model & URL contract ([ADR-019](#adr-019-shared-card-identity--reorder-and-the-filter-expression--url-contract)) and the PX-02 Entity Identity / EmptyState / Skeleton ([ADR-020](#adr-020-the-application-frame-sidebar-shell-pane-collection-layout-and-entity-identity)). It adds no persistence, no migration and no runtime dependency.
- **Context.** [DESIGN_SYSTEM.md → Timeline/Activity Feed](../design/DESIGN_SYSTEM.md#timeline) mandate that a record-scoped **Timeline** and a workspace/scope **Activity Feed** are the *same component at different scopes*, never forked, and that both render the one shared Activity model — grouped by day, filterable, virtualised. FND-05 already supplies the model and the read repository (`listForEntity` for a Timeline, `listForWorkspace` for a Feed, both newest-first, cursor-paginated, N+1-free). What was unsettled: how kernel `ActivityRecord`s become renderable UI without leaking D1/repository/cursor internals or weakening branded types; how an unknown or newly-registered event type renders safely; and how to virtualise thousands of variable-height events without a data-grid dependency.
- **Decision.**
  - **§21.1 One shared renderer, two configurations.** [`app/shared/activity-feed`](../../app/shared/activity-feed) exposes ONE `ActivityStream`; `Timeline` (`scope="timeline"`) and `ActivityFeed` (`scope="feed"`) are thin presets that differ only in the `loadPage` they are given and their accessible label. There is no separate Timeline/Feed item or list — grouping, the event item, filtering, pagination and virtualisation are shared by construction.
  - **§21.2 One typed presentation view-model boundary.** `toActivityItem(record, options)` is the single seam mapping a kernel `ActivityRecord` → a renderable `ActivityItem`. It **preserves** the branded `ActivityType`, the open validated-string actor/subject fields, the UTC `occurredAt` and the validated `payload` unchanged (no `any`, no down-branding), attaches resolved entity identity per subject, and selects a primary subject deterministically. The model is **React-free** and re-exported from [`~/shared/activity-feed/model`](../../app/shared/activity-feed/model.ts) (import-guard tested), so a server surface can map/group/order/page/filter without resolving React.
  - **§21.3 The component API hides the store.** The shared component takes an opaque `loadPage(cursor) → { items, nextCursor, hasMore }` loader, not a repository, D1 binding, cursor internals or workspace control. The route owns the `resolveWorkspaceScope(env).activity` call and record→item mapping behind that loader; the trusted workspace boundary is fixed server-side and is never selected or overridden by client input through the component. Entity resolution is supplied **batched** by the route, so the UI introduces no N+1 lookups.
  - **§21.4 Descriptors, not a product switch. Conservative unknown-type fallback.** A per-type `ActivityTypeDescriptor` is the only place a specific event gets a specialised rendering; the seven kernel-reserved lifecycle types ship default descriptors, and modules register their own. There is **no** large switch over Tasks/Projects/Goals/Areas/People/Notes/Diary. An unregistered or newly-invented type falls back to a generic rendering that stays readable, shows the humanised event type, the actor, the time and available subjects, **never crashes on an unfamiliar payload, and never dumps raw unbounded JSON** — the payload summariser shows only a bounded set of primitive top-level fields and skips nested objects/arrays entirely.
  - **§21.5 Deterministic ordering, day grouping, one date seam.** The stream is totally ordered newest-first by `(occurredAt, id)` with `id`-descending tie-breaking (matching the kernel's `ORDER BY occurred_at DESC, id DESC`), grouped by **UTC calendar day**, and flattened to heading+item rows so one windowed list keeps day headings associated with their events. All day/time formatting flows through ONE `ActivityDateFormatter` that formats **manually against UTC getters** with fixed month/weekday tables — not `Intl` — so server and client render byte-identical text (hydration-safe); relative "Today"/"Yesterday" are opt-in via a caller-threaded reference instant. Timestamps render as semantic `<time datetime>`.
  - **§21.6 In-house windowed virtualisation, no dependency.** Long streams are virtualised by a small, pure `computeWindow` core (unit-tested) plus a measurement hook inside a **bounded scroll region**: only rows near the viewport are rendered, positioned by measured offsets with top/bottom spacers that keep total scroll height stable, so variable-height content does not overlap or jump and loading more never resets position (new items append below). We reject a general-purpose data-grid dependency ([AGENTS.md §16](../../AGENTS.md#16-performance-expectations), the DS-04 reorder precedent): a small native implementation meets the requirement at zero dependency/licence/bundle/Workers-compat cost. The event item is memoised so one item changing does not rerender the list; mapping/grouping/filtering are memoised so they do not rerun unnecessarily.
  - **§21.7 Filtering reuses DS-07; opening entities reuses DS-03.** DS-05 builds **DS-07** `FilterFieldDefinition`s over the `ActivityItem` view-model (event type, actor type, referenced entity type, date) and hands them to the shared `FilterBar`; the DS-07 evaluator matches and the DS-07 URL contract carries filter state, preserving unrelated params including DS-03's `drawer` params. It adds no product-specific operator and no timeline-only filter UI. A referenced entity opens through the **DS-03** Drawer (`DrawerTrigger` by default) — no bespoke modal, no second drawer — and an unresolved/deleted/inaccessible entity degrades to calm non-link text, disclosing nothing.
  - **§21.8 States reuse the shared components.** Initial-loading (Skeleton), genuinely-empty (EmptyState), filtered-empty (DS-07 `FilterEmptyState` with clear-filters recovery), loading-more, page-load-failure with retry, end-of-feed and unknown-type/unresolved-subject are all handled, reusing existing shared components and the `role="alert"` error convention. Newly-loaded events are announced via a polite live region. The feed uses `role="feed"` with `aria-posinset`/`aria-setsize` and accessible day-group headings (real `h2`/`h3`/`h4` kept in the accessibility tree, not `aria-hidden`).
  - **§21.9 Development fixture only.** One dev-only route (`/design/activity-feed`, excluded from production by the same `NODE_ENV` guard in `app/routes.ts` as the DS-02/03/04/07 fixtures) proves both configurations end to end over in-memory fixture records shaped exactly like the kernel model. It is not a module, ships no product route or business rule, and introduces no fake production data; the shared infrastructure ships in the production bundle, the fixture does not.
- **Consequences.** *Easy:* any future record page drops a `Timeline` into its DS-02 Activity tab, and any workspace/scope surface drops an `ActivityFeed` in, by supplying one `loadPage` and (optionally) descriptors + a batch entity resolver — grouping, filtering, virtualisation, drawer-opening, states and accessibility all come for free, and a new module renders its event types by registering descriptors, never editing DS-05. *Hard:* client-side filtering applies over loaded pages (the type filter can additionally be pushed to the repository via the loader when a surface needs server-side narrowing); virtualisation owns a bounded scroll region rather than the document scroll. *Accepted:* the demonstration is a fixture; real product Timelines/Feeds arrive when a module adopts DS-05 (e.g. PROJ-04). **No new dependency, no migration, no persistence.**
- **Alternatives considered.**
  - *Separate Timeline and Feed implementations.* Rejected: the explicit anti-goal (DESIGN_SYSTEM Timeline/Activity Feed rules); two scopes of one renderer keep event rendering identical everywhere.
  - *A virtualisation library (react-window / react-virtuoso / TanStack Virtual).* Rejected for now: a small pure window calculation plus a measurement hook meets the variable-height, day-heading-association and scroll-stability requirements at zero dependency cost, consistent with the DS-04 no-DnD-library precedent. The windowing core is isolated, so a library could replace it later without touching the public API.
  - *Passing the `ActivityRepository` (or D1) into the component.* Rejected: it would leak the store, cursor internals and workspace control into shared UI; an opaque `loadPage` keeps the component entity- and storage-agnostic and lets the route own trusted scope.
  - *A product switch statement over entity/event types.* Rejected: it centralises what modules should own; per-type descriptors with a safe generic fallback keep DS-05 entity-agnostic and crash-proof on unknown types.
  - *`Intl.DateTimeFormat` for headings/times.* Rejected: its output can differ between the Workers runtime and a browser (ICU/locale-data variance) and defaults to the machine timezone, risking hydration mismatch; manual UTC formatting is deterministic and identical on both.

## ADR-022: Shared Forms & field controls — declared save model, validation boundary, and the entity-link picker

- **Status.** Accepted (2026-07-19). Delivers [DS-06](../roadmap/ROADMAP_V2.md#-ds-06--shared-forms--field-controls). Builds on DS-01 tokens ([ADR-017](#adr-017-design-tokens-and-the-shared-record-layout)), the FND-04 EntityLink kernel ([ADR-002](#adr-002-entitylinks)/[ADR-011](#adr-011-entitylink-persistence-and-lifecycle)) and the FND-08 Markdown pipeline ([ADR-006](#adr-006-markdown-strategy)/[ADR-015](#adr-015-markdown-source-and-safe-rendering-pipeline)). It adds no runtime dependency, no persistence and no migration. Resolves the design-system half of [DEBT-03](../product/PRODUCT_DEBT.md) and advances [DEBT-08](../product/PRODUCT_DEBT.md).
- **Context.** Every create/edit surface DalyHub will build needs the same things done consistently: one control per field type, forgiving inline validation, a save behaviour the user can *predict*, and a way to relate records. Historically these diverged — different forms saved differently (autosave vs. explicit vs. modal), eroding trust ([DEBT-03](../product/PRODUCT_DEBT.md)), and cross-entity links were modelled per feature ([DEBT-08](../product/PRODUCT_DEBT.md)). DS-06 must supply ONE entity-agnostic system with the discipline of the earlier DS items (small typed API, React-free model boundary, dev-only fixture, WCAG 2.2 AA, zero new dependency, no product coupling), and it must make the EntityLink relationship real through the existing kernel rather than inventing a second store.
- **Decision.**
  - **§22.1 One entity-agnostic forms system.** [`app/shared/forms`](../../app/shared/forms) knows nothing of specific entities, D1/SQL, workspace selection, routes, product modules or a central entity-type switch. Consumers pass typed values, field definitions, validation and persistence callbacks; server loaders/actions keep the trusted workspace scope and data access. There is no `TaskForm`/`NoteForm` — one system, composed.
  - **§22.2 A React-free model boundary.** The validation combinators, dirty comparison, tags rules, deterministic date model, the explicit-save reducer, the autosave coordinator and the entity-link option filtering are framework-free and re-exported from [`~/shared/forms/model`](../../app/shared/forms/model.ts). A static import-guard test forbids React in those files (mirroring the DS-05/DS-07 model boundaries), so a server module can share the vocabulary without resolving React.
  - **§22.3 Declared, visible save model (the DEBT-03 resolution).** The save mode is part of the component contract, never inferred. `useForm` is explicit Save/Cancel: dirty tracking, duplicate-submit prevention, validation on blur and submit, first-invalid focus, server-authoritative errors, complete-draft preservation on failure, baseline reset on success, and unsaved-navigation interception. `useAutosaveField` is field-by-field with a documented trigger (a restrained debounce and/or a valid blur) and calm `Unsaved`/`Saving`/`Saved`/`Couldn’t save` states. The autosave coordinator is a pure state machine guaranteeing one save in flight, coalesce-to-latest, stale-response rejection, input preservation on failure, explicit retry, and no save while invalid — so the user can always predict when a value is committed.
  - **§22.4 Validation boundary.** Client validation is convenience and UX; the SERVER is authoritative. Field/form server errors are shown even when client validation passed, no save is attempted while a value is invalid, stale async validation responses are ignored, and no raw exception, database error or opaque code is ever surfaced — only specific, recovery-oriented messages. The error summary is an assertive live region that links/focuses each offending field.
  - **§22.5 Native-first accessible controls, no UI framework.** One implementation per field type over native elements (a real checkbox/switch, native date/datetime-local, native `<select>` for the link-type choice), reaching for ARIA only where native semantics are insufficient — the editable **combobox/listbox** (`useCombobox`), which the select control and the entity-link picker share. Consistent with [ADR-017 §17.4](#adr-017-design-tokens-and-the-shared-record-layout) and [ADR-019](#adr-019-shared-card-identity--reorder-and-the-filter-expression--url-contract): a small native implementation meets every requirement at zero dependency/licence/bundle cost. No second design-token system, no visual component framework.
  - **§22.6 Markdown control over the one sink.** The Markdown control edits FND-08 SOURCE (never HTML), preserves it byte-for-byte, and previews only through the shared `renderMarkdownSource` → `MarkdownContent` pipeline. It introduces no second parser and no new HTML sink; the heavy `unified` renderer is lazy-loaded only when the preview opens. The single-sink guard test still passes.
  - **§22.7 Deterministic, timezone-safe dates.** A date-only value is the literal ISO `YYYY-MM-DD` string, validated and compared as integers and never routed through `Date`, so it cannot shift by a zone. A datetime value is an ISO-8601 UTC instant; the control edits the UTC wall-clock explicitly (the one documented rule). A zone-less wall-clock time is deliberately NOT a field type because it cannot be serialised unambiguously.
  - **§22.8 Entity-link picker integration via FND-04, with a SERVER-authoritative policy.** The picker UI is entity-agnostic and callback-driven; it never imports D1, bindings or repositories. Its client configuration (allowed target types, link-type descriptors, direction, single/multiple) is **presentation only** — a calm reflection of the rules that keeps the user from attempting a no-op. The AUTHORITY is a server-supplied `EntityLinkPickerPolicy` (anchor id, `allowedDirections`, `linkTypes` each with `allowedTargetTypes`, `multiple`) that a loader/action constructs from trusted context and passes to `createLinkWithPolicy` in the entity-agnostic server service ([`app/platform/entity-links`](../../app/platform/entity-links)). That function treats the submitted `targetId`/`linkType`/`direction` as untrusted and validates every one against the policy — direction well-formed and allowed; link type a valid, permitted kernel slug; not a self-link; anchor and target exist and are accessible in the bound workspace; the target's entity type allowed for the link type; the single-selection limit — before delegating to `EntityLinkRepository.create`, whose own guarantees (workspace scope, endpoint existence, reserved-type refusal, idempotent create/restore, duplicate uniqueness) are the last word. It returns a **typed, safe outcome** (`{ok:true,…} | {ok:false, reason, message}`) — a raw repository/SQL error never escapes. Direction is honoured exactly (`outgoing` → anchor is the source; `incoming` reverses the endpoints). Removal is equally authorised (`unlinkWithPolicy`): a client hands back only a link id, which is untrusted, so the service verifies the link exists in the workspace, is anchored to `policy.anchorId`, and its type/direction are permitted before deleting — a crafted id for another anchor, an un-offered type or another workspace is refused. The `multiple:false` limit is **concurrency-safe**: a pre-check + create is a TOCTOU race the kernel's identity-uniqueness does not cover ("one link total" ≠ "no identical duplicate"), so after creating, the service reconciles — it keeps the deterministically-earliest active policy link and rolls back its own if it lost, so concurrent creators converge on exactly one. Links are created/removed through the EXISTING repository only — no second relationship table, no spine-parentage bypass, no migration. The `searchTargets` contract lets DS-08 provide real search later without replacing the picker.
  - **§22.9 Navigation safety, and a real modal confirm.** A dirty explicit form arms `useUnsavedChangesPrompt`, which intercepts BOTH a full-page unload (native prompt) and in-app navigation (React Router `useBlocker`). Because the DS-03 Drawer stack lives entirely in the URL's repeated `drawer` search parameter and a drawer close/replace is a same-pathname, search-param-only navigation, a pathname-only guard would miss it: a drawer-hosted form passes its `drawerKey`, and the guard then blocks any navigation whose next drawer stack no longer contains that key (close, Escape, Back, `closeDrawer`, `replaceDrawer`, param removal, or replacing the top with another record) while allowing harmless changes that keep the form mounted (a deeper drawer pushed on top, an unrelated filter param). The confirm is a real modal (WAI-ARIA `alertdialog`): background made `inert`, Tab/Shift+Tab trapped, initial focus on the safe action, Escape chooses Stay, focus restored to the initiating control on Stay, generated ids so instances don't collide.
  - **§22.10 Submission snapshot & Markdown lazy-load.** `useForm.handleSubmit` captures ONE immutable snapshot of the values at submit time and uses it for all sync/async submit validation, the `onSubmit` callback and the success baseline — never re-reading a newer draft — so an edit made mid-submission stays dirty and cannot be silently discarded; a reset or unmount cleanly abandons an in-flight submission via a generation guard. The Markdown control lazy-imports the `unified` renderer only when its preview opens (keeping the parser out of the initial production bundle); the `unified` stack is declared in Vite `optimizeDeps.include` so that first import doesn't trigger a dev-server re-optimisation reload.
  - **§22.11 Development-only demonstration.** `/design/forms`, excluded from production by the same `NODE_ENV` route guard as the other DS fixtures, demonstrates every requirement with in-memory fixtures. Fixtures never become production records or production navigation; the real FND-04 integration and the server policy are proven by a Workers/D1 test against the same server service.
- **Consequences.** *Easy:* a product module builds a create/edit screen by composing shared controls + `useForm`/`useAutosaveField` and passing validators + a persistence callback; relating records is one `EntityLinkPicker` wired to a loader/action that constructs the trusted `EntityLinkPickerPolicy` and calls `createLinkWithPolicy`. Save behaviour is predictable and identical everywhere, and a crafted client request cannot create a link the form did not offer. *Hard/accepted:* the pure autosave coordinator is deliberately explicit (an effect-returning reducer the hook executes) rather than magical — more code, but every stale/overlap/failure case is testable; the picker's target search is a bounded title scan until DS-08 supplies real search (disclosed, and swappable behind the same contract); the picker demonstrates with an in-memory store in the dev fixture, with the real kernel path and the server policy proven separately in the Workers/D1 test; the drawer-aware guard couples the forms package to DS-03's exported `drawer` URL contract (accepted — drawer-hosted forms are the intended consumer). **No new dependency, no migration, no persistence, no product module.**
- **Alternatives considered.**
  - *A form/validation library (React Hook Form, Formik, Zod-driven resolver).* Rejected: it would dictate API shape and add a runtime dependency for problems a small typed core solves, and none gives the declared autosave-vs-explicit save model DS-06 needs; the framework-free model + two hooks are cheaper to own and test.
  - *A headless combobox dependency (Radix/Downshift/React-Aria).* Rejected for now: consistent with the zero-dependency precedent, a focused `useCombobox` over native input+listbox meets the interaction and accessibility bar; the door stays open to adopt one later behind the same control API if requirements outgrow it.
  - *Inferring the save mode from context.* Rejected: it is exactly the DEBT-03 unpredictability; the mode is an explicit, documented part of the contract.
  - *A new link/relationship table or a picker that talks to D1.* Rejected: it would duplicate the FND-04 kernel and bypass its workspace/lifecycle invariants; the picker stays UI-only and links flow through the existing repository via an injected server service.
  - *Trusting the picker's client configuration as the link boundary.* Rejected: client config is spoofable, so it is presentation only; the server-supplied `EntityLinkPickerPolicy` enforced by `createLinkWithPolicy` is the authority, returning typed, safe outcomes rather than leaking repository errors.
  - *A pathname-only unsaved-changes guard.* Rejected: DS-03 stores its Drawer stack in the URL's `drawer` search params, so closing a drawer that hosts a dirty form is a same-pathname navigation a pathname-only guard would miss; the guard is drawer-key aware, and the confirm is a real focus-trapped, inert-background modal.
  - *Serialising dates through `Date`/`Intl` round-trips.* Rejected: it silently shifts calendar dates by timezone; integer-based date-only values and an explicit-UTC datetime avoid the whole class of bug.

## ADR-023: Shared Search — registry-driven providers, runtime orchestration, and safe navigation
- **Status.** Accepted (2026-07-19). Delivers [DS-08](../roadmap/ROADMAP_V2.md#-ds-08--shared-search). Builds on [ADR-013](#adr-013-module-registry-contract-and-discovery) (the module registry and its `listSearchProviders()` seam, `ModuleRuntimeContext` and the static/runtime split), [ADR-010](#adr-010-server-side-workspace-context) (server-derived workspace scope), [ADR-018](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) (the DS-03 Drawer, its repeated-`drawer` URL contract and the focus/inert/scroll-lock hooks) and [ADR-020](#adr-020-the-application-frame--sidebar-shell-pane-collection-layout-and-entity-identity) (the PX-02 frame, the Search affordance, entity identity, and the mobile-nav hook reuse).
- **Context.** FND-06 defined the search-provider contract but deliberately built no search surface, orchestrator or UI; DS-08 is the first real consumer. The contract had to be validated against a real surface, the trusted server boundary composed, and the result made to open in the DS-03 Drawer — all without leaking product specifics into a shared, entity-agnostic system (AGENTS.md §9.1/§9.8) and without a second modal, drawer or focus-trap.
- **Decision.**
  - **§23.1 One entity-agnostic Shared Search, split into model / runtime / UI.** [`app/shared/search`](../../app/shared/search) contains no task/project/note logic, no central entity-type or provider switch, no D1, no workspace selection, no direct binding access, no product-repository imports and no Drawer-key parsing. Pure model, runtime orchestration and React UI are separated so the model and orchestrator carry no React and the UI carries no server code.
  - **§23.2 A React-free model boundary.** [`~/shared/search/model`](../../app/shared/search/model.ts) (import-guard tested like DS-05/DS-07) exposes pure functions for query normalisation and bounds, result validation, navigation-target validation, deterministic fuzzy ranking, first-seen grouping, provider-result merging, deduplication, stable result identity, safe match ranges, the per-provider/total limits, partial-failure representation and keyboard-selection maths. It imports only type-level kernel contracts, so a server or a search provider reuses it without resolving any UI.
  - **§23.3 The FND-06 result contract is refined — once — for safe navigation.** FND-06's `SearchResultItem.navigateTo` was an **opaque string**, which would force Search to parse an arbitrary path to guess whether it is a Drawer key or a route — exactly the central-switch coupling ADR-013 forbids. It is replaced by a validated `SearchResultTarget` discriminated union — `{ kind: "drawer"; drawerKey; canonicalPath? } | { kind: "route"; to }` — so a **module owns how its result opens** and Search dispatches on `kind` without understanding product routes or ids. Targets are validated at the boundary: in-app paths must be app-relative, and `javascript:`, protocol-relative `//…`, external URLs, backslashes and control characters are rejected. The refinement stays typed, storage-independent, React-free, React-Router-free, D1-free, binding-free, module-owned, workspace-scoped at runtime, deterministic and safely validated.
  - **§23.4 A trusted runtime orchestration boundary, with bounded provider deadlines and cancellation.** The orchestrator obtains providers only through `ModuleRegistry.listSearchProviders()` (never a manual array), receives a trusted, server-derived `ModuleRuntimeContext`, never accepts a client-supplied workspace id, normalises and bounds the query, bounds the provider and per-provider counts, validates every result, deduplicates, ranks, enforces the total limit, and groups. Each provider runs under a **bounded per-provider deadline** with a per-provider `AbortSignal` (also linked to an optional caller/client signal exposed on the `SearchRuntimeContext`), so one broken *or hung* provider can never crash *or stall* global search — a provider that ignores its signal is simply stopped being waited on at the deadline, and a late resolution/rejection of the abandoned work is consumed and ignored (no unhandled work, no leak). A failing or timed-out provider becomes `ok: false` with **no stack trace, SQL, repository error, binding, timeout or raw message** leaking; a partial run returns a calm partial state, a total failure a typed retryable error, and the raw query is not logged.
  - **§23.5 Production workspace scope via the authenticated composition boundary.** The `/search` route resolves the active workspace through the SAME FND-03/FND-09 path the kernel tests cover — `resolveAuthenticatedWorkspaceScope(env, session)` — which derives the scope from trusted server configuration (`env.DEFAULT_WORKSPACE_ID`) and **verifies the workspace exists in D1** (ADR-010, ADR-016 §5.6). There is no second resolver and no `workspaceContextFromId` shortcut on the route, so the deployed path is exactly the tested path. Authentication is authoritative: a missing session is a **401**, never a Search result; a missing/invalid `DEFAULT_WORKSPACE_ID`, a nonexistent configured workspace, or a D1 lookup failure all **fail closed** to the calm, retryable Search failure (no internal detail leaks, and no provider runs). The client can neither supply nor influence the workspace id. (The dev/e2e local D1 is migrated and the configured workspace seeded by `e2e/setup-local-db.mjs` so the real resolver succeeds there too.)
  - **§23.6 Deterministic ranking and grouping.** Ranking is tiered (exact title → title prefix → token prefix → fuzzy title → subtitle) with a normalised provider score only as a tie-breaker, then title then id — so a provider's raw score range can never dominate global ranking, and ordering is total and deterministic. Results group primarily by entity type; a result with no entity type falls back to its owning module. Groups appear in first-seen order over the ranked list, so the most relevant group leads without a hard-coded entity order. A result's **global identity is `moduleId::providerId::itemId`** — the provider-local `itemId` is unique only within its provider (ADR-013), so the provider is part of the identity; two providers in the *same* module may return the same `itemId` and both survive (deduplication is by the full identity ONLY — never by title, target or entity type, which would wrongly conflate distinct records).
  - **§23.7 Incremental controller with immediate, authoritative stale invalidation.** The browser controller has one clear lifecycle: a meaningful query change **immediately** advances an authoritative generation and aborts the in-flight request — *before* the debounce fires, not when it fires — then debounces a fresh request under the reserved generation; only that generation may update state. So a slower earlier request can never display results for a query the input has already moved past (the abort is best-effort cleanup; the generation guard is authoritative — no arbitrary timeouts). Empty input returns to idle without executing providers, loading keeps valid prior results visible as stale content, a partial failure still shows healthy results, clearing and retry both invalidate any pending request, aborted requests never surface an error, and nothing updates state after unmount. A meaningful query change **also clears the active selection immediately**: while a new query loads, the prior results may stay visible but are NOT current — they carry no active option, no `aria-activedescendant`, and are rendered as inert text (not links), so neither Enter nor a plain/modified click can open a result from the previous query. Navigation re-arms only once the new results resolve, with no default selection (Arrow keys begin selection explicitly).
  - **§23.7a Reused kernel validation and a client response decoder.** Entity types are validated with the authoritative FND-02 `validateEntityType` (the `ENTITY_TYPE_PATTERN` + length contract) — never a Search-owned grammar — so valid kernel types like `meeting.follow_up` are never silently dropped and the definitions cannot drift; a malformed type degrades to a dropped field, never a thrown error. The browser treats the `/search` JSON as **untrusted** and rebuilds a bounded outcome through `decodeSearchOutcome`, reusing the same pure validators (`validateTarget`, `validateEntityType`, the shared limits): malformed JSON or a structurally-invalid outcome becomes a generic, retryable failure rather than a crash, and individual bad results/groups/ranges degrade away.
  - **§23.8 The Drawer opens results; Search reuses the modal machinery.** A result's `drawer` target navigates to the route that hosts that module's `DrawerProvider` (its `canonicalPath`, or the current path) with the Drawer key appended via the DS-03 pure URL helper, **preserving unrelated query parameters**; result rows are real links so modified/middle-click open in a new tab. The surface reuses `useDrawerFocus`/`useBodyScrollLock`/`useInertBackground` — there is **no second focus-trap, scroll-lock or inertness system** (ADR-020 §20.9). `useInertBackground` is anchored to the **modal ROOT** (which contains both the scrim and the panel), not the inner panel, so the background app becomes inert while the scrim stays interactive (a scrim click closes Search); the input keeps focus in a WAI-ARIA combobox controlling a `listbox`, with grouped `option`s, `aria-activedescendant`, a polite status region, PX-02 entity icons/accents, `<mark>` highlighting from plain text segments (no `dangerouslySetInnerHTML`), and full keyboard operation. It mounts once at the `AppShell` boundary (available from any route without navigating), is **lazy-loaded** so the full UI stays out of the initial bundle, is opened by the PX-02 sidebar affordance and the `/` shortcut (a repeated `/` while open is a no-op, and focus stays in the input), and **does not claim the `⌘K` command-palette shortcut**. Search sits ABOVE the Drawer in the z-order: opening Search over an already-open Drawer renders it on top (Drawer inert behind), and selecting a result closes Search *before* its Drawer opens, so the two surfaces never fight.
  - **§23.9 Search is not the Command Palette (DS-09).** DS-08 executes no module commands, creates no records, registers no `⌘K`, and mixes no commands with results. It exposes a clean reusable API (the surface, controller, model and orchestrator) that DS-09 can later launch or incorporate.
  - **§23.10 Open-source reuse.** For fuzzy matching, match ranges and accessible combobox behaviour, Fuse.js, fzf/fzy scoring, fuzzysort, match-sorter, cmdk's `command-score` and headless combobox libraries (Downshift/React-Aria) were reviewed (see [`REFERENCE_PRODUCTS.md`](../reference/REFERENCE_PRODUCTS.md)). For a single-user app over short titles and a handful of providers, a small in-house subsequence matcher and a native combobox are smaller, dependency-free and Workers-safe — consistent with the ADR-018/019/020/022 zero-runtime-dependency posture. **Decision: build; add no dependency.** Only the well-known subsequence-scoring *idea* was reused; no third-party code was copied, so no attribution is required.
- **Consequences.** *Easy:* any future module contributes searchable records by registering a provider that returns validated results with a Drawer/route target — grouping, ranking, highlighting, keyboard, Drawer opening and failure handling all work for free; DS-09 gets a reusable Search API. *Hard:* the result contract is now a typed target (a module must describe how its result opens), and the fixture provider's scope is config-derived until a repository-backed provider arrives. *Accepted:* the one FND-06 refinement (`navigateTo` → `SearchResultTarget`) and a config-derived (not existence-checked) workspace scope for the fixture phase. **No new runtime dependency, no migration, no new relationship model.**
- **Alternatives considered.**
  - *Keep the opaque `navigateTo` string.* Rejected: it forces Search to parse product paths/ids — the central-switch coupling ADR-013 forbids; a typed target keeps the mapping module-owned and the surface entity-agnostic.
  - *Browser-side search over a shipped workspace dataset.* Rejected: it would send the workspace to the client and break the workspace boundary; the browser sends only the bounded query and receives only bounded, display-ready results.
  - *A search dependency (Fuse.js/fzf) or a headless combobox (Downshift/React-Aria).* Rejected for now (§23.10): a small in-house matcher and native combobox meet the bar at zero dependency/bundle/Workers-compat cost; the door stays open behind the same API.
  - *Deep-linking the Search surface via its own URL param.* Rejected: Search is a transient launcher, not a record; the Drawer (which it opens) carries the deep-linkable, shareable state. A React-state modal reusing the DS-03 hooks is simpler and never competes with the `drawer` param.
  - *Letting a provider score drive global ranking directly.* Rejected: a provider's arbitrary score range would dominate; provider relevance is normalised and used only as a tie-breaker after Search's own tiered ranking.

## ADR-024: Command Palette & Quick Actions — command kinds, trusted catalogue, authenticated execution, and one shared action

- **Status.** Accepted (2026-07-19). Delivers [DS-09](../roadmap/ROADMAP_V2.md#-ds-09--command-palette--quick-actions). Builds on [ADR-013](#adr-013-module-registry-contract-and-discovery) (the module registry, its `listCommands()`/`getCommand()` seam, `ModuleRuntimeContext`, and the static-metadata/runtime-execution split), [ADR-023](#adr-023-shared-search--registry-driven-providers-runtime-orchestration-and-safe-navigation) (the reusable DS-08 Search API and the `SearchResultTarget` contract — §23.9 explicitly reserved this relationship and left `⌘K` unclaimed), [ADR-020](#adr-020-the-application-frame--sidebar-shell-pane-collection-layout-and-entity-identity) (the PX-02 frame and its Command Palette affordance, and §20.9's single focus/inert/scroll-lock hook set) and [ADR-016](#adr-016-cloudflare-access-identity-app-shell-and-registry-driven-routing) (the authenticated request boundary and workspace composition).
- **Context.** FND-06 defined a command contract before any palette or execution boundary consumed it: a single `CommandContribution` shape with a `run` handler receiving only a workspace context, no typed navigation target, no safe execution outcome, no catalogue transport, no authenticated execution route, no cancellation/deadline, and no separation between navigation and server execution. DS-09 is the first real consumer, so the contract had to be validated against a real surface (the keyboard shell `⌘K`) and quick actions on Cards/Headers — without shipping executable handler functions to the browser, without a central module switch or command-ID string-parsing, and without a second Search, Drawer, Card or focus-trap.
- **Decision.**
  - **§24.1 One entity-agnostic shared action + command model.** [`app/shared/commands`](../../app/shared/commands) contains no Task/Project/Today rule, no fixture id, no D1, no workspace selection, no route-specific switch and no Drawer-key parsing. It is split into a React-FREE model, a React runtime (provider, hooks, surface, adapters, transports) and the server boundary, so the model and the server carry no React and the UI carries no server handler.
  - **§24.2 A React-free model boundary.** [`~/shared/commands/model`](../../app/shared/commands/model.ts) (import-guard tested like DS-05/DS-07/DS-08) exposes pure functions for catalogue validation, query-driven ranking, deterministic grouping, the merge with DS-08 results, the shortcut model (normalise/display/match/collision/reserved), the presentation context, the execution state machine and the bounds. It REUSES DS-08's React-free model — query normalisation, fuzzy matching and keyboard-selection maths — rather than shipping a second matcher.
  - **§24.3 The FND-06 command contract is refined into a discriminated union.** A command is EITHER a `navigate` command (a trusted declarative target, reusing the validated `SearchResultTarget`) OR an `execute` command (a server handler). A command that is both, or neither, is a hard registry-construction error. The kernel validates the `kind`, validates a navigation target with the shared validator, and refuses a command that reassigns a reserved global shortcut (`Mod+K`, `/`). The target validator was relocated from `app/shared/search/target.ts` into the kernel ([`navigation-target.ts`](../../app/kernel/modules/navigation-target.ts)), colocated with the `SearchResultTarget` type; DS-08 re-exports it, so both DS-08 and DS-09 share ONE implementation (ADR-023 §23.3 introduced this validation — a narrowly-required cross-reference).
  - **§24.4 Static metadata versus runtime execution.** `CommandHandler` now receives a `CommandRuntimeContext` (workspace scope + an `AbortSignal`, mirroring `SearchRuntimeContext`) and returns a typed `CommandExecutionOutcome` — never `void`, never a raw error. The registry stores the handler but never invokes it; a navigation command's target is serialisable; an executable command's handler is dropped at the catalogue boundary. The browser can never receive an executable server handler.
  - **§24.5 Command kinds and the typed safe outcome.** `CommandExecutionOutcome` is `{ ok: true; message?; target? }` or `{ ok: false; reason: "unavailable"|"conflict"|"failed"; message }` — bounded, display-ready, no SQL/stack/infra codes. Optional post-success navigation reuses the validated shared target. A timeout is reported honestly as `failed` (the side effect may have completed) — never "cancelled".
  - **§24.6 Registered commands versus contextual actions, and the context model.** Registered commands are immutable, registry-discovered and global/module-relevant. Contextual actions are transient, supplied by the current surface/Drawer/selection through a scoped, React-owned provider ([`CommandContextProvider`](../../app/shared/commands/CommandContextProvider.tsx)) mounted once at the AppShell boundary — never by mutating the registry, never through a module singleton or a global event bus. Registrations are removed on unmount (so no stale action survives a route/Drawer change), deterministically ordered, and de-duplicated by id (first wins). The palette merges contextual actions, registered commands and DS-08 record results into one list without confusing them. The presentation context carries only safe UI facts (pathname, module, Drawer-open + top Drawer key as OPAQUE data, entity type, selection count) — never a client workspace id, session claims, a D1 handle or a product blob; it affects relevance/ordering only and NEVER authorises server execution.
  - **§24.7 A trusted command-catalogue boundary.** The browser receives serialisable metadata, not handlers. [`buildCommandCatalogue`](../../app/platform/commands/command-catalogue.ts) discovers commands ONLY through `ModuleRegistry.listCommands()` (no manual module-command list), retains ownership, bounds and orders metadata deterministically, omits handlers, and validates navigation targets. It is served from a small authenticated resource route `GET /commands` (the smallest coherent transport, mirroring `/search`) — never accepting a client workspace id — so the palette UI code stays out of the initial bundle.
  - **§24.7a A browser-side decoder.** The catalogue JSON is untrusted: [`decodeCommandCatalogue`](../../app/shared/commands/catalogue.ts) rebuilds a bounded catalogue from validated pieces (never a cast), dropping malformed entries; a structurally-unusable response yields a calm palette error state.
  - **§24.8 An authenticated command-execution boundary.** `POST /commands/:commandId` requires the session (401), resolves the trusted workspace through `resolveAuthenticatedWorkspaceScope` (never client-supplied), discovers the registry, looks up the exact command, rejects an unknown command (404) and a navigation-only command (400), runs an executable handler ONCE under a bounded deadline + cancellation signal, returns a typed safe outcome, applies `no-store`, and never exposes the handler or a raw exception. The submitted id is the only client input that matters; module id, title, target and "permission" flags from catalogue metadata grant no authority. There is no automatic retry — a retry is a deliberate new invocation. The core is extracted into [`runRegisteredCommand`](../../app/platform/commands/run-command.ts) so its decision logic is unit-testable with a fabricated registry.
  - **§24.9 Deadline, cancellation and stale protection.** [`executeCommand`](../../app/shared/commands/execute-command.ts) bounds each run with a deadline and an `AbortSignal` (linked to the request), never rejects, cleans up timers/listeners, and turns a thrown handler into a calm failure. The client execution state machine advances a monotonic token so a stale response cannot settle a newer activation, and blocks a second activation while one is pending — so a command is never double-invoked from one intent.
  - **§24.10 DS-08 Search reuse.** The palette composes DS-08's `useSearchController` and reuses its query normalisation, endpoint, decoder, stale-request controller, target validation, Drawer/route navigation helper and highlighting — there is no second global-search implementation. Command results appear immediately while record search loads; a stale Search response cannot overwrite newer input; a partial or total Search failure leaves commands fully usable; selecting a record result opens through the existing DS-03 Drawer/route.
  - **§24.11 Contextual ranking.** Ranking is deterministic: contextual eligibility (group order), then exact title, title prefix, token prefix, keyword, fuzzy title, subtitle, then a context boost and stable tie-breaks. No arbitrary module score can dominate; record results keep DS-08's ranking. With an empty query the palette shows a restrained, bounded set of suggested + contextual commands (recent commands are remembered in memory for the session only — never persisted; no analytics or profiling). The recents store lives on `CommandContextProvider` (mounted once at AppShell), **not** the palette controller, so it survives the palette unmounting on close and the recent/suggested ordering is visible across openings.
  - **§24.12 Shortcut ownership and collisions.** ONE shared dispatcher ([`useCommandShortcuts`](../../app/shared/commands/useCommandShortcuts.ts)) owns all shortcuts — never a listener per command. It normalises `mod` per platform, ignores ordinary shortcuts while typing (permitting the reserved `Mod+K`), ignores auto-repeat, resolves collisions by precedence so one event triggers one action, and `preventDefault`s only when a binding claims the event. The reserved vocabulary (`Mod+K` palette, `/` Search, `g`+letter go-to, movement, selection, open, `e` primary action, `Escape`, `?`, `[`) is documented; the kernel refuses a command that reassigns a reserved global shortcut.
  - **§24.13 AppShell modal ownership.** AppShell owns Search and palette open state; the two are MUTUALLY EXCLUSIVE (opening one closes the other); `Mod+K` toggles the palette (a repeat closes it — the documented policy); `/` retains DS-08 behaviour; Escape closes the top-most surface; opening from mobile navigation closes the sheet first; focus returns to the trigger; the palette sits at `--dh-z-modal`, above the Drawer, and reuses the DS-03 focus/inert/scroll-lock hooks — no second focus-trap. The palette is lazy-loaded so its chunk stays out of the initial bundle. The ONE shared dispatcher is installed by `CommandShortcutLayer` (mounted once inside `CommandContextProvider`): it owns the reserved bindings AND the shortcuts declared by **navigation** commands — registered `navigate` commands (from the catalogue) and contextual `navigate` actions — so a declared navigation shortcut actually navigates, not just displays a hint (precedence reserved → contextual → registered; a disabled action never fires). **Executable** command / contextual-run shortcuts are deferred: dispatching one with the palette closed needs a pending/success/failure surface outside the palette (the DS-10 feedback surface excluded here). Because they cannot run globally yet, the palette does NOT advertise them — a shortcut hint appears only for a `navigate` command/action (whose shortcut is actually dispatched), never an executable command, so no hint promises a control that does nothing; global dispatch and the hint land together with DS-10. Wiring navigation dispatch loads the catalogue in the always-on shell — a deliberate, documented departure from the fully-lazy posture that pulls only the small catalogue transport and the pure navigation helper, never the palette UI/controller.
  - **§24.14 Quick Action adapters.** ONE `AppAction` identity is projected into a `CardAction`, a `RecordAction`, a palette command and a keyboard action through [adapters](../../app/shared/commands/action.ts) — no new Card/Header component, one execution path, consistent pointer/keyboard behaviour, and always a text-based accessible name. Commands are intentionally curated (two or three on a Card/Header; the long tail lives in the palette). **Disabled is one concept across every surface.** `disabled` ≠ unavailable (unavailable is *omitted* from the registration); a disabled action is shown but never activatable. The flag travels the whole palette pipeline (`AppAction` → `PaletteCommand` → ranked → `PaletteOption` → render), never derived from CSS. Keyboard selection **skips disabled options** (the active index is resolved during render, so `Enter`/`aria-activedescendant` never target one and an all-disabled list has no active option); a disabled palette option renders as a **non-interactive** element with `aria-disabled="true"` and a visible “Unavailable” cue (never colour alone); the controller's `activate()` guard is the authoritative boundary (no navigate/run/pending/recents/close); **Retry** never re-invokes an action that became disabled (it clears the stale banner); and `appActionToShortcutBinding` yields `enabled: false` so the shared dispatcher never fires it — the same “disabled” the Card and Record Header adapters enforce. Registered catalogue commands are never disabled.
  - **§24.15 Today production proof (honest, no persistence).** Today registers two registry-discovered NAVIGATION commands — "Go to Today" and "Focus Quick Capture" (which navigates to `/today?capture=1`; Today focuses the existing textarea and cleans the bounded param without a Back-button trap, without clearing the draft and without claiming a save). The Pane-Header Quick Capture button and the palette command share ONE `AppAction`. Today's fixture-backed Complete/Reopen is adapted through the shared action so the Card action, the keyboard path and (when a task's Drawer is open) the palette contextual action share one execution path — it remains an IN-MEMORY UI demonstration and says so; nothing is persisted.
  - **§24.16 Contextual proof without leaking product meaning.** Today supplies a task-specific contextual action only while that task's Drawer is open, parsing the opaque `task:<id>` key itself; the shared infrastructure never learns what it means.
  - **§24.17 Open-source reuse.** cmdk, Radix/React-Aria/Ariakit/Downshift command & combobox patterns, and Raycast/VS Code/Linear command architectures were reviewed (see [`REFERENCE_PRODUCTS.md`](../reference/REFERENCE_PRODUCTS.md)). **Decision: build; add no dependency.** DS-08 already built the in-house fuzzy matcher, combobox semantics and safe navigation the palette reuses, and no large UI framework fits the tokens-only, kernel-clean constraints. No third-party code was copied, so `THIRD_PARTY_NOTICES` is unchanged.
- **Consequences.** *Easy:* any module lights up the palette by declaring commands in its manifest (from day one — the FND-06 seam already existed); a curated action appears on a Card, a Header, the palette and the keyboard from one definition; the keyboard-first identity is real. *Hard:* commands are now a discriminated union (both/neither is a compile-time and runtime error), and an executable command must pass through the authenticated boundary rather than run in the browser. *Accepted:* the one relocation (the target validator into the kernel, re-exported by DS-08) and the one FND-06 refinement (a single `run` shape → a `navigate`/`execute` union with a typed outcome and a cancellation-bearing context). **No persistence, no migration, no new relationship model, no DS-10 toast/Inspector/Settings.**
- **Alternatives considered.**
  - *Keep FND-06's single `run` shape and route everything through a handler.* Rejected: a declarative navigation needs no server round-trip, no handler and no execution boundary; forcing it through one loses type safety and ships needless handlers.
  - *Ship the catalogue in the app-shell loader.* Rejected in favour of a small resource route: it keeps the initial HTML lean and parallels `/search`; the catalogue is fetched on first palette open.
  - *Adopt cmdk (or a headless combobox).* Rejected (§24.17): DS-08's in-house matcher, combobox and safe navigation already cover the need with zero runtime dependency, and cmdk owns no execution boundary, catalogue or contextual model.
  - *Let contextual actions register into the module registry.* Rejected: the registry is immutable and workspace-agnostic; transient, surface-owned actions belong to a React-owned provider that cleans up on unmount.
  - *Trust catalogue metadata as authority on execution.* Rejected: the client submits only a command id; authority comes from the server-resolved workspace and the registry, never from anything the browser echoes back.

## ADR-025: The global interaction layer — Feedback platform (notifications, undo, background operations) and the shared Inspector

- **Status.** Accepted (2026-07-20). Delivers the Inspector + feedback-states scope of [DS-10](../roadmap/ROADMAP_V2.md#-ds-10--inspector-settings-and-feedback-states) (Settings-layout is split to a follow-up — see Consequences). Builds on [ADR-018](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) (the DS-03 focus/inert/scroll-lock hooks and the URL-driven open model, both reused), [ADR-022](#adr-022-shared-forms--field-controls--declared-save-model-validation-boundary-and-the-entity-link-picker) (the Inspector edits through DS-06 controls with the autosave coordinator), [ADR-020](#adr-020-the-application-frame--sidebar-shell-pane-collection-layout-and-entity-identity) (the AppShell boundary where the feedback provider mounts, and the entity identity), and [ADR-024 §24.13](#adr-024-command-palette--quick-actions--command-kinds-trusted-catalogue-authenticated-execution-and-one-shared-action) (which explicitly reserved "the DS-10 feedback surface" as the prerequisite for global executable-command dispatch).
- **Context.** Every earlier DS item deferred a common need: a place to *tell the user what happened*. DS-09 could not dispatch executable command shortcuts globally because there was no pending/success/failure surface outside the palette; [PRODUCT_EXPERIENCE #8](../design/PRODUCT_EXPERIENCE.md) flagged that TODAY-02's first real mutation would ship silent without a toast/undo layer; the [interaction philosophy](../../AGENTS.md#7-interaction-philosophy) mandates "optimistic and reversible… prefer undo over confirmation"; and the DESIGN_SYSTEM's [Inspector](../design/DESIGN_SYSTEM.md#inspector) promised one depth-editing surface every module reuses instead of bespoke edit drawers. The forces: it must be ONE implementation for the whole app, entity-agnostic, calm (no toast spam, no modal overload), keyboard-complete, WCAG 2.2 AA, mobile-adapted, and reuse the existing modal machinery rather than growing a second focus-trap or overlay system.
- **Decision.**
  - **§25.1 One entity-agnostic feedback platform + one Inspector, split model / React / (no server).** [`app/shared/feedback`](../../app/shared/feedback) and [`app/shared/inspector`](../../app/shared/inspector) contain no Task/Project/Today rule, no D1, no workspace selection and no product-repository import. Each is split into a React-FREE model (import-guard tested like DS-05/06/07/08/09) and a React runtime. DS-10 is a pure client interaction layer: it adds **no** migration, **no** persistence, **no** server route and **no** new dependency (the zero-dependency precedent of ADR-018–024 holds).
  - **§25.2 A React-free feedback model.** [`~/shared/feedback/model`](../../app/shared/feedback/model.ts) owns the notification-queue reducer and the background-operation lifecycle reducer as pure, deterministic, clock-free functions over immutable state — mirroring the DS-06 autosave split (a pure reducer plus a timing hook). The queue enforces **intelligent stacking** (a `dedupeKey` coalesces a repeat onto the existing record — count bumps, fields refresh, timer restarts, it moves to front — instead of stacking; the antidote to toast spam), a **bounded stack** (the oldest *auto-dismissing* entry is retired first, so a sticky error is never dropped by a burst), and newest-first order. The operation reducer is the state machine pending → running → success | failure with retry (a new attempt) and removal (cancel/dismiss/auto-clear).
  - **§25.3 One provider, mounted once at the AppShell boundary.** [`FeedbackProvider`](../../app/shared/feedback/FeedbackProvider.tsx) is the single implementation for the entire application. It owns the id/clock seams (kept out of the pure model), the dismissal timers with **pause-on-hover/focus** (a toast is never yanked away mid-read), the real `AbortController`s for cancellation and the retry launchers, the Undo commit/reverse mechanism, and the ARIA live regions. There is exactly one; no module owns a notification implementation.
  - **§25.4 The hidden Feedback API.** Modules touch ONLY [`useFeedback()`](../../app/shared/feedback/feedback-context.ts): `notifySuccess`/`notifyInfo`/`notifyWarning`/`notifyError`, `notifyUndo`, `runOperation`, `dismiss`. The implementation is completely hidden — a module cannot render a toast, hold a queue or build an operation tray. `runOperation({ label, run, cancellable?, retryable?, successMessage?, errorMessage? })` is the ONE execution model for long-running work (AI, imports, exports, sync, future integrations): it returns the work's result (or rejects), surfacing the unified lifecycle regardless of the caller.
  - **§25.5 Undo is a platform capability, not per-module logic.** `notifyUndo` raises a success notification carrying a time-boxed Undo. Choosing Undo runs the reverse handler and opts out of commit; letting the window elapse OR dismissing the toast early runs the commit handler (dismissing an optimistically-applied action commits it — the Gmail model). Any reversible action (delete, archive, complete, move, close, dismiss) gets Undo by calling one method; there is no delete-specific or archive-specific undo code. **Two guarantees make this correct:** (a) Undo notifications are **non-coalescing** — each carries its own reverse/commit handlers, so they never merge (merging would drop one action's handlers and desynchronise the handler id from the queue id); (b) commit finalisation is **centralised in one reconciliation effect** that fires `onExpire` exactly once whenever an undo record leaves the queue by ANY path — timer expiry, manual dismiss, dismiss-all, coalescing replacement, bounded-stack eviction, or provider unmount — so an evicted or bulk-cleared undo still commits and no handler leaks.
  - **§25.6 Accessibility of feedback.** Tone is carried by icon + text, never colour alone. Two always-mounted, visually-hidden live regions announce feedback — polite for success/info, assertive for warning/error — kept SEPARATE from the visible toasts so screen readers announce once, not on every re-render. Critically they use **bare `aria-live`, not `role="status"`/`role="alert"`**: an implicit status/alert role on an app-global region would make every other loading/error region ambiguous to `getByRole` and to assistive tech; `aria-live` announces just as reliably without claiming the role. Actions and dismiss are real, keyboard-operable buttons with text names and ≥44px targets; motion is CSS and disabled under `prefers-reduced-motion`; the centre is anchored so it never covers primary UI (bottom-right desktop, bottom full-width safe-area on mobile).
  - **§25.7 The Inspector reuses the Drawer machinery — no second focus-trap.** [`InspectorProvider`](../../app/shared/inspector/InspectorProvider.tsx) is mounted per surface (like the DS-03 Drawer), and its open state lives ENTIRELY in the URL (`?inspector=<key>`, a single value — not the Drawer's stack), so an editing surface is deep-linkable, refresh-proof and Back/Forward-correct. A surface supplies `renderInspector(entry)` mapping the opaque key to a presentation — the exact `renderDrawer` contract — so no module builds its own edit drawer. The panel REUSES the DS-03 `useDrawerFocus`/`useBodyScrollLock`/`useInertBackground` hooks (ADR-020 §20.9): the focus hook does focus-in + restore in both presentations, and its Tab trap, the scroll lock and the inert walk engage ONLY in the compact (mobile) sheet.
  - **§25.8 One panel, two presentations.** Desktop: a **non-modal, resizable** right-side `complementary` landmark; the surrounding content reflows via the layout's padding so the panel never covers it and the page stays interactive (bulk edit / multi-select possible). Resizing is an accessible `separator` with pointer AND keyboard (Left/Right/Home/End) control, clamped and persisted. Mobile (below `md`): a **modal sheet** — focus-trapped, background inert, body-scroll-locked. The mode is resolved after mount, so SSR renders the docked form and the modal behaviour only ever engages on the client.
  - **§25.9 The Inspector edits through DS-06, saving optimistically.** The Inspector is for DEPTH (all editable fields via the shared DS-06 controls, field-by-field autosave through the ADR-022 coordinator); the DS-03 Summary/Drawer is for essentials. Field controls are shared, never duplicated between them.
  - **§25.10 The DS-09 hook is now unblocked.** DS-10 provides the pending/success/failure surface ADR-024 §24.13 named as the prerequisite for global executable-command dispatch. Wiring `runOperation` into the shared shortcut dispatcher (so an executable command's shortcut dispatches globally with a hint) is a small, now-unblocked follow-up kept out of this change to preserve "one item, one coherent change"; DS-10 delivers the surface it depends on.
  - **§25.11 Open-source reuse.** Radix UI Toast, React-Aria/React-Spectrum (toast + landmark), Ariakit, Sonner and `react-hot-toast` were reviewed for the notification layer, and Radix Dialog / React-Aria for the Inspector (see [`REFERENCE_PRODUCTS.md`](../reference/REFERENCE_PRODUCTS.md)). **Decision: build; add no dependency.** DS-03 already owns the accessible modal machinery the Inspector reuses, and the calm queue/undo/operation semantics (coalescing, sticky errors, commit-on-dismiss undo, one execution model) are DalyHub policy a general toast library does not encode; a toast dependency would add a second overlay/z-index/portal system for behaviour a small tokens-only implementation covers. No third-party code was copied, so `THIRD_PARTY_NOTICES` is unchanged.
- **Consequences.** *Easy:* any module gets calm, consistent, announced feedback and platform Undo by calling `useFeedback()`; any record becomes depth-editable by mounting `InspectorProvider` with a `renderInspector` callback; long-running work has one lifecycle; DS-09 executable-shortcut dispatch is unblocked. *Hard:* feedback is now a singleton at the AppShell boundary (a module must not build its own); the Inspector is URL-driven (its open state is a navigation, not ephemeral state). *Accepted & disclosed:* **Settings-layout — the third strand of the roadmap's DS-10 title — is deliberately split to a follow-up** so this PR stays one coherent change (the global interaction/feedback layer + Inspector); the ROADMAP_V2 DS-10 entry records the split. The `pending` operation status is modelled and available for a future queue/scheduler; `runOperation` starts an operation at `pending` and transitions to `running` immediately, so today `pending` is momentary.
- **Alternatives considered.**
  - *Adopt a toast library (Sonner / react-hot-toast / Radix Toast).* Rejected (§25.11): it would add a second overlay/portal/z-index system and a dependency, and none encodes DalyHub's calm queue policy or the commit-on-dismiss Undo/one-execution-model semantics.
  - *Render the Inspector as a Drawer level.* Rejected: the Drawer is a modal navigation stack; the Inspector is a non-modal, resizable, selection-reflecting properties panel on desktop. It reuses the Drawer's hooks but is a distinct, single-valued, non-modal surface — while still becoming a modal sheet (and reusing the full machinery) on mobile.
  - *Keep undo per-module.* Rejected: undo is a cross-cutting platform capability; per-module undo is duplication and drift. One `notifyUndo` with a reverse + commit contract covers every reversible action.
  - *Make feedback a per-route provider like the Drawer.* Rejected: notifications/operations must survive route changes and be callable from anywhere, so the provider is global at AppShell; the Inspector, which is surface-specific (each surface knows how to render its records), stays per-route like the Drawer.

## ADR-026: Shared Settings layout — composition primitives, declared change behaviour, and the dangerous-action contract

- **Status.** Accepted (2026-07-20). Delivers [DS-10b](../roadmap/ROADMAP_V2.md#-ds-10b--settings-layout), the Settings-layout strand split out of [DS-10](../roadmap/ROADMAP_V2.md#-ds-10--inspector-settings-and-feedback-states) by [ADR-025](#adr-025-the-global-interaction-layer--feedback-platform-notifications-undo-background-operations-and-the-shared-inspector). Builds on [ADR-022](#adr-022-shared-forms--field-controls--declared-save-model-validation-boundary-and-the-entity-link-picker) (the DS-06 save models and controls it reuses), [ADR-025](#adr-025-the-global-interaction-layer--feedback-platform-notifications-undo-background-operations-and-the-shared-inspector) (the DS-10 Feedback platform for confirmation, and the Inspector this is the last section of), [ADR-018](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) (the DS-03 focus/inert/scroll-lock hooks the confirmation dialog reuses) and [ADR-017](#adr-017-design-tokens-and-the-shared-record-layout) (tokens only).
- **Context.** Every scope in DalyHub needs settings — application, workspace, module, a single record, and the final **Settings** tab of a record Inspector/Drawer — and the DESIGN_SYSTEM's [Settings](../design/DESIGN_SYSTEM.md#settings) pattern mandates ONE layout with "same setting type → same control" and "no bespoke settings screens", with dangerous settings visually separated and confirmed. The forces: it must be one implementation across every scope; entity-agnostic (no Project/Task/Person rule); reuse the DS-06 forms/validation/save models and the DS-10 Feedback platform rather than growing a second engine; provide a calm, WCAG 2.2 AA, responsive-to-320px layout; and give a reusable, accessible dangerous-action pattern without inventing a second focus-trap/overlay.
- **Decision.**
  - **§26.1 One entity-agnostic Settings system, presentation + interaction only.** [`app/shared/settings`](../../app/shared/settings) contains no product rule (no deletion/archival logic), no D1, no workspace selection, no route, no persistence and no central switch. A consumer supplies typed values, async apply/confirm callbacks and copy. Like DS-05/06/07/10 it splits a React-FREE model from the React UI, import-guard tested (`test/unit/settings/react-free.test.ts`). It adds **no** migration, persistence, server route or dependency (the zero-dependency precedent of ADR-018–025 holds).
  - **§26.2 Small composable primitives, not a schema-driven engine.** `SettingsLayout` (a `region` + optional heading, container-query responsive so it is correct in a full route and a 320px Drawer alike), `SettingsGroup` (a labelled section; `tone="danger"` is the differentiated dangerous region) and `SettingsRow` (label · description · control · optional status, side-by-side or stacked) are strongly-typed React primitives. There is **no** central settings schema/registry: the existing FND-06 module registry already exposes a `settings` capability seam, and a declarative schema engine would be premature and rigid. A row accepts any accessible control and names a bare control via a render-prop supplying `aria-labelledby`/`aria-describedby` ids — so DS-06 self-labelled fields and bare native controls both work with no double-labelling.
  - **§26.3 Change behaviour is DECLARED by reuse, not reinvented.** Explicit-save settings use DS-06 `useForm` (dirty/validate/Save/Cancel/duplicate-submit/first-invalid-focus — pristine · dirty · validating · saving · saved · save-failed · retry · reset/revert); quiet autosave uses DS-06 `useAutosaveField` + `SaveStatusIndicator`. Immediate settings use `useImmediateSetting`, a thin coordinator over a NEW pure model (`immediate.ts`) whose ONE semantic difference from the DS-06 autosave coordinator is **revert-on-failure** (a toggle that fails to save flips back, whereas an editable field keeps its draft); it composes the DS-10 `useFeedback` platform for success/error (no new toast system). The immediate model guarantees optimistic value, single-flight with coalesce-to-latest, stale-response rejection and duplicate-apply safety.
  - **§26.4 The dangerous-action contract reuses the DS-03 modal machinery.** `ConfirmationDialog` is a WAI-ARIA modal that REUSES `useDrawerFocus`/`useBodyScrollLock`/`useInertBackground` (as the DS-10 Inspector's sheet does — no second focus-trap or overlay). Its pure model (`confirmation.ts`) owns the rules: an optional **typed confirmation** (an exact, case- and whitespace-significant phrase) gates Confirm, and a single-flight phase machine (idle → pending → idle | error) prevents duplicate submissions and re-enables retry on failure. Initial focus is the typed input or the safe Cancel button (never the destructive one); a failure surfaces an inline `role="alert"` keeping the dialog open for retry; success raises a shared Feedback toast; Cancel/Escape/scrim cancel. `DangerousAction` composes it with a `SettingsRow` and the destructive button. Focus is restored to the trigger on close by a **deterministic post-close safety net** on the persistent wrapper (mirroring DrawerProvider/InspectorProvider), because a browser can reset focus to `<body>` when the focused node is removed.
  - **§26.5 Accessibility & responsive built in, not deferred to DS-11.** Semantic headings/grouping, associated labels/descriptions, keyboard-complete operation, announced status/errors (the dialog `alert`; a row status may be a polite `aria-live`, never `role="status"`), no colour-only meaning (the danger region carries icon/shape + text + border), ≥44px targets, focus placement/restoration for confirmations, reduced motion, safe-area spacing, and no 320px overflow — all met by DS-10b itself. DS-11 will audit/automate the baseline; DS-10b is accessible now.
  - **§26.6 Open-source reuse.** No new library. Confirmation dialogs (Radix AlertDialog, React-Aria) were considered but rejected: DS-03 already owns the accessible modal machinery, and the typed-confirmation + single-flight + shared-Feedback semantics are DalyHub policy a general dialog library does not encode. No third-party code copied; `THIRD_PARTY_NOTICES` unchanged.
- **Consequences.** *Easy:* any module builds a settings surface at any scope by composing `SettingsLayout`/`SettingsGroup`/`SettingsRow`, picks a save model by reusing DS-06, and gets a confirmed destructive action from `DangerousAction` — no bespoke screen, toast, form engine or focus-trap. Settings drops in as the last record tab (a container-query-narrow surface) with no change. *Hard:* the change-behaviour choice (immediate vs explicit-save) is now an explicit, declared decision per setting; modules must keep product rules (what persists, what a destructive action does) in the module, not in the shared layer. *Accepted:* `useImmediateSetting` introduces a small new pure coordinator rather than forcing immediate toggles through the autosave coordinator, because their failure semantics genuinely differ (revert vs keep-draft) — this is one small model, not a second save engine.
- **Alternatives considered.**
  - *A schema-driven settings engine (declare fields as data, render generically).* Rejected (§26.2): premature and rigid for the current architecture; small typed primitives compose better and keep product control in the module. The FND-06 registry already carries the `settings` capability seam if a registry is ever needed.
  - *Route immediate toggles through DS-06 `useAutosaveField`.* Rejected (§26.3): autosave preserves the draft on failure, which is wrong for a toggle/select that must visibly revert; the one-field difference is a tiny pure model, not a second engine.
  - *Build a new confirmation modal/overlay.* Rejected (§26.4): would add a second focus-trap/overlay/z-index system; DS-03's hooks already provide accessible modal machinery, exactly as the DS-10 Inspector reuses them.
  - *Mark DS-10 fully done here.* Rejected: DS-10 shipped the interaction/feedback layer + Inspector (ADR-025); this PR completes the Settings-layout strand (DS-10b). The roadmap records both, and DS-10 becomes ☑ only once DS-10b lands.

## ADR-027: Accessibility & Responsive Baseline — automated enforcement and the inherited platform

- **Status.** Accepted (2026-07-20). Delivers [DS-11](../roadmap/ROADMAP_V2.md#-ds-11--accessibility--responsive-baseline). Builds on every prior DS/PX item (the shared components it audits) and on [ADR-017](#adr-017-design-tokens-and-the-shared-record-layout) (the token contrast/parity tests it reuses for colour), [ADR-018](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) (the DS-03 modal machinery whose keyboard/focus contract it audits centrally) and [ADR-020](#adr-020-the-application-frame--sidebar-shell-pane-collection-layout-and-entity-identity) (the shell landmarks/skip link). It closes the promise ADR-026 §26.5 made ("DS-11 will audit/automate the baseline").
- **Context.** Accessibility (WCAG 2.2 AA) and responsiveness were **requirements of every earlier item** ([AGENTS.md §15](../../AGENTS.md#15-accessibility-requirements), the Definition of Done) and were built in from DS-01 onward — but they were only ever verified per-item, by hand and by role-based Playwright assertions. There was **no automated gate**: nothing failed the build when a future change regressed a landmark, skipped a heading, broke the 320px layout or removed a focus restoration. DS-11's job is not to rebuild anything (the components are already accessible) but to **make the baseline permanent and inherited**: audit every shared surface, fix the few real gaps the audit finds, and add automated regression coverage that every future module passes through. The forces: the gate must catch genuine regressions, avoid brittle assertions, add no shipped dependency, and not fork or redesign accepted components.
- **Decision.**
  - **§27.1 The baseline lives in the shared layer; DS-11 hardens and enforces, it does not rebuild.** Every improvement is to an existing shared component, the app shell, or the tokens, so **every future module inherits it by composition** — there is no second implementation of anything (AGENTS.md §9.8). DS-11 adds **no runtime/shipped dependency, no migration, no persistence, no product feature**.
  - **§27.2 Automated accessibility gate — axe-core in Playwright.** `@axe-core/playwright` (MPL-2.0, **dev/test-only**, recorded decision in [`THIRD_PARTY_NOTICES`](../../THIRD_PARTY_NOTICES.md)) scans every `/design/*` fixture and every real route (each rendered inside the real PX-02 shell) in **light and dark**, and with the **Drawer, Search, Command Palette and dangerous-confirmation open**, scoped to the WCAG 2.0/2.1/2.2 A + AA tag set plus axe best-practice (`e2e/accessibility.spec.ts`, `e2e/helpers.ts`). It fails the build on any violation with an actionable rule/impact/target/why list. The suite runs under the existing `pnpm test:e2e`, so CI enforces it with no new job.
  - **§27.3 Documented, standing rule exclusions instead of brittle assertions.** Four axe rules are disabled globally, each with a recorded reason (`GLOBALLY_DISABLED_RULES` in `e2e/helpers.ts`): `color-contrast` (proven deterministically by the DS-01 token contrast test; pixel-derived contrast is flaky), and three that conflict with **deliberate, ADR-backed, test-asserted** designs — `landmark-unique` (DS-02's repeatable "Summary"/"Content" `region` landmarks legitimately coexist when records stack; best-practice, not AA), `nested-interactive` (DS-08/09 listbox options intentionally wrap a `tabindex="-1"` result link so records stay middle-clickable, with selection via `aria-activedescendant`), and `aria-required-children` (DS-05's grouped, virtualised `role="feed"` interleaves day headings). Every **other** rule stays enforced — including `region` and `heading-order`, which DS-11 fixed at source. The two WCAG-adjacent exclusions are tracked as [DEBT-14](../product/PRODUCT_DEBT.md) / [DEBT-15](../product/PRODUCT_DEBT.md) for a future DS-05/08/09 hardening pass. This is the task's "avoid brittle assertions": the excluded rules are brittle against accepted DalyHub contracts, not evidence of inaccessible UI.
  - **§27.4 Responsive regression matrix.** `e2e/responsive.spec.ts` sweeps every shared surface for **no horizontal overflow** across the canonical viewport matrix — **320 / 375 / 390 / 768 / 1024 / 1440 / 2560** — plus open overlays at the extremes and the mobile navigation overlay. Breakpoints stay tokens; the check is structural (`documentElement.scrollWidth ≤ clientWidth + 1`), not pixel snapshots.
  - **§27.5 Platform keyboard audit.** `e2e/keyboard.spec.ts` proves the cross-cutting guarantees once, centrally: the skip link is the first tab stop and moves focus to `main`; the shell exposes one `main` + a `navigation` landmark; Search/Command entries are reachable with no trap; and the shared DS-03 modal machinery traps focus, locks background scroll, and **restores focus to the opener** on Escape for Search, the Palette and the Drawer. Surface-specific keys (RecordTabs roving, Forms listbox, Card reorder) stay in each component's own spec. A companion `e2e/touch-targets.spec.ts` exercises the shared 44px helper (`--dh-touch-target-min`, WCAG 2.2 §2.5.8) against the shell, Command Palette and Search controls so a control regressing below the documented target size fails the build.
  - **§27.6 The real gaps the audit fixed.** (a) `viewport-fit=cover` was added to the viewport meta so the `env(safe-area-inset-*)` padding the shell/Drawer/Inspector/toast layer already reference resolves to real insets on notched devices (it was silently `0`); the mobile bar gained top/leading safe-area padding. (b) A forced-colors (Windows High Contrast) block re-pins the focus ring to the system `Highlight` colour. (c) The sidebar Search/Command block became a `search` landmark so all shell content sits inside a landmark (`region` rule). (d) **`Card` gained a `headingLevel` prop** (2|3|4, default 3) so card titles nest one level below the surrounding heading — the one real component-API addition, benefiting every future collection; the Collection/Cards fixtures were updated to model correct nesting (`heading-order` rule). (e) **The shared DS-03 focus trap (`useDrawerFocus`) was fixed** so it excludes `tabindex="-1"` controls from its focusable set: the selector matched native `button`/`a[href]` even at `tabindex="-1"`, so a listbox's inner `tabindex="-1"` option control (present in the Command Palette at rest, and in Search once results show) became the trap's "last" element and Tab from the real last control escaped to `<body>` instead of wrapping. The keyboard audit caught it; the fix hardens the trap for **every** modal that reuses these hooks (Drawer, Search, Palette, Inspector, confirmation), with a Playwright regression guard. (f) The **Search close button** was 44px wide but only 32px tall (it set `min-width` but not `min-height`); it now meets the 44px touch target in both dimensions like the Command Palette close, caught by the new touch-target spec.
  - **§27.7 Shared test helpers, not per-file duplication.** `e2e/helpers.ts` consolidates the previously copy-pasted `hasNoHorizontalOverflow`, the hydration-gated navigation and the touch-target check, and adds the viewport matrix and the axe helpers, so the whole suite asserts the baseline one way.
  - **§27.8 Open-source reuse.** Only `@axe-core/playwright`/`axe-core` (MPL-2.0, dev-only) were added; the responsive/keyboard helpers are built on Playwright built-ins; colour reuses the DS-01 token test. Assessment recorded in [`REFERENCE_PRODUCTS.md`](../reference/REFERENCE_PRODUCTS.md). No third-party code was copied into the app.
- **Consequences.** *Easy:* a future change that regresses a landmark, heading order, focus restoration, scroll lock or the 320px→ultra-wide layout fails CI automatically; a new module inherits the whole baseline by composing the shell + shared components and adds its surface to the two sweeps; `Card` titles nest correctly anywhere. *Hard:* the gate is only as wide as the surfaces enrolled in the sweeps, so **every new module must add its `/design/*` fixture or route to `accessibility.spec.ts` and `responsive.spec.ts`** (documented in [`ACCESSIBILITY_RESPONSIVE.md`](../development/ACCESSIBILITY_RESPONSIVE.md) → future-module requirements). *Accepted & disclosed:* four axe rules are globally disabled with recorded reasons (§27.3); two of them (feed children, nested option link) are genuine tightening opportunities deferred to DS-05/08/09 as DEBT-14/15 rather than redesigned in this baseline PR, to keep it focused and avoid breaking accepted, tested contracts.
- **Alternatives considered.**
  - *Rewrite DS-02/05/08/09 to satisfy `landmark-unique`/`aria-required-children`/`nested-interactive`.* Rejected for this PR: each is a deliberate, documented, test-asserted design (region landmarks; middle-clickable result links; grouped feed). Rewriting them would break their contracts and suites and materially redesign shipped components — contradicting "improve existing components only, no second implementations". Tracked as DEBT-14/15 instead.
  - *Enforce axe `color-contrast` from the browser.* Rejected: the DS-01 token contrast test is deterministic and authoritative; pixel-derived contrast is flaky and duplicative.
  - *Add a separate a11y CI job / a non-Playwright scanner (pa11y, Lighthouse).* Rejected: axe-core in the existing Playwright run reuses the browser, auth and fixtures already stood up, and needs no new job or dependency beyond the dev-only axe packages.
  - *Pixel-snapshot responsive tests.* Rejected as brittle; a structural no-overflow assertion across a fixed viewport matrix catches real layout regressions without churning on copy/spacing changes.

## ADR-028: Task Drawer persistence and composition — the additive task-detail slice

- **Status.** Accepted (2026-07-20). Delivers [TODAY-02](../roadmap/ROADMAP_V2.md#-today-02--task-drawer). Concretely extends [ADR-014](#adr-014-spine-hierarchy-completion-and-rollup-semantics) (the FND-07 spine, which it composes rather than replaces) and builds on [ADR-009](#adr-009-data-kernel-storage)/[ADR-010](#adr-010-server-side-workspace-context) (workspace-scoped D1 repositories), [ADR-011](#adr-011-entitylink-persistence-and-lifecycle) (EntityLinks), [ADR-012](#adr-012-activity-persistence-and-atomic-mutation-recording) (atomic Activity), [ADR-015](#adr-015-markdown-source-and-safe-rendering-pipeline) (Markdown source), and the shared [DS-03 Drawer](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) / DS-02 Record Layout / [DS-06 Forms](#adr-022-shared-forms--field-controls--declared-save-model-validation-boundary-and-the-entity-link-picker) / [DS-05 Timeline](#adr-021-the-shared-timeline--activity-feed--one-renderer-one-presentation-view-model-in-house-virtualisation) contracts, which it consumes unchanged.
- **Context.** [TODAY-02](../roadmap/ROADMAP_V2.md#-today-02--task-drawer) opens any task from Today in the shared Drawer as a complete, editable, persistent record — title, completion, workflow status, due/scheduled dates, priority, description, project/goal/area relationships, linked records and its activity timeline — without leaving `/today`. FND-07 (ADR-014 §4.5) deliberately kept the spine minimal (identity, title, a single `completedAt`, a structural parent link) and recorded that status, priority, dates and descriptions belong to later module work. TODAY-01 was fixture-only. TODAY-02 must therefore add the *smallest honest* persistence slice for these fields — without prematurely building the whole future Tasks module, forking any shared component, or creating a second task-detail/Drawer/form/Activity/EntityLinks system.
- **Decision.**
  - **§28.1 The spine stays authoritative; a new repository composes it.** A task remains an ordinary `entities` row plus its `spine_records` completion and its structural parent EntityLink. Identity, title-as-hierarchy, completion (`spine.complete`/`reopen`) and structural parentage stay the `SpineRepository`'s. TODAY-02 adds a workspace-bound `TaskRepository` (`app/kernel/tasks` contract, `app/platform/storage/d1/d1-task-repository.ts` adapter) that owns ONLY the additive detail fields and reads the whole task (spine + details + resolved relationships) back as one `TaskView`. It is exposed on `WorkspaceScope.tasks` alongside `spine`/`entities`/`entityLinks`/`activity`, bound to the same trusted `WorkspaceContext` + Activity actor.
  - **§28.2 One additive table, not columns on `entities`/`spine_records`.** Migration `0006_create_task_details.sql` adds a STRICT `task_details (workspace_id, entity_id, entity_type='task', status, priority, due_date, scheduled_date, description, updated_at)`, keyed `(workspace_id, entity_id)` with a composite foreign key `(workspace_id, entity_id, entity_type) → entities(workspace_id, id, type)` (the same technique `spine_records.kind` uses) so a details row can only ever attach to a `task`. It is purely additive (no backfill, no change to `entities`/`spine_records`/`entity_links`/`activities`); a task simply has no details row until first edited, and the repository treats an absent row as documented defaults.
  - **§28.3 Closed value sets in the schema.** `status` is `CHECK IN ('todo','in_progress')` and `priority` is `CHECK IN ('low','medium','high')` or `NULL` — small, closed, first-class sets, as DB CHECKs (like `spine_records.kind`). Dates are date-only `YYYY-MM-DD` (`GLOB '????-??-??'`, never routed through a timezone). `description` is Markdown **source** (FND-08), stored and rendered through the ONE shared sanitising pipeline — never HTML.
  - **§28.4 "Done" is completion, not a status value.** Completion stays the spine's `completedAt`; `status` carries only the open-state workflow position. The displayed status pill is derived (`completedAt ? "Completed" : status`), so the two can never disagree in a way the user sees, and there is no redundant second completion field. A richer editable workflow status (planned/active/on-hold/cancelled) is deferred to the future Tasks module ([DEBT-16](../product/PRODUCT_DEBT.md)).
  - **§28.5 Atomic edits with guarded Activity, writing only changed fields.** `updateTask` writes `entities.title`/`updated_at` and upserts `task_details` in ONE `D1Database.batch()`, appending exactly one `entity.updated` event guarded on the entity update's `changes()` via the shared `D1ActivityRecorder`. It writes **only the columns that actually changed** — the title is set only when it changed, and the `task_details` `ON CONFLICT DO UPDATE` touches only the changed detail columns (a title-only edit omits the details statement entirely) — so an omitted/unchanged field is never rewritten and a concurrent partial update to a *different* field cannot be clobbered by this update's stale snapshot (the "omitted fields are left unchanged" contract holds under concurrency). A no-op update writes nothing and appends nothing; a task deleted mid-flight (the guarded update matches nothing) writes nothing and surfaces `TaskNotFoundError`. Completion continues to append `task.completed`/`task.reopened` through the spine. There is **no** second task-history table — the shared Activity model is the only event source, rendered by the DS-05 Timeline.
  - **§28.5a Non-task ids never reach a mutation.** Because every task route is addressed by a task id, the action resolves `tasks.getTask(taskId)` (a task-typed read) and returns the calm not-found **before** dispatching any mutation — so a Project/Goal/Area id can never reach `spine.complete`/`reopen` (which also complete Goals/Projects) or become a `task.relates_to` picker anchor, and nothing is mutated for a non-task id. The `/link-targets` search loader applies the same guard before searching.
  - **§28.5b Today-card completion is never silently lost.** The Today focus card completes optimistically and writes through the same `/today/task/:id` action via `useFetcher`; a revalidation reconciles the optimistic override, and a small hook watches the fetcher's outcome and raises the shared DS-10 error notification on a failed completion, so the owner always learns when a completion did not persist (the Drawer's completion control already surfaces its own failures).
  - **§28.6 Real, derived relationships.** `project`/`goal`/`area` on a `TaskView` are resolved by walking the spine hierarchy (structural parent → the project's `advances_goal` goal → the area), never stored twice, so they are true relationships not copied labels. A task's structural parent is exactly one of an Area or a Project; the Goal (and the Area for a project-parented task) are derived. Additional non-structural associations use a NEW **non-reserved** `task.relates_to` EntityLink through the existing FND-04 repository and the DS-06 entity-link picker service (`createLinkWithPolicy`/`unlinkWithPolicy`), so link/unlink is server-policy-enforced and workspace-isolated; structural parentage is never mutated through the generic link repository.
  - **§28.7 Composition only — no forked shared UI.** The Drawer body is the shared DS-02 Record Layout (Header + Summary + Details/Links/Activity tabs, Activity last per the shared tab vocabulary), hosted by the existing DS-03 `DrawerProvider` on `/today` — no second Drawer, Record Layout, form system, Activity model or EntityLinks implementation. The editable Details form is DS-06 (`useForm`, `TextField`/`MarkdownField`/`SelectField`/`DateField`, explicit Save/Cancel, server-authoritative validation, `UnsavedChangesGuard` wired to the Drawer key). The task record composition lives in the **today** module (`app/modules/today/task/*`) — it does not import the `tasks` module's internals (the module-boundary rule) — with three module-owned resource routes (`/today/task/:taskId` loader+action, its `/activity` Timeline page, its `/link-targets` search) addressed by the Drawer content so the user never leaves Today.
  - **§28.8 Today integration by data-source swap.** The `/today` loader replaced only the *focus* fixture seam with real workspace-scoped tasks (`tasks.listTasks`, completed included so completion stays stable on the card); the other sections stay fixture-backed. A focus-card completion writes through the same `/today/task/:id` action the Drawer uses, and React Router revalidation keeps Today and the Drawer consistent with no hard reload. The dashboard composition is unchanged (it gained one optional `onCompleteTask` callback and a server `done` field).
  - **§28.9 No new dependency.** The whole slice is built on the existing TypeScript/D1/Workers/React Router/Vitest/Playwright stack — a kernel contract, a D1 adapter, one additive migration and shared-component composition. Assessment recorded in [`REFERENCE_PRODUCTS.md`](../reference/REFERENCE_PRODUCTS.md).
- **Consequences.** *Easy:* a task is a full, persistent, editable record composed entirely from the shared layer; completion has a single authority (the spine); every meaningful edit appends one uniform Activity event with no bespoke history; the future Projects/Tasks surfaces read the same `TaskView` through one typed seam; relationships are always consistent because they are derived. *Hard:* `updateTask` is a trusted domain writer of `entities.title` alongside `spine.rename` (both guarded on `type='task'` and active, both emitting `entity.updated`) — the two title-writers must stay in step; the task-record UI currently lives in the today module and will relocate when a real Tasks module exists. *Accepted & disclosed:* a deliberately minimal detail model (two-value status, three-value priority, two dates, one Markdown description) and derived-only relationships, with the richer workflow status, a Waiting/blocking field (TODAY-03) and search-over-real-tasks deferred and recorded as [DEBT-16](../product/PRODUCT_DEBT.md)/[DEBT-17](../product/PRODUCT_DEBT.md).
- **Alternatives considered.**
  - *Add status/priority/date/description columns to `entities` or `spine_records`.* Rejected: ADR-009 keeps `entities` a generic substrate and ADR-014 §4.5 explicitly excludes these from the spine; an additive per-task table keeps both kernels clean and the composite FK guarantees task-only attachment.
  - *A separate free `status` enum that includes "done".* Rejected: it would duplicate the spine's `completedAt` and invite states the user sees disagree; deriving the pill from completion keeps one source of truth.
  - *Build the full standalone Tasks module (creation, board, planning, Waiting).* Rejected as out of scope for TODAY-02; only the fields the Task Drawer needs were added, per the roadmap's "do not prematurely implement the entire future Tasks module".
  - *Model project/goal/area as stored columns on the task.* Rejected: they are already the spine hierarchy; storing them would duplicate a relationship EntityLinks/spine own and could drift.
  - *A module-specific task detail panel / second Drawer.* Rejected (AGENTS.md §9.8): the shared DS-02/03/05/06 compose into a complete task record; a bespoke panel would be Product Debt the moment it merged.

## ADR-029: Task Waiting — additive state, a reserved EntityLink, and a derived first-class display state

- **Status.** Accepted (2026-07-20). Delivers [TODAY-03](../roadmap/ROADMAP_V2.md#-today-03--waiting). Extends [ADR-028](#adr-028-task-drawer-persistence-and-composition--the-additive-task-detail-slice) (the additive `task_details` slice it composes), [ADR-014](#adr-014-spine-hierarchy-completion-and-rollup-semantics) (completion authority) and [ADR-011](#adr-011-entitylink-persistence-and-lifecycle) (EntityLinks); consumes the DS-02/03/04/05/06/07 shared UI unchanged. Resolves the Waiting half of [DEBT-16](../product/PRODUCT_DEBT.md#-debt-16--minimal-task-detail-model-no-rich-workflow-status-or-waiting--p3).
- **Context.** TODAY-03 makes "waiting for" a real, persistent workflow: mark a task as blocked on someone/something, record what/whom and since when, see all waiting tasks in one view, and clear it — with meaningful Activity, on desktop and mobile. The audit found no existing waiting concept: `TaskStatus` is a closed `todo`/`in_progress` set, completion is the spine's `completedAt`, and `task_details` explicitly excluded waiting. The task-domain precedent (ADR-028) is an additive `task_details` slice that COMPOSES the spine; the relationship precedent (ADR-014/028) is that a task's real relationships (project/goal/area, `task.relates_to`) are EntityLinks resolved live, never copied labels.
- **Decision.**
  - **§29.1 A combination model, chosen after auditing.** Waiting is the smallest honest additive slice: two nullable columns on `task_details` (`waiting_since`, `waiting_note`) for the STATE and a free-text subject, plus a reserved `task.waiting_on` **EntityLink** for an entity-backed subject (Person/Project/Goal/Area/Task), resolved live to its current title like the structural parent. Rejected alternatives: (a) a dedicated waiting table — a second store when `task_details` already is the task-domain layer and Activity already provides history; (b) storing the entity target as `waiting_target_id`/`type` columns — would duplicate a relationship EntityLinks own and lose bidirectional visibility; (c) a free-text-only model — would force a Person record for "waiting for the electrician". `waiting_since IS NOT NULL` is the single authority for "is waiting"; a note requires it; the subject is EXACTLY ONE of a note XOR an active `task.waiting_on` link (application-enforced, tested — a portable `ALTER TABLE ADD COLUMN` cannot express a cross-column/cross-table CHECK without a destructive rebuild).
  - **§29.2 The TaskRepository owns waiting atomically; `task.waiting_on` is reserved.** `setWaiting`/`clearWaiting` each write the state, replace/clear the `task.waiting_on` link and append exactly one Activity event in ONE `D1Database.batch()` guarded on the entity update's `changes()` — mirroring how the SpineRepository writes structural links in-batch. `task.waiting_on` is added to a `RESERVED_TASK_LINK_TYPES` set the generic EntityLink repository refuses (like `RESERVED_SPINE_LINK_TYPES`), so the TaskRepository is the SOLE writer and the stored state and the link can never diverge. Migration `0007` adds a partial unique index enforcing at most one active `task.waiting_on` per task; changing the subject soft-deletes the old link and creates/restores the new one in the same transaction, so there is never a duplicate active waiting link and re-waiting on a prior subject restores its stable link id.
  - **§29.3 Waiting is a derived first-class display state, not a status value.** Precedence in the pill: completion → waiting → the open-state workflow position (`completedAt ? "Completed" : waiting ? "Waiting" : status`). `status` stays `todo`/`in_progress`, so a task can never present as "Completed and Waiting" or "In progress and Waiting" as competing primary states, and there is no visual-only flag. The Waiting view, Today cards, the Task Drawer, filters (exclusion from focus) and Activity all read the same derived state.
  - **§29.4 Semantics (documented, predictable).** One active waiting state and one primary subject at a time; changing the subject PRESERVES the original `since` (the same waiting episode). Completing a task CLEARS its waiting state (see §29.4a) so a completed task never shows active-waiting UI and never appears in the Waiting view; reopening does NOT restore it. A deleted/unlinked entity target degrades gracefully to an unresolved subject ("someone no longer available"), never a crash. Cross-workspace targets, non-task anchors, the self-target and disallowed target types are rejected server-side; a rejected or no-op mutation appends no Activity.
  - **§29.4a Completion-and-clear is ONE atomic task-domain operation.** `TaskRepository.completeTask(taskId)` is the sole path for completing a task: a single `D1Database.batch()` performs the spine completion, the `updated_at` bump, the `waiting_since`/`waiting_note` clear, the active `task.waiting_on` soft-delete, the `task.completed` event and — ONLY when the task was actively waiting — exactly one `task.waiting_cleared` event. Either everything commits or nothing does, so a task can never be left completed-but-still-waiting. The **FND-07 spine remains the completion authority**: the completion write is the shared `buildSpineCompleteStatement`/`buildEntityUpdatedAtBumpStatement` builders (extracted from the SpineRepository and reused by both, so the completion SQL lives in ONE place — no duplication). The route calls this ONE operation; it does **not** coordinate the invariant through `spine.complete()` + `tasks.clearWaiting()` (two separate transactions, the pre-fix bug), and there is no route-level compensation. The waiting-clear statements gate on the freshly-written completion (`EXISTS spine_records.completed_at = nowTs`) and on `waiting_since IS NOT NULL`, so the `task.waiting_cleared` event is appended iff the task was waiting AND completion committed. Completing an already-completed task is an idempotent no-op (no batch, no Activity). Reopening still goes through `spine.reopen` and never restores waiting. Proven by real-D1 rollback tests injecting a forced failure at each post-completion statement (waiting update, waiting-link mutation, and either Activity insert): after the failure, `completed_at`, `waiting_since`, `waiting_note` and the active link are all unchanged and no completion or waiting-clear Activity survives.
  - **§29.5 Server boundary & routes.** Waiting is driven by two new `intent`s on the existing `/today/task/:taskId` action (`set_waiting`/`clear_waiting`) — no fragmented endpoints — plus a `/today/task/:taskId/waiting-targets` entity search and a real `/today/waiting` view (registry route under Today, no separate sidebar module). All authenticate through the accepted boundary, resolve the workspace from trusted server context, verify the anchor is an active task, return calm 404s, use parameterised SQL, return typed discriminated results and `cache-control: no-store`.
  - **§29.6 Activity.** Three new module-registered (non-reserved) types — `task.waiting_started`, `task.waiting_changed`, `task.waiting_cleared` — declared by the **tasks** module manifest (which already owns `task.completed`/`task.reopened`) and given DS-05 Timeline descriptors. Payloads are structured and safe (subject kind, and an entity's non-sensitive id/type) — free-text content is never logged. Reuses the shared Activity model and Timeline; no task-history table.
- **Consequences.** *Easy:* a task carries a persistent, editable waiting state composed entirely from the shared layer; the Waiting view, Today summary and Drawer read one derived state; the entity subject is a real relationship (a Person's tasks-waiting-on-them become visible for free later); completion and waiting can never visibly contradict. *Hard:* the TaskRepository is now a trusted writer of a reserved EntityLink type alongside the spine's structural writers — the reserved set and the one-active partial index are load-bearing. *Accepted & disclosed:* one primary subject only (multi-target waiting is out of scope); the cross-field "exactly one subject" invariant is application-enforced, not a DB CHECK (documented, tested); the entity `waiting_on` target is NOT added as an Activity subject yet (the target's timeline does not surface "you are blocking task X" — a deliberate, minimal deferral).
- **Alternatives considered.** A dedicated `task_waiting` table (rejected §29.1); entity target as plain columns (rejected §29.1 — duplicates an EntityLink relationship); `waiting` as a `status` enum value (rejected — would duplicate/contradict completion and the open-state status, per §29.3); managing the waiting link through the generic DS-06 picker service like `task.relates_to` (rejected — cannot write the link, the state column and the Activity event atomically, and would produce two events for one action).

## ADR-030: Task Planning — the scheduled date as commitment, atomic single + bulk mutations, and planning Activity

- **Status.** Accepted (2026-07-21). Delivers [TODAY-04](../roadmap/ROADMAP_V2.md#-today-04--planning). Extends [ADR-028](#adr-028-task-drawer-persistence-and-composition--the-additive-task-detail-slice) (the additive `task_details` slice it reuses — `scheduled_date` already exists) and [ADR-029](#adr-029-task-waiting--additive-state-a-reserved-entitylink-and-a-derived-first-class-display-state) (waiting independence), builds on [ADR-012](#adr-012-activity-persistence-and-atomic-mutation-recording) (atomic Activity) and [ADR-014](#adr-014-spine-hierarchy-completion-and-rollup-semantics) (completion authority), and consumes the DS-02/03/04/06/07 and PX-02 shared UI unchanged.
- **Context.** TODAY-04 turns Today into a deliberate planning workspace: each morning the owner decides what to do today, what can wait, and what moves to another day. The audit found the substrate already exists — TODAY-02 (ADR-028) added `task_details.scheduled_date` (a date-only `YYYY-MM-DD`), and `TaskView`/`TaskListItem` already carry it. Planning must therefore reuse that field rather than invent a second store or a parallel planning model, must not conflate the scheduled date (the owner's commitment) with the due date (a hard deadline), and must keep planning fully independent of waiting and completion. This is task planning, not calendar scheduling: no recurrence, reminders, notifications, AI, estimates, dependencies, drag-between-dates or month view (all explicitly out of scope, deferred to later items).
- **Decision.**
  - **§30.1 Planning is the scheduled date; no migration, no second model.** The scheduled date IS the commitment ("I intend to work on this today"); the due date remains "must be finished by". Planning mutates ONLY `task_details.scheduled_date` — it never touches the due date, the waiting columns/link, or `spine_records.completed_at`. No new table, column, index or migration is added; TODAY-04 is pure behaviour + UI over the ADR-028 slice.
  - **§30.2 Atomic single-task mutations with guarded planning Activity.** The workspace-bound `TaskRepository` gains `planTask(id, {scheduledDate})` and `clearPlan(id)`. Each is ONE `D1Database.batch()`: a guard-anchor `entities.updated_at` bump (RETURNING), exactly one `changes()`-guarded Activity event, and a `task_details` write that sets or clears ONLY `scheduled_date` (the INSERT branch supplies the row's other columns from the current view so a first-plan creates the details row without disturbing anything else). The event type is derived from the transition: `task.planned` (was unplanned), `task.rescheduled` (moved to a different date) or `task.plan_cleared`. A no-op (same date, or clearing an unplanned task) writes nothing and appends nothing (`changed: false`), exactly like `updateTask`/`clearWaiting`. Three new module-registered (non-reserved) Activity types are declared on the **tasks** manifest with DS-05 Timeline descriptors; **payloads carry only the non-sensitive calendar dates** (`scheduledDate`, `previous`) — never free text — and there is no second task-history table.
  - **§30.3 Bulk planning is ONE atomic operation.** `planTasks(ids, {scheduledDate})` and `clearPlans(ids)` validate + de-duplicate the id list (bounded by `MAX_PLAN_BATCH_SIZE`), then **resolve every id first** — any id that is not a task in this workspace (missing, soft-deleted, wrong type, or cross-workspace) rejects the WHOLE operation with `TaskNotFoundError`, so nothing is partially applied. Only tasks whose date actually changes contribute statements; each contributes its own `[bump, guarded-activity, subjects, details]` group to a single batch, and because each group's own bump immediately precedes its activity insert, the `changes()` guard resolves correctly for many events in one transaction (the same technique `completeTask` uses). Tasks already in the requested state are counted as `unchanged`. Either every change commits together or none does.
  - **§30.4 Independence is a hard, tested rule.** Planning never changes the due date (a distinct column, never written), never changes waiting (the waiting columns/link are untouched, so "removing a plan never changes waiting" holds), and never changes completion (the spine is not touched). Conversely, waiting tasks are EXCLUDED from the planning sections by the loader (`excludeWaiting: true`), so a waiting task never silently becomes today's work; completed tasks appear ONLY under "Completed today". These are proven by regression tests.
  - **§30.4a Planning applies to OPEN work only — enforced server-side, race-safe.** A completed task can never be planned, rescheduled, plan-cleared or bulk-planned, and no planning Activity is ever recorded against completed work — even by a crafted or stale request. Each planning mutation rejects a completed task up front (`completedAt !== null` → a `TaskValidationError` in the existing task error family, surfaced as a calm message; no new transport error), AND the invariant is re-enforced INSIDE the guarded write: the planning guard-anchor bump and both `task_details` writes gate on `EXISTS (spine_records.completed_at IS NULL)`. So if a completion RACES between the read and the write, the open-gated bump matches nothing → `changes() = 0` → the guarded planning Activity is not appended and the `scheduled_date` write no-ops; the mutation is then rejected (a re-read distinguishes a completed vs. deleted racer) rather than silently swallowed. Bulk planning stays atomic: any supplied id that is missing, cross-workspace, deleted OR completed rejects the WHOLE operation before any write, and the open-gated per-task writes are a second line of defence if one completes mid-batch. Proven by real-D1 tests including a `planRaceHook` that injects a concurrent completion between validation and the write.
  - **§30.5 The surface is re-composed by a pure view-model.** A React-free `planning-view` buckets tasks by their scheduled date relative to the owner's calendar day (`ownerCalendarIso`, the TODAY-01 timezone seam) into **Overdue** (scheduled before today), **Today** (the commitment), **Upcoming**, **Anytime** (unscheduled backlog) and **Completed today** (completed with an owner-calendar completion date of today), plus a calm `planningSummary` (planned · overdue · waiting · completed today) and the quick-plan target dates (`planTargets`: today/tomorrow/next-week via deterministic UTC date arithmetic). It is a pure function of typed data + `todayIso`, so it is unit-tested directly and renders identically on server and client. The `/today` loader feeds it from a **dedicated planning query** (`TaskRepository.listPlanningTasks`, the `listWaitingTasks` precedent) — NOT a single generic capped list. The planning query fetches three bands INDEPENDENTLY and bounds each: all scheduled (planned) open tasks ordered scheduled-date ascending (so overdue/today are preserved first and only far-future upcoming is ever truncated), a bounded page of the unscheduled backlog, and a bounded page of the most-recent completions (filtered to "completed today" in the owner's timezone by the caller). This guarantees a large early-due unscheduled backlog can never crowd out the owner's real commitments or today's completions — the defect a single `listTasks({includeCompleted})` page (ordered by due date, capped) would have had.
  - **§30.6 Composition only — shared UI, no forked patterns.** The dashboard planning sections are DS-04 Cards in the PX-02 CollectionLayout; each card carries contextual plan quick actions; multi-select uses the DS-04 Card `selection` + the CollectionLayout **selection slot** for a bulk action bar (Plan today / Tomorrow / Next week / Clear plan / inline custom date). The Task Drawer gains a **Planning section** in the DS-02 Summary (Scheduled + Due read display, then Today / Tomorrow / Next week / Clear / an inline DS-06 `DateField` — no modal-in-modal); its Scheduled/Due are removed from the summary metadata to avoid duplication. Single-task planning posts `plan`/`clear_plan` intents to the existing `/today/task/:taskId` action; bulk + per-card planning posts to a new action-only `/today/plan` resource route. Both authenticate through the accepted composition boundary, resolve the workspace from trusted server config, use parameterised SQL and `no-store`.
  - **§30.7 Keyboard commands are exposed, not owned.** Planning is exposed as shared contextual `AppAction`s while a task's Drawer is open — "Plan for Today" (`P`), "Move to Tomorrow" (`Shift+P`) and "Clear plan" — carrying shortcut metadata and driving the SAME `submitPlan` path as the cards and bulk bar (one identity, one execution path, ADR-024). The full keyboard palette + global shortcut dispatch remain TODAY-05's; TODAY-04 only makes the commands architecturally ready.
  - **§30.8 No new dependency.** Built entirely on the existing kernel/D1/React Router/Vitest/Playwright stack — repository methods, a pure view-model, one resource route and shared-component composition.
- **Consequences.** *Easy:* planning is a first-class, persistent, undoable-by-re-planning workflow composed entirely from the shared layer; the scheduled date has one meaning everywhere; every plan change is one uniform Activity event; bulk planning is safe (atomic, workspace-isolated); the future Projects/Tasks surfaces read the same `TaskView`/list seam. *Hard:* the `TaskRepository` is now the writer of `scheduled_date` through both `updateTask` (the raw Details form field) and the planning mutations — both are guarded on the active task and both keep the field date-only, but the two writers must stay in step (the Details form still exposes the raw scheduled-date field for completeness). *Accepted & disclosed:* the planning surface is bounded (`PLAN_LIMIT`, a calm daily view, not a report); an **Anytime** backlog section is added beyond the brief's five section names as the natural source the owner plans FROM (Things-3-style); "completed today" uses the owner-calendar date of a UTC completion instant (the documented SET-01 timezone seam); per-card actions surface the common plan actions while the full set (incl. custom date) lives in the Drawer and bulk bar for calm at 320px.
- **Alternatives considered.**
  - *A dedicated `planning`/`plan_date` table or a standalone planner module.* Rejected: `task_details.scheduled_date` already models the commitment; a second store would duplicate a field and invite drift, and the brief explicitly excludes a standalone planner.
  - *Route planning through the generic `updateTask` (which already writes `scheduled_date`).* Rejected: planning needs its own legible Activity (`task.planned`/`rescheduled`/`plan_cleared`), transition-aware and free of the raw `entity.updated` field-diff — so a dedicated atomic mutation is clearer and avoids a duplicate history.
  - *Best-effort bulk planning (skip missing ids, apply the rest).* Rejected: "bulk planning is atomic" — resolving all ids first and rejecting the whole operation on any invalid id keeps the outcome predictable and cross-workspace-safe, with no partial application.
  - *A separate scheduled-date store distinct from the due date, or a single "date" field.* Rejected: the scheduled date (commitment) and the due date (deadline) are deliberately distinct concepts and must never be merged (a core TODAY-04 rule); they are two existing columns.

## ADR-031: Today keyboard workflow — contextual command ownership, one shortcut dispatcher, the editable-control boundary, and roving focus

- **Status.** Accepted (2026-07-21). Delivers [TODAY-05](../roadmap/ROADMAP_V2.md#-today-05--keyboard-workflow). Builds ENTIRELY on [ADR-024](#adr-024-command-palette--quick-actions--command-kinds-trusted-catalogue-authenticated-execution-and-one-shared-action) (the one command/shortcut model), [ADR-025](#adr-025-the-global-interaction-layer--feedback-platform-notifications-undo-background-operations-and-the-shared-inspector) (the Feedback platform that unblocks global run-shortcut dispatch), [ADR-018](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) (the Drawer lifecycle + shared modal machinery), and [ADR-028](#adr-028-task-drawer-persistence-and-composition--the-additive-task-detail-slice)/[029](#adr-029-task-waiting--additive-state-a-reserved-entitylink-and-a-derived-first-class-display-state)/[030](#adr-030-task-planning--the-scheduled-date-as-commitment-atomic-single--bulk-mutations-and-planning-activity) (the task routes/domain operations it drives). **No migration, no new store, no second command registry, no second palette, no Today-only keyboard engine.**
- **Context.** TODAY-04 (ADR-030 §30.7) made planning commands *architecturally ready* but only dispatchable inside the palette while a Drawer was open. TODAY-05 must make Today fully operable without a mouse: navigate the task collection, open/close/complete/reopen tasks, plan/clear, mark/clear waiting, invoke Quick Capture, move between sections, and find/execute everything through the DS-09 palette — deliberately, discoverably and safely, reusing the existing systems rather than adding a parallel one.
- **Decision.**
  - **§31.1 One dispatcher, now including contextual run shortcuts.** The ONE shared dispatcher (`useCommandShortcuts`, installed once by `CommandShortcutLayer`) is extended to dispatch **contextual `run` action** shortcuts globally (`P` / `Shift+P` / `C`), not only navigation shortcuts. This was deferred at DS-09 because a run shortcut fired with the palette closed needs a pending/success/failure surface OUTSIDE the palette; DS-10 shipped it (the shared Feedback platform), and a contextual run action reports its own feedback through it — so global dispatch is honest. Precedence is unchanged (reserved → contextual → registered); a disabled action still yields `enabled: false` and never fires. **Registered `execute` (server) command shortcuts stay deferred** — they need the authenticated-execution feedback surface, which no module uses yet. The palette now advertises a shortcut hint for navigation commands AND contextual actions (both are dispatched), never for a registered executable command — so no hint promises a control that does nothing.
  - **§31.2 The editable-control boundary is the dispatcher's, reused not re-derived.** Single-key shortcuts never fire while focus is in an `input`, `textarea`, `select`, `contenteditable` element, date picker, autocomplete or any form control — this is the dispatcher's existing `isTypingTarget` guard (`Mod+K` remains the only `allowInInput` binding). TODAY-05 adds NO document-level key listeners of its own for shortcuts; every Today shortcut is a contextual `AppAction` with `shortcut` metadata that rides that one dispatcher. Shortcuts also do not fire while a modifier the shortcut does not require is held (the exact-match rule in `matchesShortcut`).
  - **§31.3 Contextual command ownership: the Drawer owns the open task, the dashboard owns the ACTIVE roving task only.** Per-task commands are a single `AppAction` set built by a pure `buildFocusedTaskCommands` (Open/Close, Complete/Reopen, Plan today/tomorrow/next-week, Clear plan) plus a state-dependent Clear-waiting. **When a task's Drawer is open, `TaskDrawerContent` registers that task's commands** — it is the one place with the task's live state (completion, scheduled date, waiting) AND the refresh path that keeps the Drawer's own planning/completion/waiting displays consistent after a keyboard mutation; its commands drive the Drawer's own callbacks (`toggleCompletion`/`planTask`/`clearPlan`/`clearWaiting`) which revalidate both the Drawer and `/today`. **The dashboard registers the roving task's commands ONLY when BOTH (a) no Drawer/overlay is open AND (b) keyboard focus is currently WITHIN the task collection.** This closes a stale-shortcut defect: the roving *tab-stop* id is deliberately RETAINED when focus leaves the collection (so Shift+Tab restores focus to the same card), but it must NOT remain the command target — otherwise pressing `C`/`P`/`Shift+P` behind the keyboard-help / a project/note/upcoming Drawer, or after Tabbing out to Quick Capture, would complete or replan a task the owner can no longer see. The roving-focus controller therefore exposes a `focusWithin` flag (tracked by container `focusin`/`focusout`) and an `activeId` (the focused id ONLY while `focusWithin`) distinct from the retained `focusedId`; the dashboard's target is `drawerOpen ? null : activeId`. The two registration sites stay mutually exclusive (a Drawer-open task is never also the dashboard's active task), so there is no double registration and no stale Drawer. Availability is by OMISSION and reflects task state: a completed task exposes only Reopen (never a planning command); an unplanned task exposes no Clear plan; only a waiting task exposes Clear waiting. The security/correctness boundary stays the server — a stale client is still rejected by the trusted route (ADR-028/029/030); UI availability is never the boundary. This is a controller-state fix, NOT a second listener or registry.
  - **§31.4 Global Today commands are contextual, discovered through the surface — not hard-coded in the palette.** Navigation commands that work from anywhere (Open Today, Focus Quick Capture, Open Waiting) stay REGISTERED module commands (the existing `commands.ts` manifest). Commands that only make sense on the Today surface and need a client handler (Focus task list, Go to <section>, Select all open tasks, Clear selection, Keyboard shortcuts) are transient CONTEXTUAL actions the dashboard registers through `CommandContextProvider` — the palette's documented "context" group. Nothing Today-specific is hard-coded inside the palette component; both kinds are discovered through the existing registry/context system.
  - **§31.5 Roving focus over the open task collection — a single composite widget.** The open planning sections (Overdue/Today/Upcoming/Anytime) are wrapped in ONE plain container that behaves as a single composite widget with **exactly one tab stop**: Tab enters the collection once and lands on the current task, Arrow keys move between tasks, and Tab (or Shift+Tab) leaves (re-enters) the collection — the owner never Tabs through every card's controls. A DS-04 Card gains an additive, backward-compatible `rovingTabIndex` prop applied to **only the card's primary open control** (`0` for the active card, `-1` for the rest); the card's SECONDARY controls (selection checkbox, quick/overflow action buttons) are always taken out of the tab order (`tabindex="-1"`) while roving is active, so Tab never stops on them, yet each stays fully operable — the checkbox by **Space** on the focused card (the collection's keyboard model), and every quick action by its **direct shortcut** (`C`/`P`/`Shift+P`) and its **contextual command / Command Palette** entry (one identity, one execution path). This is the accessible roving-`tabindex` composite the design system already uses for RecordTabs/reorder — deliberately NOT a `listbox`/`aria-activedescendant` widget, since a card legitimately contains interactive children (an open link, a checkbox, action buttons) that a listbox option may not. A small `useTodayRovingFocus` hook + a pure, unit-tested `roving-model` own the arithmetic: Arrow Up/Down cross section boundaries and clamp at the ends (no wrap), Home/End move within the current section, Enter opens, Space selects. The container's keyboard/focus listeners are attached imperatively to the ref (so the wrapper carries no interactive role and the cards remain the interactive elements). **Section jumps ("Go to <section>") ESTABLISH that section as the navigation context** — they move focus to the section's first task (a pure `sectionFirstIdOf` helper) so it becomes the single tab stop and subsequent Arrow/Home/End continue from that section, not the previous one, and scroll its heading into view. They are NAVIGATE commands (see §31.5c), so the palette closes and the target survives focus restoration deterministically.
  - **§31.5a Multi-select is fully keyboard-operable.** Space enters selection on the focused task; "Select all open tasks" fills it; the TODAY-04 bulk action bar (Plan today/Tomorrow/Next week/Clear/custom date) lives in the CollectionLayout selection slot OUTSIDE the roving container, so it is reached by an ordinary Tab; and — added here — bulk **commands** ("Plan selected for Today/Tomorrow/Next Week", "Clear plan for selected") drive the SAME atomic `/today/plan` path from the palette when a selection exists. Escape (or "Clear selection") exits selection. No partial application.
  - **§31.5b Command coverage against the roadmap (explicit — nothing silently omitted).** Present as commands/shortcuts: navigate Today, open/close/complete/reopen a task, plan today/tomorrow/next-week, clear plan, clear waiting, select-all/clear-selection, bulk plan, keyboard help, open Waiting. Present through keyboard-accessible **visible controls** rather than a dedicated command (a deliberate, documented boundary — the control needs typed input or is already a first-class keyboard widget, so a command would be a redundant or dishonest placeholder): **mark waiting** and **change waiting subject** (the DS-06 waiting editor in the Drawer), **choose a custom planning date** (the DS-06 `DateField` in the Drawer Planning section and the native date input in the bulk bar), and **open Task Activity** (the DS-02 RecordTabs tablist — a roving-tab keyboard widget). These are tracked in [DEBT-18](../product/PRODUCT_DEBT.md) as future dedicated commands; none is missing from the keyboard, only from the palette.
  - **§31.6 Direct shortcuts, calm and conflict-free.** `P` (plan today), `Shift+P` (tomorrow), `C` (complete/reopen), `Enter` (open focused), `Space` (select focused), `Escape` (close the top overlay via the existing hierarchy, or clear a selection when focus is in the collection and no overlay is open), `?` (keyboard reference). They do not collide with the browser, text editing, form controls, the reserved palette/Search vocabulary, or the Drawer's own keys. The `g`+letter go-to chords, `j/k`, `x`, `o`, `e` and `[` remain reserved for a future cross-app pass; TODAY-05 does not claim them.
  - **§31.6a Only the interactive TOP drawer owns its task shortcuts.** The shared stack keeps a lower drawer MOUNTED when another is pushed above it (its local state intact), so a task drawer's `C`/`P`/`Shift+P`/Clear-waiting must not keep firing against the now-hidden task behind, say, the keyboard-help drawer. `TaskDrawerContent` receives `isTop` (from `DrawerEntry.isTop`, threaded by the renderer) and registers its contextual task commands ONLY while `isTop` — when it drops to a lower level the commands are unregistered (empty registration) and return when it becomes top again; local state is preserved because the component never unmounts. This is a registration-lifecycle gate, NOT a new listener or a task-aware drawer system.
  - **§31.5c "Go to <section>" / "Focus task list" are NAVIGATE commands, not run actions — so the palette closes and the target survives focus restoration.** A contextual `run` action does not close the palette (only a target-bearing or navigation command does), and when the user later closes the palette it restores focus to its opener — which, if the palette was opened from a focused task, re-fires the collection's `focusin` and overwrites the section target back to that task. The fix reuses the EXISTING Focus-Quick-Capture pattern: these commands carry a validated `/today?…&today-nav=<list|bucket>` target, built by a pure, unit-tested `keyboard/nav-target.ts` (`buildTodayNavTarget`). Crucially the target **removes the ENTIRE Drawer stack** from the current params — via the shared `withAllDrawersRemoved` helper (ADR-018), never by hand-parsing `drawer` keys — while preserving every OTHER param. Removing the stack (rather than preserving `drawer=` params) is what makes a section command run from *inside* an open drawer, or a stack of them, navigate the drawers away cleanly in one forward push, without touching the Drawer provider's own history entry or push token: the browser Back button reopens the previous drawer and Forward returns to Today with it closed, and the roving context stays coherent. The `today-nav` value is bounded by a type guard (`isTodayNavValue`) so an arbitrary query value can never become a section identifier. Navigating naturally closes the palette AND the drawer stack (the controller's existing navigation path plus the URL-driven Drawer lifecycle), and a single post-navigation effect — depending only on the search params + stable callbacks, so it fires once per navigation, never per render — moves focus to the target task after the modal surfaces have unmounted and cleans the param via a `replace`. Because the effect runs on the post-navigation render (after the palette closed and restored focus), the target wins DETERMINISTICALLY, with no `setTimeout`/`requestAnimationFrame`, no re-setting after unmount, no manual `closeDrawer`/history manipulation, and no suppression of focus restoration.
  - **§31.7 Keyboard help reuses the shared Drawer.** The discoverable shortcuts reference is hosted by the SAME DS-03 Drawer (a `help:shortcuts` key), reusing its focus trap, inert background, scroll lock and focus restoration — no bespoke help modal, no second focus trap. It is Today-scoped by design (it documents the Today execution workflow); a cross-app `?` overlay can be layered on later without changing this contract.
  - **§31.8 Quick Capture stays honest.** The keyboard path reuses the existing (registered) Focus-Quick-Capture command / the header action — it focuses the existing control and does not pretend to persist (TODAY-01's disclosed fixture boundary is unchanged; TODAY-05 adds no capture persistence).
  - **§31.9 No new dependency.** Built on the existing command/Drawer/Feedback/Card systems, a pure view-model, one shared hook and small additive props.
- **Consequences.** *Easy:* Today is fully keyboard-operable through the same trusted routes and one command identity as the mouse; the palette advertises exactly the shortcuts that actually fire; a keyboard mutation refreshes the Drawer and Today consistently; the roving/Card/help pieces are reusable by future task-like surfaces. *Hard:* the open-task command set now lives in `TaskDrawerContent` and registers only after the Drawer's async load (a brief, correct delay); the Card gains one optional prop it must thread to its controls. *Accepted & disclosed:* Marking/changing the waiting SUBJECT needs typed input, so those stay on the visible, keyboard-accessible waiting control (not a placeholder command) — only Clear waiting is a command; the keyboard help and the section-focus commands are Today-scoped; roving covers the open sections (the collapsed "Completed today" keeps natural tabbing); the broader reserved vocabulary (`g`-chords etc.) is left to a future cross-app item ([PRODUCT_DEBT DEBT-18](../product/PRODUCT_DEBT.md)).
- **Alternatives considered.**
  - *A Today-only keyboard engine / document key listeners in components.* Rejected by the brief and the architecture: there is ONE dispatcher and ONE command model; Today contributes data (contextual `AppAction`s), not a parallel engine.
  - *Register the open task's commands from the dashboard (driving the bulk `/today/plan` route).* Rejected: the dashboard's route revalidation does not refresh the Drawer's independently-fetched record, so a keyboard plan would leave the open Drawer's Planning display stale. The Drawer owns its task's commands because it owns the refresh path.
  - *A `listbox`/`aria-activedescendant` composite for the task collection.* Rejected: cards legitimately contain interactive children (an open link, action buttons, a checkbox), which a listbox option may not; roving `tabindex` over real controls is the correct accessible pattern here (and matches RecordTabs/reorder).
  - *A bespoke keyboard-help modal.* Rejected: the shared DS-03 Drawer already provides the modal contract; a second modal would duplicate the focus trap the design system forbids duplicating.

## ADR-032: Mobile Today — touch swipe quick actions as an additive shared-Card accelerator, and the touch-target corrections

- **Status.** Accepted (2026-07-21). Delivers [TODAY-06](../roadmap/ROADMAP_V2.md#-today-06--mobile). Builds ENTIRELY on [ADR-019](#adr-019-shared-card-identity--reorder-and-the-filter-expression--url-contract) (the one shared Card), [ADR-024](#adr-024-command-palette--quick-actions--command-kinds-trusted-catalogue-authenticated-execution-and-one-shared-action) (the one `AppAction`/`CardAction` execution path), [ADR-018](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) (the mobile Drawer sheet), [ADR-027](#adr-027-accessibility--responsive-baseline--automated-enforcement-and-the-inherited-platform) (the DS-11 baseline), and [ADR-028](#adr-028-task-drawer-persistence-and-composition--the-additive-task-detail-slice)/[030](#adr-030-task-planning--the-scheduled-date-as-commitment-atomic-single--bulk-mutations-and-planning-activity) (the task routes the actions drive). **No migration, no new store, no second Card/Drawer/toast/focus-trap, no new dependency, no touch-only functionality.**
- **Context.** DS-11 (ADR-027) inherited a WCAG 2.2 AA + responsive baseline; TODAY-01…05 composed the Today experience from the shared layer. TODAY-06 must make the *complete* Today workflow comfortable and dependable on a phone with touch — review/plan/reschedule/complete/select tasks, use Quick Capture, the Drawer and Waiting — and the roadmap specifically requires an adapted layout and **swipe quick actions**. The repo has no gesture library and, per [`OPEN_SOURCE_POLICY.md`](../governance/OPEN_SOURCE_POLICY.md) and the DS-08/DS-09 in-house precedent, a swipe recogniser is UX logic to own, not a commodity to depend on (evaluation recorded in [`REFERENCE_PRODUCTS.md`](../reference/REFERENCE_PRODUCTS.md)).
- **Decision.**
  - **§32.1 Swipe is an ADDITIVE capability on the ONE shared Card, not a Today-specific card.** The Card gains an optional `swipeActions?: readonly CardAction[]` prop. When given, the Card renders a clip/position wrapper (`.dh-card-swipe`) holding an action tray BEHIND the translated card surface (the article, unchanged internally). The wrapper is purely additive: on a non-touch device the reveal stays `0`, the tray is fully hidden behind the opaque surface, and desktop rendering is identical. Structure depends only on the prop (SSR-safe); it never depends on the device gate, so there is no hydration mismatch.
  - **§32.2 The tray reuses the EXISTING action model — one identity, one execution path.** The tray renders the SAME `CardAction`s the consumer supplies as visible `quickActions`; Today passes ONE `planQuickActions(...)` array to both. A tray action drives the same `onSelect`/`href` (which post to the trusted `/today/plan` or `/today/task/:id` routes, ADR-030/028) — never a parallel handler and never new task-mutation logic. The server route stays the correctness boundary; availability is state-dependent by OMISSION (a completed task's tray offers only Reopen; an unplanned task offers no Clear plan; a waiting task offers Clear waiting) exactly as the keyboard commands (ADR-031).
  - **§32.3 The tray is an ACCELERATOR, not the only path — no gesture-only functionality.** Every action in the tray is ALSO present as an ordinary visible `quickAction` on the card (keyboard/pointer/AT reachable) AND in the task Drawer. The tray is therefore `aria-hidden` and its buttons are out of the tab order — a visual duplicate, so assistive tech and keyboard use the real controls and are never asked to perform a custom gesture (DS-11: "no gesture-only functionality"; the swipe gesture itself need not be exposed to AT, but every action it reveals has an ordinary accessible control).
  - **§32.4 Gesture logic is pure and unit-tested; the hook only plumbs the DOM.** A React-free `swipe-model.ts` owns every decision — direction intent (`resolveSwipeIntent`: pending within a 12px deadzone, then the larger axis wins, ties to vertical so page scroll is favoured), offset projection + boundary clamping (`projectOffset`/`clampOffset`), the open/closed snap (`resolveRelease`: open past 40% of the tray width), and the single-open registry (`createSwipeRegistry`) — the roving-model/reorder precedent. `useCardSwipe` wires pointer events, measures the tray, drives a CSS reveal variable, captures the pointer only once horizontal is claimed, and suppresses the tap-to-open a touch swipe would otherwise fire.
  - **§32.5 Horizontal intent vs vertical scroll is deferred to the platform AND guarded in code.** The surface sets `touch-action: pan-y`, so the browser owns vertical scrolling natively (a vertical drag never moves the card and fires `pointercancel`); the model additionally only claims the gesture once movement is *clearly* horizontal past the threshold. Natural page scrolling is preserved when a vertical gesture starts; a minor accidental drag never reveals the tray; a handled swipe never also opens the Card.
  - **§32.6 Touch-first gate — desktop mouse/keyboard is untouched.** The hook responds to pointers only when `matchMedia('(hover: none) and (pointer: coarse)')` matches (the repo's `@media (hover: none)` convention), so a desktop mouse never reveals a tray and keyboard operation (TODAY-05) is unchanged. The structural DOM is identical regardless, so the gate toggles only behaviour.
  - **§32.7 One tray open at a time; close when appropriate.** A process-wide `sharedSwipeRegistry` closes any other open tray when one opens. A tray also closes on an outside pointer interaction, when a record Drawer opens (`closeActiveSwipeTray()`, called by Today's one record-open path), and after one of its actions fires.
  - **§32.8 Reduced motion + tokens.** The snap uses a token transition that `base.css` collapses to instant under `prefers-reduced-motion`; during an active drag the transition is disabled so the surface tracks the finger. Every distance/surface/timing value is a DS-01 token; the tray action meaning is icon + label, never colour alone.
  - **§32.9 Touch-target corrections surfaced by the mobile audit (shared-layer, honouring existing contracts).** The mobile pass axe-scanned real phone states for the first time and fixed two latent baseline gaps: (a) the shared Card selection control now meets the 44px touch target — the checkbox is wrapped in a `label` cell sized to `--dh-touch-target-min` on touch (the DS-06 completion-control pattern), the DS-04 contract already promised "44px targets on touch"; and (b) the app shell's mobile bar is a `header` so its brand + menu toggle are contained by the `banner` landmark on mobile (where the rail sidebar's banner is `display:none`), fixing a WCAG `region` gap. Both are corrections that honour existing documented contracts, not new patterns.
  - **§32.10 Composition-only; no forks.** The mobile Drawer is the unchanged DS-03 sheet (ADR-018); mobile selection + the bulk bar are the unchanged TODAY-04 flow in the CollectionLayout selection slot; safe-area handling extends the existing `env(safe-area-inset-*)` convention (the sticky pane header clears a notch on mobile). No second Drawer, sheet, toast, focus-trap, selection model or planning model.
- **Consequences.** *Easy:* the same trusted routes and one action identity back the mouse, the keyboard AND the swipe; the swipe capability is reusable by every future Mobile item (PROJ-06, AREA-04, …) by passing `swipeActions` to the shared Card; the touch-target/landmark fixes benefit every module. *Hard:* the Card gains one optional prop and an additive wrapper element (clipped with `overflow: hidden`, so a Today roving focus-within ring is applied to the wrapper); the single-open invariant uses a small process-wide registry (client-only, never touched during SSR). *Accepted & disclosed:* the swipe is an ACCELERATOR layered over the always-visible quick actions (a deliberate redundancy that guarantees no gesture-only action and keeps the DS-04 "never hide quick actions from touch" rule); swipe reveals a trailing tray in the app's LTR direction (a right-to-left reveal); the gesture is gated to touch-first devices, so a hybrid touch-laptop with a fine pointer keeps the unchanged desktop behaviour.
- **Alternatives considered.**
  - *A gesture library (`use-gesture`, `react-swipeable`, `framer-motion`).* Rejected — build. A swipe-to-reveal-existing-actions recogniser is UX logic the repo owns (the DS-08 fuzzy-matcher / DS-10 resize-hook precedent); the docs constrain swipe to mapping onto existing Quick Actions, so no novel physics is needed, and no code was copied (no attribution required). Evaluation recorded in [`REFERENCE_PRODUCTS.md`](../reference/REFERENCE_PRODUCTS.md).
  - *Execute a consequential mutation directly from a casual swipe (iOS-Mail-style swipe-to-complete).* Rejected as the default: a reveal-a-tray interaction is calmer and reversible, and the brief prefers it; the actions themselves remain the existing state-appropriate ones.
  - *Hide the visible quick actions on touch and rely on swipe + the Drawer.* Rejected: it would make the swipe a gesture-only path and violate the DS-04 "never hide quick actions from touch" rule; the accelerator model keeps every action reachable without a gesture.
  - *A Today-specific mobile Card / a parallel mobile route tree.* Rejected by AGENTS.md §9.8 — the one shared Card gains an additive capability; there is no second card and no `isMobile` markup fork.

## ADR-033: Re-homing the task record surface to a shared module boundary

- **Status.** Accepted (2026-07-21). Enables [PROJ-01](../roadmap/ROADMAP_V2.md#-proj-01--overview) to reuse the ONE task record Drawer without duplication. Builds on [ADR-028](#adr-028-task-drawer-persistence-and-composition--the-additive-task-detail-slice) (the task-detail slice), [ADR-018](#adr-018-the-shared-drawer--url-driven-history-stacked-focus-isolated) (the Drawer) and [ADR-013](#adr-013-module-registry-contract-and-discovery) §18 (the module import boundary). **No migration, no new store, no second Drawer/form/action route/completion path.**
- **Context.** TODAY-02…05 built the editable task record — the `TaskDrawerContent` component, its tabs/sections, its view-model, and four resource routes (`/today/task/:id*`) — INSIDE the Today module. PROJ-01 requires a task opened from a Project to use the SAME task record, actions and Drawer as Today, with NO Projects-only task Drawer, second task edit form, second task action route or second completion path. The architecture boundary (ADR-013 §18, enforced by a repository test) forbids one product module importing another's internals, so Projects cannot import Today's task surface, and the task surface cannot stay "too deeply inside Today" (the PROJ-01 brief's own words).
- **Decision.**
  - **§33.1 The reusable task surface is promoted to `app/shared/task-record/`.** The `TaskRecordDrawer` (the record composition + all mutations), the Details/Links/Timeline tabs, the waiting + planning sections, the pure `task-view` view-model, the `plan-targets` arithmetic and a `contract.ts` of the route↔drawer JSON types move to a shared surface any module may import (like DS-04/DS-06). React UI cannot live in the kernel, and a shared product surface is not a module, so `app/shared/` is the only home that both Today and Projects (and later Areas/Meetings) can consume without crossing the module boundary.
  - **§33.2 The task resource routes are re-homed to the Tasks module at `/tasks/:taskId*`.** The four action/loader routes (record, activity, link-targets, waiting-targets) move to the (previously placeholder) Tasks module — the module that OWNS the task entity type, its structural links and its Activity types. Modules address them by URL (a string), never by import, so no module→module import is introduced. The task Drawer's browser URL is unchanged (`?drawer=task:<id>`); only the internal fetch/POST target changed from `/today/task/:id` to `/tasks/:id`.
  - **§33.3 Today keeps a thin wrapper that adds ONLY its keyboard commands.** Today's `TaskDrawerContent` is now a wrapper that renders the shared `TaskRecordDrawer` and registers the TODAY-05 contextual keyboard commands (C / P / Shift+P / Clear waiting) around it, observing the drawer's live state + mutation handlers through a small `onApiChange` seam. All TODAY-05 command-ownership rules (only the interactive top drawer owns its shortcuts; the dashboard defers when a Drawer is open) are preserved; the shared drawer knows nothing of Today's keyboard workflow.
  - **§33.4 Owner-calendar helpers become shared.** `formatTodayDate`/`ownerCalendarIso`/`OWNER_TIME_ZONE` move to `~/shared/datetime` (Today re-exports the old path for compatibility), since Today, Tasks and Projects all resolve the owner's day.
- **Consequences.** *Easy:* Projects (and every future record surface) opens a task the ONE canonical way — same record, same forms, same routes, same completion path, same Drawer/focus machinery — by importing `TaskRecordDrawer` and deep-linking `?drawer=task:<id>`; a mutation revalidates whatever route hosts it (a project's roll-up + task list, or Today), for free. *Hard:* a mechanical relocation of ~12 files + their test imports; the route ids changed (`today.task*` → `tasks.record*`). *Accepted & disclosed:* Today's `date.ts` re-exports the shared helpers rather than being deleted (a one-line compatibility shim), keeping the Today module's own import paths stable.
- **Alternatives considered.**
  - *Keep the surface in Today; have Projects import it.* Rejected — violates the ADR-013 §18 module import boundary (enforced by a test) and leaves a project's task Drawer fetching a `/today/*` URL.
  - *Duplicate `TaskDrawerContent` for Projects.* Rejected by the PROJ-01 brief and AGENTS.md §9.8 — a bespoke duplicate is product debt the moment it merges, and would fork completion/waiting/planning.
  - *Put the surface in a `tasks` module barrel and import it cross-module.* Rejected — the boundary test forbids module→module imports even through a public barrel; `app/shared/` is the sanctioned cross-module home.

## ADR-034: The Projects module — a read-only projection over the spine, no second project model

- **Status.** Accepted (2026-07-21). Delivers [PROJ-01](../roadmap/ROADMAP_V2.md#-proj-01--overview). Builds on [ADR-014](#adr-014-spine-hierarchy-completion-and-rollup-semantics) (the spine authority + rollup), [ADR-010](#adr-010-server-side-workspace-context) (workspace-bound repositories), [ADR-028](#adr-028-task-drawer-persistence-and-composition--the-additive-task-detail-slice) (the Task contract) and [ADR-033](#adr-033-re-homing-the-task-record-surface-to-a-shared-module-boundary) (the shared task record). **No migration, no new persisted state, no second project table.**
- **Context.** PROJ-01 must let a user browse real projects and open a project overview (summary, tasks, key links, roll-up progress, create paths, basic actions). A Project is already a first-class spine record (FND-07); its identity, completion, parentage and authoritative rollup are the `SpineRepository`'s. The collection needs each project's Area/Goal context and task counts WITHOUT an N+1 (one rollup call per project), and the record needs its child tasks efficiently — but the FND-07 spine deliberately has no "list all projects" or "list a project's tasks" read.
- **Decision.**
  - **§34.1 A read-only `ProjectRepository` projection (kernel/projects + D1 adapter).** Exposed on `WorkspaceScope.projects`, it performs NO mutations. `listProjects({state,orderBy,limit,cursor})` resolves each project's Area (directly or through its Goal) and its active direct-task counts in ONE bounded query (a grouped task-count subquery + LEFT-JOINed structural links — no per-project rollup call). `getProjectOverview(id)` resolves one project's header/summary + Area/Goal. It adds no persisted state and no migration; the counts use the SAME definition as the spine project rollup (active direct child tasks), computed live — never a cached column.
  - **§34.1a Keyset cursor pagination makes every project reachable.** `listProjects` is paginated with an opaque, versioned cursor (`app/kernel/projects/project-cursor.ts`) that mirrors the spine/EntityLink cursor pattern (ADR-014 §11): base64url over a versioned JSON array, decoded with a FATAL UTF-8 pass, contents untrusted and every value still SQL-bound. The cursor is **scope-bound to workspace + state filter + ordering**, so a cursor issued for one filter/ordering/workspace is rejected (`InvalidSpineCursorError`) rather than silently reinterpreted against a different result set. Ordering is a stable keyset — `(created_at, id)` for `created`, `(updated_at, id)` for `recent` — with `id` as the deterministic unique tiebreaker; the query fetches `limit + 1` to detect a further page without a second COUNT and returns `nextCursor` (null when exhausted). No unbounded "load everything" query is ever issued.
  - **§34.2 The record's displayed progress is `SpineRepository.getRollup`, the single source of truth.** The projection's collection counts are a batched, N+1-free convenience for the collection cards; the project RECORD reads `spine.getRollup(projectId)` directly. An empty project is presented as 0% / "No tasks yet", never 100%.
  - **§34.3 The child-task query extends the authoritative Task contract.** `TaskRepository.listProjectTasks(projectId, {state,limit,cursor})` (ADR-028's contract) returns the project's child tasks as bounded, deterministic summaries in ONE workspace-scoped statement — matched by the active `task.belongs_to_project` link to an active project, so a wrong-kind/missing id yields no rows and never discloses cross-workspace existence. No `getTask()` per task; waiting tasks carry their TODAY-03 representation. It is **keyset-paginated** with its own versioned cursor (`app/kernel/tasks/task-project-cursor.ts`) **scope-bound to workspace + project id + state filter**, ordered `(created_at, id)` (a stable keyset simplifying the previous open-first/due-date presentation ordering so the cursor is bulletproof), fetching `limit + 1` and returning `nextCursor` so every matching task is reachable. The record's roll-up total stays `SpineRepository.getRollup` and remains authoritative while only some task pages are loaded — the page bounds ROWS, never the reported counts.
  - **§34.4 All project mutations go through existing atomic domain operations.** Create-project uses `SpineRepository.createProject` (the parent KIND is resolved server-side from the id — the client never asserts kind or ownership; a Goal parent derives its Area through the hierarchy, never stored twice); create-task uses `createTask` with the parent bound to the route project server-side (no client substitution); rename/complete/reopen use `spine.rename`/`complete`/`reopen`. Route code never inserts into `entities`/`spine_records`/structural links, and never mutates reserved spine types through the generic repositories. Non-structural relationships use the non-reserved `project.relates_to` EntityLink through the existing FND-04 policy service — no project-specific link table.
  - **§34.5 The module is composed entirely from the shared frame.** Two page routes (collection, record) + resource routes for create/mutate/link-target-search, the **project-task pagination page** (`/projects/:projectId/tasks?state=&cursor=`) and the **parent search** (`/projects/parent-options?q=`); the collection uses CollectionLayout + the one Card + a restrained URL-reflected state segment (Open/Completed/All — the DS-07 clause-builder is deferred to the richer filtering PROJ-02 needs); the record uses the DS-02 Record Layout; tasks open the shared `TaskRecordDrawer` (ADR-033) over the project; create/rename use DS-06 forms in the DS-03 Drawer; feedback uses DS-10. Resource routes return real JSON so the DS-06 forms post with `fetch`, and a page-loader revalidation reconciles the roll-up and task list after a mutation.
  - **§34.6 "Load more" is a shared, accessible affordance; pagination never disturbs drawer state.** Both the collection and the Tasks tab accumulate keyset pages behind the shared `app/shared/load-more` control (built from the DS button styles, announcing failures via `role="status"` and retrying with the same control). The next page is fetched with `useFetcher().load(...)` — the collection re-uses its own loader; the Tasks tab hits the dedicated `/projects/:projectId/tasks` endpoint — so **the URL is never navigated**: the record's `?drawer=task:<id>` state, scroll and focus survive loading more. The Tasks-tab accumulation is **reconciled** — dropped back to the fresh first page — when the task set may have changed underneath it: a `?tasks=` filter change OR a **mutation revalidation** (a task completed / edited / created, whose action revalidates the record loader with the URL unchanged), so a completed/edited task never lingers as a stale row and a new task surfaces from the authoritative page. It is NOT reset when only the `?drawer=` param changed (opening/closing the Drawer), so pagination and drawer state stay fully independent. Rows are de-duplicated by id so an overlapping page boundary can never render a card twice, and the subtitle says how many are "loaded" (never a false total) while more remain.
  - **§34.7 The New-Project parent picker is server-backed and searchable.** The set of eligible parents (every active Area and Goal) can exceed any static bound, so the "Area or Goal" field uses `SelectField`'s async `onSearch` against `/projects/parent-options?q=`, reusing the shared workspace-scoped, bounded, parameterised `searchLinkTargets` (restricted to `area`/`goal`, kinds resolved server-side). The client only chooses an id; the create action independently re-verifies the parent's kind and ownership server-side (§34.4), so the browser is never trusted to assert kind or workspace. Loading, no-results and error states are handled and in-flight searches are aborted so a slow response can't clobber a newer one.
- **Consequences.** *Easy:* browsing, opening, creating and acting on projects works over real D1 with the spine as the single authority and the rollup never cached; **every** project and task is reachable through keyset "Load more", not just a first bounded page; the projection is reusable by the Today "Continue working" read model (mapped to a plain shape, no cross-module import). *Hard:* a new small read-only kernel contract + D1 adapter (mirroring how ADR-028 added the Task repository), plus two versioned scope-bound cursors mirroring the spine/EntityLink pattern. *Accepted & disclosed:* the collection's live task counts and the record's `getRollup` share one definition but are two code paths (batched vs single), a deliberate efficiency choice; the paginated project-task list orders `(created_at, id)` rather than the open-first/due-date presentation order, trading a richer sort for a bulletproof keyset (the state filter still segments open/completed); PROJ-02–06 (health, notes, activity tab, settings, mobile) are explicitly not started.
- **Alternatives considered.**
  - *A second `projects` table with cached counts/percentages.* Rejected by the PROJ-01 brief and FND-07 — progress is derived, not stored; the audit proved no new project-specific state is needed.
  - *Per-project `getRollup` in the collection loop.* Rejected — an N+1; the batched projection computes all counts in one query.
  - *Compose the record from raw `spine` calls in the route.* Rejected for testability — `getProjectOverview` is one tested method resolving Area/Goal in one query, keeping domain resolution out of the route.
  - *Use the DS-07 FilterBar clause-builder for the state filter.* Deferred — a single mutually-exclusive server-side state is served more calmly by a segment; the full clause-builder fits PROJ-02's health/area/date filtering.

## ADR-035: Project Health — a derived, non-persisted signal over the spine, tasks and Activity

- **Status.** Accepted (2026-07-22). Delivers [PROJ-02](../roadmap/ROADMAP_V2.md#-proj-02--health). Builds on [ADR-034](#adr-034-the-projects-module--a-read-only-projection-over-the-spine-no-second-project-model) (the read-only projection), [ADR-014](#adr-014-spine-hierarchy-completion-and-rollup-semantics) (the spine authority + rollup, "derived, never cached"), [ADR-028](#adr-028-task-drawer-persistence-and-composition--the-additive-task-detail-slice)/[ADR-029](#adr-029-task-waiting--additive-state-a-reserved-entitylink-and-a-derived-first-class-display-state)/[ADR-030](#adr-030-task-planning--the-scheduled-date-as-commitment-atomic-single--bulk-mutations-and-planning-activity) (task status, waiting, due vs scheduled) and [ADR-012](#adr-012-activity-persistence-and-atomic-mutation-recording) (the Activity stream). **No migration, no new persisted state, no cached health column.**
- **Context.** PROJ-02 must signal whether a project is on track using progress, staleness, blockers and upcoming work, and explain WHY, at both the collection and record levels. Every input already exists: the spine's rollup + completion, the TODAY-02/03/04 task-detail fields (status, due date, scheduled date, waiting), and the FND-05 Activity stream. A health value that could disagree with those sources (a stored score, a cached column) is the failure mode to avoid; and the collection must compute health for a WHOLE bounded page without an N+1.
- **Decision.**
  - **§35.1 Health is DERIVED and NON-PERSISTED.** A pure evaluator `evaluateProjectHealth(facts, clock)` (`app/kernel/project-health`) maps raw facts to a health result; nothing is stored or cached, so health is recomputed every read and can never drift from tasks, Activity or the rollup (the same ethos as the spine's derived rollup). The evaluator is React-free and a pure function of `(facts, injected clock)`, so the whole rule matrix is unit-tested without a DB, a React tree or the wall clock, and every decision is made from STRUCTURED fields — never by parsing a display string.
  - **§35.2 Vocabulary.** Five health-only states, chosen to NOT collide with existing DalyHub vocabulary (`open`/`completed` is completion; `todo`/`in_progress` is task workflow; a richer project status model is PROJ-05's): `on_track`, `stale`, `blocked`, `at_risk`, `completed`. Each result carries a calm label, a presentation tone (a strict subset of the shared `CardTone` — `neutral`/`success`/`info`/`warning`/`danger`, mapped 1:1), one or more machine-readable reason codes with structured counts/days, the supporting facts the UI renders, and the evaluation instant. Tone is always paired with its text label (never colour alone).
  - **§35.3 Rules and precedence.** Primary-state precedence: **completed → at_risk → blocked → stale → on_track**. A completed project is never an active warning (only a calm "open tasks remaining" note when applicable). For an open project: `at_risk` when it has overdue due work or slipped scheduled commitments (a missed commitment is the most concrete signal — overdue is `danger`, slipped is `warning`); `blocked` when its remaining open work is effectively ALL waiting (`actionableOpen === 0 && waitingOpen > 0`); `stale` when it has OPEN work and no meaningful activity for the threshold; otherwise `on_track` (an empty project is a calm "No tasks yet", never 100%; an all-complete project is on track, never stale). Whichever state wins, EVERY applicable reason is preserved (primary first) — a stale project with overdue work and waiting tasks shows all three. Progress uses the authoritative rollup definition (active direct child tasks). Due dates (deadline) and scheduled dates (commitment) are kept strictly distinct.
  - **§35.4 Thresholds and meaningful activity (named domain constants).** `STALE_AFTER_DAYS = 14` (inclusive — a fortnight is a calm review cadence), `LONG_WAIT_AFTER_DAYS = 14`, `UPCOMING_WITHIN_DAYS = 7`. Staleness is computed from the latest MEANINGFUL activity across the project AND its child tasks — `MEANINGFUL_HEALTH_ACTIVITY_TYPES` = creation, edits, completion/reopen, planning and waiting changes; structural `entity_link.*` plumbing and `entity.deleted`/`entity.restored` are deliberately excluded as not "momentum" (there are no read events, so passive reads can never inflate momentum). Child-task activity therefore contributes to project momentum — `project.updated_at` alone is NOT treated as meaningful project activity. All day counts use the owner's calendar (`~/shared/datetime`) and UTC timestamps; date-only `YYYY-MM-DD` values are compared as strings, never routed through a timezone.
  - **§35.5 A read-only facts repository, bounded and N+1-free.** `ProjectHealthRepository` (on `WorkspaceScope.projectHealth`) gathers `ProjectHealthFacts` for a bounded set of project ids in a FIXED number of grouped queries — a project-base read, one grouped child-task aggregate (rollup + waiting/overdue/slipped/upcoming counts + oldest wait) and one grouped latest-meaningful-activity read (project + child-task subjects unioned, `MAX(occurred_at)` so duplicate subjects/links cannot inflate a count or date). Ids are chunked at 40 to stay within D1's 100 bound-variable limit; a whole page is a small fixed number of statements, never one query per project. Every statement is workspace-scoped and parameterised; soft-deleted tasks and unlinked (soft-deleted) links never contribute; a wrong-kind/missing/cross-workspace id is simply absent (never disclosed). The evaluator runs SERVER-SIDE in the loader with the owner-calendar clock, so health crosses to the browser as JSON — no client-only calculation over a partial task page.
  - **§35.6 Presentation reuses the shared frame; no second card.** A restrained `HealthIndicator` (a toned pill + the primary reason as accessible text) drops into the existing Card `metadata` slot on the collection (distinct from the open/completed `status` pill), and a `ProjectHealthPanel` explains all reasons + supporting facts in the DS-02 Record Summary. Today's "Continue working" cards show the SAME derived health, but only when a project needs attention (calm by default). Sensitive free-text waiting subjects are NEVER exposed in health facts, payloads or telemetry — only counts and the oldest `waiting_since` instant.
- **Consequences.** *Easy:* health is always consistent with its sources; a mutation's existing revalidation refreshes it with no cached column to invalidate; the derived model is reusable by later Area/Goal/Review surfaces (they gather the same facts and call the same evaluator). *Hard:* a new small read-only kernel contract + D1 adapter and a pure evaluator (mirroring how ADR-034 added the projection); the facts read is chunked to respect D1's variable limit. *Accepted & disclosed:* the collection's health facts and the record's `getRollup` share the rollup definition but are two code paths (batched vs single), a deliberate efficiency choice matching ADR-034 §34.2; the bare project RECORD page has a pre-existing PROJ-01 heading-order condition (task cards are `h4` under the record `h1`) surfaced — not caused — by PROJ-02's record axe scan, tracked in [`PRODUCT_DEBT.md`](../product/PRODUCT_DEBT.md) and out of scope here; PROJ-03–06 were not started.
- **Alternatives considered.**
  - *A persisted health score/column.* Rejected — it can disagree with tasks/Activity/rollup; the derived model cannot drift, matching the spine's "rollups are derived, never cached".
  - *An opaque weighted numeric score.* Rejected — the roadmap wants a calm state + transparent reasons; a simple state with explicit reasons is more honest and testable.
  - *Per-project activity/waiting queries in the collection loop.* Rejected — an N+1; the grouped, chunked facts reads compute a whole page in a fixed number of statements.
  - *A Today-only health calculation.* Rejected — Today reuses the exact same shared evaluator and facts, never a second model.
  - *Treating one waiting task as "blocked", or high completion as "healthy".* Rejected after the audit — blocked means the remaining open work is effectively all waiting; a project can be high-completion yet at risk from overdue/blocked remaining work.
